# Hosting PHP with Nginx: A Practical Setup

Nginx doesn't run PHP natively. You need PHP-FPM (FastCGI Process Manager) as the bridge between them. Here's how to get it working.

## Install PHP and Nginx

Run this on Ubuntu or Debian:

```bash
sudo apt-get update
sudo apt-get install -y php8.1-fpm nginx php8.1-mysql php8.1-curl php8.1-gd php8.1-mbstring php8.1-xml php8.1-zip
```

This pulls in Nginx, the PHP-FPM processor, and common PHP extensions you'll probably need for most applications.

## Configure Nginx for PHP

Replace your nginx config at `/etc/nginx/sites-available/default` with this:

```bash
sudo tee /etc/nginx/sites-available/default > /dev/null << 'EOF'
    server {
        listen 80;
        server_name _;
        root /var/www/html;
        index index.php index.html;

        # PHP application under /php-app
        location /php-app {
            alias /var/www/html/php-app;
            try_files $uri $uri/ /php-app/index.php?$args;

            location ~ \.php$ {
                include snippets/fastcgi-php.conf;
                fastcgi_param SCRIPT_FILENAME $request_filename;
                fastcgi_pass unix:/run/php/php8.1-fpm.sock;
            }
        }
    }
EOF

sudo systemctl restart nginx
sudo systemctl restart php8.1-fpm
```

The key parts: `fastcgi_pass` points to the PHP-FPM socket, and `SCRIPT_FILENAME` tells PHP which file to actually execute.

If your PHP files live somewhere other than `/var/www/html/php-app`, update both the `alias` and the `root` directive to match.

## Create a Test PHP File

Make a test endpoint so you can confirm everything is working:

```bash
mkdir /var/www/html/php-app/
sudo tee /var/www/html/php-app/index.php > /dev/null << 'EOF'
    <?php
    header('Content-Type: application/json');
    echo json_encode([
        'status' => 'success',
        'message' => 'PHP is working!',
        'php_version' => phpversion(),
        'server_time' => date('Y-m-d H:i:s')
    ]);
EOF
```

Visit `http://your-server/php-app/` and you should see a JSON response. If you get a 502 error, the PHP-FPM socket path might be wrong. If you see the file content downloaded instead of executed, the PHP location block isn't matching correctly.

## Watch the Logs While You Work

When something breaks, the logs tell you why:

```bash
tail -f /var/log/nginx/access.log /var/log/nginx/error.log ngrok.log &
sleep 1h
```

The error log catches configuration mistakes. The access log shows what's being requested.

## Key Takeaways

- Nginx needs PHP-FPM to process PHP files, not a module
- The `fastcgi_pass` directive connects Nginx to the PHP-FPM socket
- `try_files` handles clean URLs and falls back to index.php
- Check error.log first when something breaks

## FAQ

**What if PHP files download instead of running?**
Your `location ~ \.php$` block isn't matching. Make sure it's inside the right server block and that the regex pattern is correct. Also verify `fastcgi_pass` is pointing to a valid socket.

**Why 502 bad gateway?**
Usually means Nginx can't reach the PHP-FPM socket. Check that PHP-FPM is running (`systemctl status php8.1-fpm`) and that the socket path in your config matches what PHP-FPM is actually creating.

**Can I run multiple PHP versions?**
Yes. Install multiple PHP-FPM versions (like php8.1-fpm and php8.2-fpm), listen on different sockets, and point different `server` blocks or `location` blocks to the appropriate one.

**Is this setup production-ready?**
The basics here work, but for production you'd want to add PHP-FPM process management settings (pm dynamic vs static), opcode caching, and tighter file permissions. This is a solid starting point though.

**What does the alias directive do?**
It maps the URL path `/php-app` to the filesystem path `/var/www/html/php-app`. This separation between URL structure and file location gives you flexibility in how you organize your projects.
