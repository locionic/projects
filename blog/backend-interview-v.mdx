## Cơ Bản

### API và HTTP

#### 1. API endpoint là gì?

Một API endpoint là một URL cụ thể đóng vai trò là điểm truy cập vào một dịch vụ hoặc một chức năng cụ thể trong dịch vụ đó.

Thông qua API endpoint, các ứng dụng khách có thể tương tác với máy chủ bằng cách gửi yêu cầu (đôi khi kèm theo dữ liệu dưới dạng payload) và nhận phản hồi từ nó. Thông thường, mỗi endpoint được ánh xạ tới một chức năng duy nhất bên trong máy chủ.

#### 2. RESTful API là gì và những nguyên tắc cốt lõi của nó?

Để một API được coi là RESTful, nó cần tuân theo:

- **Client-server architecture**: Các dịch vụ dựa trên HTTP đều tuân theo mô hình này.
- **Uniform interface**:
  - Định danh tài nguyên qua URI (Unique Resource Identification).
  - Sửa đổi tài nguyên thông qua biểu diễn của chúng.
  - Thông điệp phải tự mô tả.
  - HATEOAS (Hypermedia as the Engine of Application State) - client có thể phát hiện hành động có sẵn từ phản hồi server.
- **Stateless**: Mỗi yêu cầu phải chứa đủ thông tin để xử lý, không lưu trạng thái trên server.
- **Layered system**: Có thể có trung gian mà không ảnh hưởng giao tiếp client-server.
- **Cacheable**: Tài nguyên có thể được cache bởi client hoặc server.
- **Code on Demand** (tuỳ chọn): Server có thể gửi code để client thực thi.

#### 3. Chu kỳ yêu cầu/phản hồi HTTP điển hình?

1. **Mở kết nối**: TCP đến server (port 80 cho HTTP, 443 cho HTTPS).
2. **Gửi yêu cầu**: HTTP method, URI, phiên bản HTTP, headers, body (tuỳ chọn).
3. **Xử lý yêu cầu**: Server xử lý và chuẩn bị phản hồi.
4. **Gửi phản hồi**: Phiên bản HTTP, mã trạng thái (200 OK, 404 Not Found...), headers, body (tuỳ chọn).
5. **Đóng kết nối**: HTTP/1.1 hỗ trợ giữ kết nối mở cho các yêu cầu tiếp theo.

#### 4. HTTP là stateless, ảnh hưởng gì đến backend?

HTTP không lưu trạng thái, mỗi yêu cầu độc lập. Điều này buộc backend phải tự triển khai giải pháp quản lý trạng thái (session, token...) khi cần theo dõi người dùng.

#### 5. Versioning API như thế nào?

- **Giữ trong URL**: `/v1/api/users`
- **Dùng header tùy chỉnh**: `api-version`

### Cơ Sở Dữ Liệu

#### 6. SQL vs NoSQL - Sự khác biệt?

| SQL | NoSQL |
|-----|-------|
| Schema cố định, định nghĩa trước | Không có schema, linh hoạt |
| Dữ liệu có cấu trúc nhất quán | Bộ sưu tập không bắt buộc cùng cấu trúc |
| Quan hệ qua foreign key | Dữ liệu nhúng hoặc reference |

#### 7. Làm thế nào để triển khai tìm kiếm toàn văn bản?

1. **Tiền xử lý văn bản**: Tách từ, stemming, loại bỏ stop words.
2. **Tạo chỉ mục ngược**: Liên kết mỗi từ duy nhất với các bản ghi chứa từ đó.
3. **Chuẩn hóa input**: Xử lý tương tự văn bản gốc.
4. **Tìm kiếm và chấm điểm**: Xếp hạng theo tần suất xuất hiện.

Có thể dùng chức năng tìm kiếm gốc của MySQL, PostgreSQL hoặc ElasticSearch.

#### 8. Chiến lược quản lý kết nối database khi tải cao?

1. **Connection pool**: Tái sử dụng kết nối, giảm thời gian thiết lập mới.
2. **Cân bằng tải** các truy vấn giữa một nhóm database.
3. **Tối ưu hóa truy vấn**: Giảm thời gian sử dụng mỗi kết nối.

#### 9. Idempotent trong REST API?

- Dùng HTTP verb tự nhiên idempotent: GET, PUT, DELETE.
- Áp dụng logic dựa trên key để tránh lặp lại thao tác.

#### 10. Deadlock trong giao dịch database?

- **Sắp xếp thứ tự khóa**: Đảm bảo khóa được lấy theo thứ tự nhất quán.
- **Timeout**: Hủy giao dịch chạy lâu.
- **Optimistic concurrency control**: Tránh giữ khóa quá lâu.

#### 11. Optimistic vs Pessimistic Locking?

**Optimistic Locking:**
- Giả định xung đột hiếm khi xảy ra.
- Cho phép truy cập đồng thời, kiểm tra xung đột trước commit.
- Phù hợp: đọc cao, ghi thấp.

**Pessimistic Locking:**
- Giả định xung đột thường xuyên.
- Khóa dữ liệu trong suốt giao dịch.
- Phù hợp: ghi cao, yêu cầu toàn vẹn dữ liệu.

### Bảo Mật

#### 12. Bảo vệ khỏi SQL Injection?

- **Prepared Statements** với truy vấn tham số hóa.
- **ORM** (Object-Relational Mapping).
- **Escaping** special characters.

#### 13. Bảo mật API mới?

- **Xác thực**: OAuth, JWT, Bearer token.
- **Mã hóa**: HTTPS.
- **CORS** hợp lý.
- **Quyền truy cập** mạnh mẽ.

#### 14. Bảo mật giao tiếp giữa các microservices?

- **TLS** ngăn man-in-the-middle.
- **API Gateway** quản lý và xác thực.
- **Xác thực và phân quyền** giữa các service.

#### 15. Mã hóa dữ liệu trong ứng dụng tập trung quyền riêng tư?

- **Dữ liệu đang truyền**: HTTPS.
- **Dữ liệu ở trạng thái nghỉ**: AES, RSA, ECC. Lưu khóa trong KMS.

#### 16. Tuân thủ GDPR?

1. Chỉ thu thập dữ liệu cần thiết, có sự đồng ý người dùng.
2. Bảo mật dữ liệu cả khi truyền và lưu trữ.
3. Cung cấp quyền đọc, sửa, xóa dữ liệu.

### Backend Tổng Quát

#### 17. Quản lý phiên làm việc?

1. **Tạo phiên** khi đăng nhập.
2. **Lưu thông tin phiên** trong memory hoặc database.
3. **Gửi session ID** qua cookie.
4. **Client gửi session ID** trong mỗi yêu cầu.
5. **Server truy xuất** dữ liệu phiên.
6. **Đóng phiên** khi đăng xuất hoặc hết hạn.

#### 18. Quản lý session trong môi trường cân bằng tải?

- **Sticky sessions**: Gán phiên vào server cố định.
- **Kho tập trung**: Redis để mọi server truy cập được.

#### 19. Xử lý tải lên tệp?

- **Xác thực**: Kiểm tra kích thước, loại tệp.
- **HTTPS** cho kênh an toàn.
- **Đổi tên** tránh trùng tên.
- **Lưu metadata** trong database.

#### 20. Xử lý tiến trình chạy lâu trong web request?

**Kiến trúc phản ứng**:
1. Chuyển yêu cầu thành message trong hàng đợi.
2. Trả phản hồi ngay cho client.
3. Client lắng nghe sự kiện hoàn thành.

#### 21. Rate Limiting?

1. Xác định giới hạn (theo giây, phút, ngày).
2. Chọn chiến lược: fixed window, sliding log, token bucket.
3. Lưu bộ đếm trong Redis.
4. Trả mã **429 Too Many Requests** khi vượt quá.

#### 22. Microservices là gì? Tách monolith?

1. Xác định ranh giới logic trong monolith.
2. Tách dữ liệu và dịch vụ theo từng phần độc lập.
3. Triển khai từng phần như microservice riêng.
4. Di chuyển dần.

#### 23. Reverse proxy và lợi ích?

- **Cân bằng tải** giữa các backend server.
- **Bảo mật**, ẩn vị trí backend.
- **Cache**, giảm tải server.
- **Chuyển hướng linh hoạt** không ảnh hưởng URL công khai.

#### 24. Webhook là gì?

Webhook là HTTP callback do người dùng định nghĩa, kích hoạt khi sự kiện xảy ra:

1. Xác định sự kiện kích hoạt.
2. Tạo endpoint xử lý.
3. Xác thực yêu cầu webhook đảm bảo bảo mật.

#### 25. Containerization và lợi ích?

Containerization là phương pháp ảo hóa nhẹ, đóng gói ứng dụng và phụ thuộc. Lợi ích:

- Môi trường nhất quán.
- Cô lập và dễ triển khai.
- Giảm xung đột phiên bản phần mềm.

#### 26. Mở rộng ứng dụng khi tăng đột biến?

**Cân bằng tải** + **mở rộng ngang** bằng cách thêm nhiều phiên bản ứng dụng.

### Testing & DevOps

#### 27. Viết test cho API endpoint mới?

- **Unit tests**: Kiểm thử logic cốt lõi, tránh test private methods.
- **Integration tests**: Test qua giao diện công khai.
- **Load/Performance tests**: Đánh giá hiệu suất dưới tải nặng.

#### 28. Load testing API?

1. Xác định mục tiêu, môi trường tương tự sản xuất.
2. Thiết kế test với JMeter, LoadRunner.
3. Tăng dần tải, theo dõi hiệu suất.
4. Tối ưu, lặp lại.

#### 29. Pipeline CI/CD cho backend?

1. Dùng **version control** làm trigger (Git, GitHub, GitLab).
2. Chọn nền tảng CI/CD: GitHub Actions, GitLab CI/CD, CircleCI.
3. Tích hợp **test tự động**.
4. **Deploy tự động** khi tất cả test thành công.
5. **Artifact repository**: JFrog Artifactory, Nexus.
6. **Rollback strategy** khi deploy lỗi.

### Kiến Trúc Hệ Thống

#### 30. Database Replication và chịu lỗi?

Sao chép dữ liệu giữa nhiều phiên bản database:

- **Cluster chịu lỗi**: Thăng cấp slave thành master khi master lỗi.
- **Slave làm read-only**: Giảm tải đọc, không ảnh hưởng hiệu suất.

#### 31. Chiến lược triển khai xanh-lam (Blue-Green)?

Hai môi trường sản xuất giống hệt nhau:
- Một môi trường đang phục vụ lưu lượng thật.
- Môi trường kia chuẩn bị cập nhật hoặc làm backup.

#### 32. Quản lý schema migration trong CD?

- Theo dõi schema trong **version control**.
- Dùng công cụ tự động: **Flyway** hoặc **Liquibase**.

#### 33. Caching phân tán cho ứng dụng high availability?

1. Triển khai **cache cluster** với phân mảnh dữ liệu.
2. **Nhân bản cache** để chịu lời.
3. **Vô hiệu hóa cache** để tránh dữ liệu lỗi thời.

#### 34. Chiến lược xóa cache phía server?

- Giới hạn dung lượng để kích hoạt xóa khi vượt quá.
- **Chính sách phổ biến**:
  - **LRU**: Xóa ít được truy cập gần nhất.
  - **LFU**: Xóa được truy cập ít nhất.
  - **FIFO**: Xóa theo thứ tự thêm vào.
  - **Random**: Chọn ngẫu nhiên.
  - **TTL**: Xóa sau khoảng thời gian nhất định.

#### 35. CAP Theorem?

Cơ sở dữ liệu phân tán không thể đảm bảo đồng thời hơn hai trong ba:

- **Consistency (Nhất quán)**: Mọi truy vấn nhận kết quả mới nhất.
- **Availability (Sẵn có)**: Mọi yêu cầu nhận phản hồi.
- **Partition Tolerance (Chịu lỗi phân vùng)**: Hệ thống hoạt động khi có sự cố mạng.

#### 36. Eventual Consistency là gì?

Đảm bảo dữ liệu trong hệ thống phân tán sẽ **nhất quán theo thời gian**, thay vì ngay lập tức.

#### 37. Gỡ lỗi ứng dụng backend?

- **Debugging IDE**: IntelliJ, Eclipse.
- **Logging**.
- **Công cụ chuyên dụng**: JProfiler, NewRelic.

#### 38. Đảm bảo mã nguồn dễ bảo trì?

- **Module hóa**.
- **Quy tắc đặt tên** nhất quán.
- **Comment** khi cần.
- **Tái cấu trúc** thường xuyên.
- **Xử lý lỗi** nhất quán.
- **Unit test** đầy đủ.

## Trung Cấp

### Message Queue & Background Tasks

#### 39. Hàng đợi tin nhắn (Message Queue)?

Hàng đợi tin nhắn là thành phần cốt lõi của kiến trúc phản ứng. Mỗi dịch vụ kích hoạt và lắng nghe sự kiện từ hàng đợi. Khi sự kiện đến, dịch vụ phản hồi mà không cần kiểm tra liên tục các dịch vụ khác.

#### 40. Quản lý tác vụ nền (Background Jobs)?

1. **Hàng đợi tác vụ**: RabbitMQ, Amazon SQS.
2. **Framework xử lý**: Celery (Python), Sidekiq (Ruby).
3. **Cron jobs**: Nếu phù hợp.
4. **Threads/Workers**: Nếu ngôn ngữ hỗ trợ.

#### 41. Xử lý hàng loạt (Batch Processing)?

Dùng framework chuyên dụng như **Hadoop** hoặc **Spark** - được thiết kế để xử lý lượng dữ liệu khổng lồ song song.

### Caching & Performance

#### 42. Theo dõi hiệu suất backend?

**APM (Application Performance Management)**: New Relic, AppDynamics, Dynatrace.

### Kiến Trúc & Thiết Kế

#### 43. SSO (Single Sign-On)?

1. Chọn **Identity Provider**: Okta, Keycloak.
2. Mỗi ứng dụng tích hợp qua giao thức: **SAML**, **OpenID**.
3. Lần đầu truy cập: xác thực với IdP, nhận access token.
4. Các lần sau: xác thực token qua IdP.

#### 44. Quản lý phụ thuộc API?

Dùng **API versioning** để đảm bảo các hệ thống dùng đúng phiên bản.

#### 45. Correlation ID trong kiến trúc phân tán?

Correlation ID là định danh duy nhất gắn vào yêu cầu để theo dõi luồng dữ liệu qua các dịch vụ. Hỗ trợ **gỡ lỗi** và phát hiện vấn đề hiệu suất.

#### 46. Phát hiện và ngăn chặn dữ liệu bất thường?

- **Xác thực dữ liệu**: Schema và ràng buộc.
- **Phiên bản hóa dữ liệu**: Dễ rollback khi lỗi.
- **Kiểm soát chất lượng**: Xác thực đầu vào, gắn cờ dữ liệu nghi ngờ.

### IoT & Real-time

#### 47. Backend xử lý dữ liệu IoT?

- **Thu thập**: Kafka hoặc AWS Kinesis, hỗ trợ MQTT, CoAP.
- **Xử lý thời gian thực**: Apache Flink, Spark Streaming.
- **Lưu trữ**: Hệ thống chuỗi thời gian như InfluxDB.

#### 48. Đồng bộ dữ liệu thời gian thực giữa thiết bị?

**Kênh giao tiếp**:
- Kết nối socket hai chiều.
- Pub/sub: Redis, Kafka.

**Giải quyết xung đột**:
- **Operational Transformation (OT)**
- **CRDTs** (Conflict-free Replicated Data Types)

#### 49. Kiến trúc microservices - Lợi ích và hạn chế?

**Lợi ích**:
- **Khả năng mở rộng**: Mở rộng độc lập từng service.
- **Linh hoạt công nghệ**: Chọn tech stack phù hợp cho từng service.
- **Triển khai nhanh**: Deploy độc lập, tăng tốc phát hành.

**Hạn chế**:
- **Phức tạp**: Khó quản lý khi mở rộng.
- **Khó debug**: Theo dõi luồng dữ liệu qua nhiều service.
- **Chi phí giao tiếp cao**: So với monolithic.

#### 50. Hệ thống lưu trữ toàn cầu, khả dụng cao?

- **Môi trường đa vùng**: AWS, Azure đảm bảo khả dụng.
- **Sao chép dữ liệu**: Nhân bản giữa các máy chủ.
- **Cân bằng tải**: Điều phối lưu lượng giữa các vùng.
- **Quản trị dữ liệu**: Quyền truy cập phù hợp, tuân thủ GDPR.

## Key Takeaways

1. **API và HTTP là nền tảng** - Hiểu sâu RESTful, stateless, các verb và status code.
2. **Bảo mật không thể thiếu** - SQL injection, XSS, CSRF, authentication, encryption.
3. **Database là trái tim** - SQL vs NoSQL, replication, sharding, indexing, locking strategies.
4. **Caching là chiến lược** - Từ LRU đến distributed caching, hiểu khi nào và làm sao.
5. **Message Queue giải quyết async** - Tách biệt các service, xử lý background jobs hiệu quả.
6. **Microservices cân bằng tradeoffs** - Linh hoạt nhưng phức tạp, cân nhắc kỹ trước khi tách.
7. **Monitoring và Observability** - Logging, APM, Correlation ID để debug hệ thống phân tán.
8. **CAP Theorem và Eventual Consistency** - Hiểu các hệ thống phân tán có giới hạn gì.
9. **CI/CD là bắt buộc** - Tự động hóa test và deploy để đảm bảo chất lượng và tốc độ.
10. **Không ngừng học hỏi** - Công nghệ thay đổi, kiến trúc tốt hơn kiến thức cụ thể.

## FAQ

### Câu hỏi thường gặp

**Q: Nên bắt đầu ôn tập từ đâu?**
A: Bắt đầu từ phần Cơ Bản, đảm bảo nắm chắc HTTP, REST, SQL, Authentication trước khi chuyển sang phần nâng cao.

**Q: Những câu hỏi nào hay gặp nhất trong phỏng vấn?**
A: RESTful API, SQL vs NoSQL, Authentication/JWT, Caching strategies, và System Design thường xuyên xuất hiện nhất.

**Q: Backend interview có cần biết DevOps không?**
A: Nên hiểu cơ bản CI/CD, Docker/Kubernetes, và cloud infrastructure vì nhiều team kỳ vọng backend dev biết deployment và infrastructure.

**Q: Cần code trong backend interview không?**
A: Tuỳ công ty. Nhiều nơi kiểm tra Algorithm & Data Structures riêng biệt, không yêu cầu code backend. Một số công ty có live coding phần API.

**Q: System Design interview nên chuẩn bị thế nào?**
A: Thực hành thiết kế hệ thống từ nhỏ đến lớn: URL Shortener → TinyURL, Chat System → Messenger, Rate Limiter → API Gateway...
