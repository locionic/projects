## Key Takeaways

- Indexes are your first performance win. Add compound indexes that match how you query.
- Filter early, project late. Get rid of unneeded data as soon as possible.
- Pagination isn't optional at scale. Skip/limit from day one.
- Test with `explain()` before shipping. The query plan tells you everything.
- `$facet` runs parallel operations in a single stage—no need for multiple queries.

---

## Use Indexes Effectively

What actually speeds up MongoDB queries? Indexes. Without them, Mongo scans every document.

**In your schema:**

```javascript
ExampleSchema.index({ user_id: 1, updated_at: -1 }) // Compound index for sorting
ExampleSchema.index({ user_id: 1, show_id: 1 })     // Compound index for grouping
```

**Force an index in a query if needed:**

```javascript
.hint({ user_id: 1 })
```

Most of the time, Mongo picks the right index. But when it doesn't, `.hint()` overrides the planner.

## Project Only Needed Fields

Stop fetching fields you throw away. `投影` (project) only what you use.

```javascript
{
  $lookup: {
    from: 'shows',
    let: { show_id: '$_id' },
    pipeline: [
      {
        $match: {
          $expr: { $eq: ['$_id', '$$show_id'] }
        }
      },
      {
        $project: {
          // Only get fields you need
          title: 1,
          coverImage: 1,
          slug: 1
        }
      }
    ],
    as: 'show'
  }
}
```

## Use $facet for Multiple Operations

Need a count alongside your results? `$facet` runs parallel pipelines in a single stage.

```javascript
{
  $facet: {
    // Get count in parallel with results
    total: [
      { $group: { _id: '$show_id' } },
      { $count: 'count' }
    ],
    result: [
      // Your main query pipeline
    ]
  }
}
```

## Early Filtering

**Filter first. Always.** Put `$match` at the start of your pipeline.

**Do this first:**

```javascript
{ $match: { user_id: ObjectId(user_id) } }
```

**Before expensive operations like:**

```javascript
{ $sort: ... }
{ $group: ... }
```

Every document you filter out early never hits your sort or group stage.

## Efficient Data Processing

Instead of stacking multiple `$lookup` stages, merge objects in projection:

```javascript
// Instead of multiple $lookup stages
{
  $project: {
    show: {
      $mergeObjects: [
        { $arrayElemAt: ['$show', 0] },
        { url: { $arrayElemAt: ['$episode.url', 0] } }
      ]
    }
  }
}
```

## Use Aggregation Stages in Right Order

Here's the natural order:

```javascript
[
  { $match: ... },     // Filter first
  { $sort: ... },      // Sort early if possible
  { $group: ... },     // Reduce document count
  { $skip: ... },      // Pagination after reducing
  { $limit: ... },     // Limit results
  { $lookup: ... },    // Join with other collections
  { $project: ... }    // Final shape
]
```

The earlier you reduce the document count, the faster everything downstream runs.

## Monitor Query Performance

Use `explain()` to see what's actually happening:

```javascript
// Add explain() to see query plan
const explain = await Example.aggregate([...]).explain('executionStats')
console.log(explain.executionStats.executionTimeMillis)
```

Check `executionTimeMillis` and `totalDocsExamined`. If that ratio is bad, you need better indexing or earlier filtering.

## Use Pagination

Don't dump everything at once. Ever.

```javascript
const perPage = 6
const page = parseInt(req.query.page) || 1
const offset = (page - 1) * perPage

// In pipeline
{ $skip: offset }
{ $limit: perPage }
```

## Error Handling and Defaults

Handle the null case so your API doesn't break:

```javascript
try {
  const result = await query
  return {
    total: total || 0,      // Default if null
    result: result || []    // Default if null
  }
} catch (error) {
  console.error('Query error:', error)
  return rep.internalServerError()
}
```

## FAQ

**How do I know if my index is being used?**

Run `.explain('executionStats')` and check `winningPlan`. If you see `IXSCAN`, the index is working. If you see `COLLSCAN`, it's scanning the whole collection.

**What's a compound index?**

An index on multiple fields. The order matters. `{ user_id: 1, updated_at: -1 }` works for queries filtering by `user_id` AND sorting by `updated_at`.

**Should I index every field?**

No. Indexes take memory and slow down writes. Only index fields you actually filter, sort, or group by.

**When should I use `$lookup`?**

When you need to join with another collection. For small, read-heavy data, consider denormalizing instead.

**My query is still slow. Now what?**

Check `executionStats.executionTimeMillis`. Aim for under 100ms for user-facing queries. If it's higher, look at your `totalDocsExamined`—you might be scanning too many documents.
