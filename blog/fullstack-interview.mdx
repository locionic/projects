> **Tóm tắt nhanh**
> - Fullstack dev làm việc trên cả frontend (HTML, CSS, JavaScript) và backend (Node.js, Python, database)
> - REST API dùng HTTP methods: GET lấy data, POST tạo mới, PUT cập nhật, DELETE xóa

## Full-stack development là gì?

Full-stack development là practice xây dựng và bảo trì cả frontend lẫn backend của một ứng dụng web. Một full-stack developer làm việc trên tất cả layers: từ UI cho đến server và database.

Ba khía cạnh chính:

- **Frontend** — HTML, CSS, JavaScript, và frameworks như React, Angular, Vue. Cái người dùng nhìn thấy và tương tác.
- **Backend** — Server-side logic, databases, APIs. Ngôn ngữ phổ biến: Python, Java, Node.js, PHP.
- **Database** — Relational (MySQL, PostgreSQL) hoặc non-relational (MongoDB, Redis).

Full-stack developers được đánh giá cao vì khả năng hiểu toàn bộ stack — giúp deliver sản phẩm hoàn chỉnh thay vì chỉ một phần.

## Client-side vs server-side programming khác nhau thế nào?

Client-side chạy trong browser của user, xử lý UI và tương tác. Server-side chạy trên server, xử lý database queries, business logic, và authentication.

### Client-side

- Chạy JavaScript trong browser — handle form validation, animations, DOM updates
- Giao tiếp với server qua REST APIs (fetch, axios) để lấy hoặc gửi data
- Ví dụ: click button hiển thị popup, load thêm items không reload trang

### Server-side

- Xử lý requests từ client, thực hiện database operations, trả về data
- Ngôn ngữ: Python, Java, Node.js
- Ví dụ: verify login credentials, trả về JSON danh sách sản phẩm

Theo Mozilla, REST APIs là cách phổ biến nhất để client-server giao tiếp trong 2025 ([MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status), 2024).

## HTML, CSS, JavaScript có vai trò gì?

- **HTML** — định nghĩa cấu trúc và nội dung trang
- **CSS** — style trang (màu sắc, layout, fonts)
- **JavaScript** — thêm tính tương tác và hành vi động

Ba thành phần này là nền tảng không thể thiếu của mọi ứng dụng web.

## REST API là gì, tại sao cần?

REST API (Representational State Transfer) là một convention để các ứng dụng giao tiếp qua HTTP. Nó cho phép clients (web browsers, mobile apps) tương tác với servers.

Đặc điểm chính:

1. **Stateless** — mỗi request phải chứa đủ thông tin để server xử lý, không reliance vào stored session
2. **Resource-based** — data và functionality được treat như "resources" accessed qua endpoints
3. **HTTP methods định nghĩa actions:**
   - GET: lấy data
   - POST: tạo resource mới
   - PUT: cập nhật existing resource
   - DELETE: xóa resource
4. **Response format** — JSON hoặc XML

Tại sao dùng REST? Interoperability (kết nối nhiều systems), scalability (stateless = horizontal scaling dễ), và flexibility (dùng được từ web đến mobile, IoT).

## GET và POST khác nhau thế nào?

Chuẩn accepted hiện tại:

- **GET** — lấy data từ server (read-only, không thay đổi state)
- **POST** — gửi data để tạo hoặc cập nhật resource

GET parameters appear in URL (visible, length-limited), POST parameters in request body (hidden, unlimited size).

## Div và Span khác nhau thế nào?

- **`<div>`** — block-level element, bắt đầu dòng mới, dùng để nhóm block-level elements khác (layout, sections)
- **`<span>`** — inline element, nằm trên cùng dòng với text, dùng để style một phần text mà không ảnh hưởng đến layout

## CSS selectors là gì, có những loại nào?

Selectors là patterns để chọn và style elements cụ thể trong HTML document.

**Các loại:**

- **Element selectors** — `p { }` chọn tất cả `<p>`
- **Class selectors** — `.my-class { }` chọn elements có class đó
- **ID selectors** — `#my-id { }` chọn đúng một element (ID unique)
- **Attribute selectors** — `[type="text"] { }` chọn theo attribute value
- **Descendant selectors** — `div p { }` chọn `<p>` bên trong `<div>`

Best practices: dùng class cho reusable styles, ID chỉ khi thật sự unique element, combine selectors cho granular control.

## `==` và `===` khác nhau thế nào?

- **`==`** — so sánh giá trị, tự động convert type trước khi so sánh. `'5' == 5` → `true`
- **`===`** — so sánh cả giá trị VÀ type, không convert. `'5' === 5` → `false`

Luôn dùng `===` để tránh unexpected type coercion bugs.

## Relational vs non-relational databases?

- **Relational** (MySQL, PostgreSQL) — lưu data trong structured tables với rows và columns. Tốt cho complex queries và relationships.
- **Non-relational** (MongoDB, Redis) — lưu data trong flexible formats (documents, key-value, graphs). Tốt cho unstructured数据 hoặc hierarchical data.

Chọn dựa trên data structure và query patterns của ứng dụng.

## Xử lý authentication trong web application?

JWT (JSON Web Token) là approach phổ biến:

1. **Frontend** — hiển thị login form
2. **Backend** — verify credentials, tạo signed token, return về client
3. **Subsequent requests** — frontend gửi token trong header, backend verify token

Best practices: hash passwords với bcrypt, dùng HTTPS, implement refresh tokens cho security.

**MongoDB (Mongoose ODM):**

```javascript
// Create
User.create({ name: 'John Doe' });

// Read

// Update
User.updateOne({ name: 'John Doe' }, { name: 'Jane Doe' });
```

**PostgreSQL (raw SQL):**

```javascript

// Update

// Delete
pool.query('DELETE FROM users WHERE name = $1', ['Jane Doe']);
```

## Responsive design là gì, implement như thế nào?

Responsive design đảm bảo website looks good trên mọi devices bằng cách adapt layout theo screen size.

**Implement:**

```css
/* Flexible grids */
.container {
  display: flex;
  flex-wrap: wrap;
}

/* Media queries cho breakpoints */
@media (max-width: 768px) {
  .container {
    flex-direction: column;
  }
}
```

Ngoài ra: dùng relative units (`%`, `em`, `rem`) thay vì fixed units (`px`) để values tự động adapt.

Theo Google, 61% người dùng sẽ rời site nếu không mobile-friendly ([Google Analytics](https://support.google.com/analytics/answer/10089681), 2024). Responsive design không còn là optional.

## Flexbox vs CSS Grid?

- **Flexbox** — one-dimensional layouts (row HOẶC column). Best cho navigation bars, centering elements.
- **CSS Grid** — two-dimensional layouts (rows VÀ columns). Best cho complex grid-based layouts như dashboards, full-page layouts.

Flexbox giải quyết align items trong container đơn giản. Grid giải quyết layout phức tạp với cả rows và columns.

## React hooks là gì, tại sao dùng?

Hooks cho phép dùng state và React features trong functional components — không cần class components.

**Examples:**

- `useState` — quản lý state
- `useEffect` — xử lý side effects (fetch data, subscriptions)
- `useContext` — global state across components

```javascript
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    document.title = `Count: ${count}`;
  }, [count]);

  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

## State management trong React hoạt động thế nào?

**Local state** — dùng `useState` trong component đó:

```javascript
const [user, setUser] = useState(null);
```

**Global state** — dùng Context API, Redux, MobX, hoặc Zustand để share state giữa many components.

```javascript
// Context API
const UserContext = createContext();
<UserContext.Provider value={{ user, setUser }}>
  <App />
</UserContext.Provider>
```

How it works: state update → action handler → re-render UI. Tránh overusing global state vì nó tạo unnecessary re-renders.

## Tối ưu performance React app như thế nào?

1. **Reduce re-renders** — `React.memo`, `useCallback` để tránh unnecessary updates
2. **Lazy loading** — load components/routes on demand với `React.lazy` và `Suspense`
3. **Efficient state** — keep state local where possible
4. **Minimize DOM updates** — dùng keys trong lists, avoid deeply nested state
5. **Code splitting** — webpack split bundle thành smaller chunks
6. **Profile** — React DevTools để identify bottlenecks

## Middleware trong Node.js/Express là gì?

Middleware là functions xử lý requests và responses trong request-response cycle. Nó có thể:

- Modify request/response objects
- Execute code (logging, parsing JSON)
- End the cycle early (handle auth failures)

```javascript
app.use((req, res, next) => {
  console.log('Request received');
  next();
});
```

Auth middleware example:

```javascript
function requireAuth(req, res, next) {
  if (!req.user) return res.status(401).send('Unauthorized');
  next();
}
```

## Async code trong JavaScript xử lý thế nào?

### Callbacks

Function passed as argument để execute sau async task:

```javascript
fs.readFile('file.txt', (err, data) => {
  if (err) {
    console.error('Error:', err);
    return;
  }
  console.log('Data:', data.toString());
});
```

Problem: deeply nested callbacks → "callback hell"

### Promises

```javascript
fetch('https://api.example.com/data')
  .then(response => response.json())
  .then(data => console.log('Data:', data))
  .catch(error => console.error('Error:', error));
```

Cleaner than callbacks, hỗ trợ chaining via `.then()`.

### Async/Await

Syntax build on top of promises:

```javascript
const fetchData = async () => {
  try {
    const response = await fetch('https://api.example.com/data');
    const data = await response.json();
    console.log('Data:', data);
  } catch (error) {
    console.error('Error:', error);
  }
};
```

Đọc như synchronous code — dễ hiểu hơn, error handling với try/catch.

## Relational databases handle relationships thế nào?

- **One-to-Many** — một record liên quan đến nhiều records. Dùng foreign keys. Ví dụ: một user có nhiều posts.
- **Many-to-Many** — cần join table trung gian. Ví dụ: students và courses với enrollments table.
- **Primary/Foreign Keys** — establish links, đảm bảo data consistency.

## Authenticate và authorize API như thế nào?

**Authentication** — verify user identity. Dùng tokens (JWT, OAuth) hoặc sessions:

```javascript
const token = req.headers['authorization'];
jwt.verify(token, secretKey, (err, decoded) => { ... });
```

**Authorization** — sau khi authenticated, kiểm tra user có quyền access resource không:

```javascript
app.use((req, res, next) => {
  if (req.user.role !== 'admin') return res.status(403).send('Forbidden');
  next();
});
```

Best practices: HTTPS, validate input, rate-limit, hash passwords.

## WebSockets vs HTTP requests?

- **HTTP** — request-response model. Client initiates mỗi interaction. Không persistent connection.
- **WebSockets** — persistent connection, full-duplex communication. Server có thể push data đến client mà không cần request.

Use cases: chat apps, live notifications, real-time dashboards. HTTP polling để so sánh: client gửi request liên tục để check new messages; WebSocket server push khi có messages mới.

## MVC architecture là gì?

**Model-View-Controller** — design pattern tổ chức code thành 3 layers:

1. **Model** — data và business logic (database interactions)
2. **View** — display data cho users (HTML, templates)
3. **Controller** — handle user input, communicate giữa Model và View

**Flow:** User interacts với View → Controller processes input → Updates Model → Changes reflected in View.

## CORS là gì, xử lý thế nào?

CORS (Cross-Origin Resource Sharing) kiểm soát access resources từ different origin (domain, protocol, hoặc port).

**Backend (Express):**

```javascript
const cors = require('cors');
app.use(cors({ origin: 'https://example.com' }));
```

**Frontend dev:** Proxy API requests qua dev server để tránh CORS issues.

## Dùng Postman test APIs như thế nào?

1. **Create request** — enter endpoint, method (GET, POST, etc.), headers
2. **Send data** — query params, body (JSON, form data)
3. **View response** — status code, body, headers
4. **Assertions** — viết JavaScript tests để validate responses:

5. **Collections** — group requests cho testing workflows

## Memory leaks trong frontend tránh thế nào?

Memory leaks xảy ra khi unused resources không được release đúng cách.

**Solutions:**

1. **Clean up event listeners:**

```javascript
useEffect(() => {
  window.addEventListener('resize', handler);
  return () => window.removeEventListener('resize', handler);
}, []);
```

2. **Abort fetch requests:**

```javascript
const controller = new AbortController();
fetch(url, { signal: controller.signal });
return () => controller.abort();
```

3. **Tránh stale references** — check component state trước khi update
4. **DevTools profiling** — monitor memory usage

## GraphQL vs REST?

**GraphQL** — query language cho APIs, client request exactly data cần.

**Key differences:**

| Aspect | REST | GraphQL |
|--------|------|---------|
| Data fetching | Fixed endpoints | Single endpoint, flexible queries |
| Batching | Multiple endpoints | Nested queries in one request |
| Versioning | New versions | Evolve schemas, no versioning |
| Overfetching | Thường fetch thừa data | Client chọn fields cần |

**Example query:**

```graphql
query {
  user(id: 1) {
    name
    posts {
      title
    }
  }
}
```

5. **Cache invalidation** — TTL, event-driven invalidation

Browser caching cũng giảm backend requests cho static resources.

Content-Security-Policy headers để restrict allowed script sources.

## CI/CD pipelines hoạt động thế nào?

**CI (Continuous Integration):** tự động build và test code khi push. Tools: Jenkins, GitHub Actions, CircleCI.

**CD (Continuous Delivery):** tự động deploy sau khi CI pass.

**Pipeline flow:**

```
Pull code → Build → Run tests → Deploy → Notify
```

Tools: Jenkins (custom workflows), GitHub Actions (easy integration), Docker (containerized builds).

## Performance monitoring trong full-stack app?

**Frontend:**

- Google Lighthouse, Web Vitals cho load times, interactivity
- New Relic Browser, LogRocket cho user behavior

**Backend:**

- APM tools: Datadog, Dynatrace cho response times, DB query performance

**Logging:**

- ELK Stack, CloudWatch logs cho centralized logging

**Database:**

- Query profilers (MySQL EXPLAIN), Percona Monitoring

**Alerting:**

- Prometheus, Grafana cho anomaly detection

## Event-driven architecture là gì?

Services communicate bằng cách emit và respond to events asynchronously.

**Components:**

- **Event Producer** — generates events (e.g., user uploads file)
- **Event Consumer** — listens and reacts (e.g., service processes file)
- **Message Broker** — facilitates delivery (Kafka, RabbitMQ)

**Use cases:**

- Real-time apps (chat, stock trading)
- Decoupled microservices
- Asynchronous workflows (order processing)

## Common challenges khi build full-stack app?

1. **Managing state across frontend và backend**
   - Solution: global state tools (Redux, Zustand), clear API data contracts

2. **Scalability**
   - Solution: optimize queries, caching, scalable infrastructure

3. **Security**
   - Solution: OAuth2, sanitize inputs, follow OWASP guidelines

4. **Code quality consistency**
   - Solution: robust testing (unit, integration, e2e), regular code reviews

5. **Keeping up with tech updates**
   - Solution: modular architecture để replace outdated tech incrementally

6. **Debugging frontend/backend interactions**
   - Solution: E2E testing (Cypress), centralized logging

## FAQ

### Cần bao lâu để prepare cho fullstack interview?

Với developer đã có 1-2 năm kinh nghiệm: 3-6 tuần focus study. Người mới: 3-6 tháng. Trọng tâm hiểu concepts sâu, không chỉ học thuộc.

### Những câu hỏi nào được hỏi thường xuyên nhất?

API design, async programming (callbacks/promises/async-await), state management (React), database queries, và security basics (SQL injection, XSS) gần như luôn có mặt.

### Cần biết những tools nào?

Git, một IDE (VS Code), Postman/Insomnia, Docker basics, và browser DevTools không thể thiếu. SQL và một language backend (Node.js hoặc Python) là foundation.

### System design questions подготовка như thế nào?

Practice với các bài toán scale: design URL shortener, design chat system, design Twitter feed. Tập trung vào tradeoff decisions và reasoning thay vì specific answers.
