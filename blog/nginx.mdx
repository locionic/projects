# Nginx Tips You Actually Need

Nginx configuration trips up a lot of people. Here are the two things I bump into most often when setting up reverse proxies and web servers.

## Trailing Slash: The Detail That Breaks Your Routes

The trailing slash in a location block changes everything about how URLs get forwarded to your backend.

**Without trailing slash:**
```
location /chatpppeee {
  ...
}
URL: example.com/chatpppeee/users
→ Proxied to: localhost:9000/chatpppeee/users
```

**With trailing slash:**
```
location /chatpppeee/ {
  ...
}
URL: example.com/chatpppeee/users
→ Proxied to: localhost:9000/users
```

Why does this matter?

The trailing slash tells Nginx whether to strip the location prefix before passing the request along. With it, `/chatpppeee/users` becomes `/users`. Without it, the full path stays intact.

This keeps your backend routes clean. You don't have to hardcode `/chatpppeee` into every route definition in your FastAPI app.

**Without the trailing slash, you've got two options:**
- Add `/chatpppeee` to every route in your application
- Configure your app to use a root path prefix

Both are annoying. Just add the trailing slash.

## Key Takeaways

- Always include the trailing slash in your location blocks for proxying
- The trailing slash strips the prefix before forwarding: `/chatpppeee/users` becomes `/users`
- Your backend sees clean URLs without the proxy path prefix
- This is the simplest way to avoid double paths and route conflicts

## FAQ

**Why does my backend see the full path instead of the stripped one?**
You probably forgot the trailing slash in your location directive. Check if you have `location /app` instead of `location /app/`.

**How do I debug what's being forwarded to my backend?**
Enable debug logging or check the access logs. You can also add a test endpoint that echoes back the full request URI so you can see exactly what Nginx is sending.

**Can I mix trailing and non-trailing slash locations?**
Yes, but the most specific match wins. A location `/app` won't catch `/app/users` the same way `/app/` does. Stick with trailing slashes for proxy passing to keep it predictable.

**What happens if I don't use a trailing slash?**
Your application has to know about the proxy path. Every route, every link, every redirect has to account for `/chatpppeee` being in the URL. It couples your app to the proxy configuration unnecessarily.

**Does this apply to all reverse proxies?**
The trailing slash behavior is specific to Nginx. Other proxies like HAProxy or Caddy may handle path manipulation differently. Check their docs if you're not using Nginx.
