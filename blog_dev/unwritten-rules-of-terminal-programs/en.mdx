# Unwritten Rules of Terminal Programs
=====================================================

As a long-time user of terminal programs, I've noticed that despite the lack of formal standards, these programs tend to follow certain unwritten rules. In this post, I'll outline these rules, which make it easier for users to understand and interact with terminal programs.

### Programs Behave Surprisingly Consistently

Although there are no formal standards for terminal programs, they tend to behave in a consistent manner. This consistency is due in part to the influence of POSIX, which dictates how terminal emulators, operating systems, and shells work together. Additionally, many terminal programs follow established guidelines and conventions, such as the [Command Line Interface Guidelines](https://clig.dev/).

### Descriptive, Not Prescriptive

The rules outlined in this post are descriptive, not prescriptive. They are based on my observations of how terminal programs typically behave, rather than a set of strict guidelines that programs must follow.

### It's Not Always Obvious Which "Rules" Are the Program's Responsibility

Some conventions, such as the location of configuration files or the use of `--help` to print help text, are clearly the responsibility of the program. However, other conventions, such as the use of readline keybindings or the behavior of Ctrl-C, are not as clear-cut.

### Rule 1: Noninteractive Programs Should Quit When You Press Ctrl-C

Noninteractive programs typically quit when you press Ctrl-C. This is because the default behavior of the operating system is to send a SIGINT signal to the program when Ctrl-C is pressed, which causes the program to exit. However, interactive programs, such as shells or text editors, may not exit when Ctrl-C is pressed, as it is used to interrupt the current operation instead.

### Rule 2: TUIs Should Quit When You Press q

Text-based user interfaces (TUIs) often quit when you press the `q` key. This is a common convention, although it may not apply to all TUIs.

### Rule 3: REPLs Should Quit When You Press Ctrl-D on an Empty Line

Read-eval-print loops (REPLs) typically quit when you press Ctrl-D on an empty line. This is because the operating system returns an EOF (end-of-file) character when Ctrl-D is pressed on an empty line, which causes the REPL to exit.

### Rule 4: Don't Use More Than 16 Colors

Terminal programs rarely use more than 16 colors, as this is the standard number of colors supported by most terminals. Using more colors can lead to display issues, especially if the user has a custom color scheme.

### Rule 5: Vaguely Support Readline Keybindings

Many terminal programs support readline keybindings, even if they don't use the readline library directly. This includes keybindings such as Ctrl-E to go to the end of the line, Ctrl-A to go to the beginning of the line, and Ctrl-W to delete the last word.

### Rule 5.1: Ctrl-W Should Delete the Last Word

Ctrl-W is commonly used to delete the last word in a terminal program. This is consistent with the default behavior of the operating system, which deletes the last word when Ctrl-W is pressed in "cooked mode."

### Rule 6: Disable Colors When Writing to a Pipe

Most terminal programs disable colors when writing to a pipe, as this can lead to display issues or incorrect color codes.

### Rule 7: - Means Stdin/Stdout

Many terminal programs use the `-` character to indicate that input should be read from stdin or output should be written to stdout.

### These "Rules" Take a Long Time to Learn

These unwritten rules take time to learn, as they are not explicitly documented and may vary between programs. However, by understanding these conventions, users can more effectively interact with terminal programs and avoid common pitfalls.
