# Enhancing Man Pages for Tcpdump and Dig

As a follow-up to my previous exploration of man pages, I've been working on adding examples to the man pages of two essential networking tools: tcpdump and dig. My goal was to provide the most basic examples of how to use these tools, targeting infrequent users or those new to them.

### Updated Man Pages

The updated man pages include:

* The dig man page with new examples
* The tcpdump man page with improved examples

These examples aim to give users a solid starting point when using these tools. By including basic use cases, I hope to make the man pages more approachable and useful for those who don't use tcpdump or dig regularly.

### Why Improve Man Pages?

I'm interested in working on official documentation for several reasons:

* **Accuracy**: Man pages can have close to 100% accurate information, thanks to the review process.
* **Discovery**: Even with basic questions, maintainers often know useful features that might not be well-known.
* **Correctness**: Documentation can be just as good as a great blog post, but with the added benefit of being accurate.

For example, while working on the tcpdump examples, I learned that using `-v` with `-w` can print a live summary of captured packets. This is a useful feature I wouldn't have discovered otherwise.

### Overcoming the Roff Language Barrier

The tcpdump project's man page is written in the roff language, which can be challenging to use. To avoid learning roff, I created a basic markdown-to-roff script to convert Markdown to roff, using similar conventions to the existing man page. This allowed me to focus on writing the examples in Markdown.

### The Complexity of Man Pages

I delved into the history of roff, its evolution since the 1970s, and the current state of the mandoc project. While I won't go into details here, I'm intrigued by the technical and cultural differences in documentation between BSD and Linux systems.

### Conclusion

Improving man pages can make a significant difference in the user experience, especially for infrequent users. By providing basic examples and accurate information, we can make documentation more approachable and useful. I'm optimistic about the potential for high-quality documentation and look forward to continuing my work in this area.
