# Modernizing GitHub Issues Navigation Performance

When working through a backlog, latency can significantly impact a developer's workflow. Even small delays can add up and disrupt the flow of work. To address this issue, the GitHub team set out to optimize the performance of GitHub Issues navigation. In this post, we'll explore the approach they took and the results they achieved.

## The Problem: Latency and Context Switching

Latency isn't just a metric; it's a context switch. When navigating through issues, latency can cause developers to lose focus and disrupt their workflow. GitHub Issues wasn't slow in isolation, but too many navigations still paid the cost of redundant data fetching, breaking the flow of work.

## The Approach: Shifting Work to the Client and Optimizing Perceived Latency

To fix this issue, the GitHub team shifted work to the client and optimized perceived latency. They aimed to render instantly from locally available data and then revalidate in the background. This approach involved building a client-side caching layer backed by IndexedDB, adding a preheating strategy to improve cache hit rates without spamming requests, and introducing a service worker to speed up navigation paths.

## The Metric: HPC (Highest Priority Content)

To measure performance, the GitHub team used HPC (Highest Priority Content), an internal metric closely aligned with Web Vitals LCP. HPC measures when the primary content on the page is first rendered. They bucketed navigations using HPC thresholds:

* Instant: HPC < 200 ms
* Fast: HPC < 1000 ms
* Slow: HPC >= 1000 ms

## The Baseline: Navigation Mix Before Optimization

Before implementing optimizations, the GitHub team analyzed the navigation mix for GitHub Issues. They identified three primary navigation types:

* Hard navigation: a full browser load where they pay the full cost of network, server rendering, asset loading, JavaScript boot, and React hydration.
* Turbo navigation: a Rails Turbo transition that updates targeted page regions without a full reload.
* Soft navigation (React): a client-side transition inside the existing React runtime.

The measured distribution at the start of the workstream showed that hard navigations made up the largest share of the baseline.

## Step 1: Client-Side Caching with IndexedDB

The GitHub team started by optimizing React soft navigations, where they had immediate architectural leverage and could ship improvements quickly. They extended their current in-memory store with a persistent client cache in IndexedDB. The implementation involved:

* Reading from local cache first and rendering immediately on soft navigation
* Issuing a background network request for freshness and reconciling the in-memory store if data changed
* Introducing a new graceful-degradation model when network is degraded

## Preheating and Cache Hit Ratios

To improve cache hit ratios, the GitHub team introduced preheating, which proactively walks high-intent issue references and prepares cache entries ahead of navigation. Preheating only hits the network when the issue is not already present in the client cache. This approach is an explicit tradeoff between freshness and capacity usage.

## Results and Tradeoffs

The GitHub team achieved significant results with their optimization efforts:

* Approximately 22% of React navigations became instant, up from 4% pre-launch
* Observed cache-hit ratio landed around one-third (~33%)
* Measured server/cache divergence at about 4.7%

However, there are tradeoffs to consider:

* Controlled staleness: the team measured server/cache divergence and treated it as an explicit operating envelope
* Capacity usage: preheating and caching can amplify request volume and create N+1-style access patterns

## Conclusion

The GitHub team's approach to optimizing GitHub Issues navigation performance demonstrates the importance of reducing latency and improving perceived performance. By shifting work to the client and optimizing perceived latency, they achieved significant results and improved the user experience. The tradeoffs and lessons learned from this effort can be applied to other data-heavy web applications.
