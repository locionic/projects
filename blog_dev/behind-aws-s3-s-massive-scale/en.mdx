# Behind AWS S3's Massive Scale

AWS S3 is a widely used cloud storage service that has grown exponentially since its release in 2006. With over 280 trillion objects stored and 100 million requests per second, S3 has become a behemoth of a service, spanning 31 regions and 99 availability zones. In this article, we'll explore the architecture and scalability of S3, delving into its components, storage fleet, replication, sharding, and heat management.

## Architecture

S3's architecture is composed of over 300 microservices, following the core design principle of simplicity. The service can be broken down into four high-level components:

* A front-end fleet with a REST API
* A namespace service
* A storage fleet full of hard disks
* A storage management fleet that performs background operations, such as replication and tiering

Each of these components operates independently, with its own teams and leaders, and interacts with other components through API-level contracts.

## Storage Fleet

At the core of S3 is its storage fleet, which consists of millions of hard disks. The storage node servers are simple key-value stores that persist object data to hard disks. The node servers only store shards of the overall object data, with the control plane replicating said shards across many different nodes.

AWS has developed a new storage backend called ShardStore, which is a simple log-structured merge tree (LSM Tree) with shard data stored outside the tree to reduce write amplification. ShardStore uses a soft-updates-based crash consistency protocol and is designed from the ground up to support extensive concurrency.

## Hard Drives

Hard drives are an old technology that's not ideal for all use cases. They're constrained for IOPS, have high seek latency, and are physically fragile. However, they've seen exponential improvement over the years, with price decreasing by 6,000,000,000 times and capacity increasing by 7,200,000 times.

Despite these improvements, hard drives are still constrained for IOPS, with a maximum of 120 IOPS. To overcome this limitation, S3 heavily leverages parallel IO.

## Replication

In storage systems, redundancy schemes are a common practice to ensure durability. S3 uses Erasure Coding (EC), a complex data protection method that breaks data into K shards with M redundant "parity" shards. EC allows S3 to recover data from any K shards out of the total K+M shards.

Erasure Coding helps S3 find a middle balance between durability and I/O efficiency, as it doesn't require replicating data multiple times.

## Sharding

S3 tries to spread shards as broadly as possible across its storage fleet. This approach provides several benefits, including:

* Hot spot avoidance: By spreading data across multiple drives, S3 avoids hot spots that can cause cascading delays.
* Burst demand: Spreading data across multiple drives allows for greater burst IO.
* Greater durability: Spreading data across multiple drives increases durability, as S3 can sustain more individual machine failures.
* No read amplification: Spreading data across multiple drives reduces read amplification, as disk seeks are distributed across multiple drives.

## Heat Management at Scale

Managing and balancing I/O demand across a large set of hard drives is a significant technical challenge for S3. The goal is to minimize the number of requests that hit the same disk at any point in time, avoiding hot spots and tail latency.

S3 experiences workload decorrelation, where the aggregate demand of millions of workloads flattens out, making it easier to balance I/O demand.

## Parallelism

Parallelism is a key aspect of S3's architecture, as it unlocks better performance for customers and allows S3 to optimize for workload decorrelation. S3 leverages parallelism in two ways:

* Across servers: S3 encourages users to create multiple parallel connections to different endpoints, utilizing many different servers.
* Intra-operation: S3 supports multipart upload for PUT requests and range-based reads for GET requests, allowing for parallelism within a single operation.

## Metadata and Strong Consistency

In 2020, S3 introduced strong read-after-write consistency, ensuring that once data is written, any subsequent read will receive the latest version of the object. This change had zero impact on performance, availability, or cost.

S3 has a discrete subsystem for storing per-object metadata, which uses a highly resilient caching technology to ensure persistence and availability.
