Mess With DNS is a DNS playground, but it ran into a critical issue: **periodic out-of-memory (OOM) errors**. The culprit was the IP address lookup database, which consumed a massive chunk of our limited 465MB of RAM.

<Callout type="danger" title="The Memory Crisis">
  Out of 465MB total RAM, the IP address lookup database was eating **117MB**. Combined with PowerDNS (100MB) and Mess With DNS (200MB), the server was constantly pushed to its breaking point.
</Callout>

Here is the journey of how we optimized it.

## The Optimization Journey

<Stepper>
  <Step title="Attempt 1: Using SQLite">
    **The Idea:** Move the database to disk using SQLite to free up RAM.
    
    **The Result:** A failure. SQLite lacked support for big integers (forcing IPv6 addresses to be stored as text), and the disk lookups were **500x slower** than the original binary search implementation.
  </Step>

  <Step title="Attempt 2: Using a Trie">
    **The Idea:** Store IPs in an in-memory Trie data structure for fast prefix matching.
    
    **The Result:** Worse than before. The Trie implementation bloated memory usage to a staggering **800MB** and was somehow slower than the baseline binary search.
  </Step>

  <Step title="Attempt 3: Optimizing the Array (Success!)">
    **The Idea:** Go back to the original array-based binary search, but optimize the data layout.
    
    **The Fixes:**
    1. **Deduplicate ASN information:** We moved repeated ASN data into a separate array and referenced it by index. *Saved 50MB.*
    2. **Use `netip.Addr` instead of `net.IP`:** Go's `netip.Addr` is much smaller and more efficient than the legacy `net.IP` type. *Saved 20MB.*
  </Step>
</Stepper>

<Callout type="info" title="Memory Profiling in Go">
  If you're tracking down memory leaks in Go, use the `runtime` and `pprof` packages:
  * Call `runtime.GC()` to force garbage collection before taking measurements.
  * Use `pprof` to generate and analyze heap profiles.
  * Pass `--alloc-space` or `--inuse-space` to differentiate between historical allocations and active memory usage.
</Callout>

## Conclusion

By analyzing our data structures and making targeted changes to our Go code, we reduced the IP lookup database's memory footprint from **117MB down to 46MB**—a **60% reduction**. The final solution retained the blazing speed of binary search without crashing the server!
