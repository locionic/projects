# Optimizing Core Unit Boot Time: A Cloudflare Story

Cloudflare's core infrastructure is comprised of centralized data centers that run the control plane, billing, and analytics. These core servers are bare metal, and when issues arise during reboot, the consequences can be severe. The boot sequence is orchestrated by UEFI, the modern firmware standard that initializes hardware and hands off control to the operating system. However, small quirks in this handoff can have significant consequences.

## The Problem: Slow Boot Times

After a routine firmware update, some of Cloudflare's core servers were taking four hours to come back online, rather than just minutes as they did before. This was caused by a firmware quirk and an over-eager linear search through every available network boot interface. The behavior was exacerbated by recent updates to the boot protocols used by servers in some of Cloudflare's locations.

## Understanding the Network Boot Interface

A network boot interface allows a server to boot its operating system over the network instead of from local storage. This is critical for centralized, automated, and scalable control over how machines start up, especially across a globally distributed fleet serving different workloads. Cloudflare uses the open-source iPXE, an open-source network boot firmware that supports modern protocols like HTTP and HTTPS.

## The Linear Search

The problem began with a fateful firmware update. Following the update, the first reports came through Cloudflare's internal channels: servers weren't coming back online. Monitoring dashboards showed machines stuck in a pre-OS state for far longer than expected. The firmware Power On Self Test (POST) completed normally, but the server sat waiting for an extended period. The console output told the story: the system was attempting an IPv4 HTTPS network boot, timing out after several minutes, then trying IPv4 iPXE, timing out again, then repeating both — all before finally reaching the IPv6 HTTPS boot interface that would actually succeed.

## Declaring the Boot Interface

To eliminate the guesswork, Cloudflare decided to declare the correct boot interface upfront so the system never wastes time on interfaces that will never respond. However, this was far from straightforward. The order of the boot automation workflow, a setting that was blocked from changing, and differing string formats from different network interface card vendors all presented obstacles.

## Restructuring the Automation Sequence

By restructuring the automation sequence to declare the network boot interface order early on in the pre-boot PXE stage for each hardware/use-case, Cloudflare was able to cut the total time by about an hour. However, this introduced two specific constraints: Legacy Support and Persistence.

## Addressing Edge Cases

To address these edge cases, Cloudflare implemented a state validation step. The firmware automation now validates the configuration post-change: if it detects that settings have been modified, it re-applies the config and triggers a reboot.

## Working with Vendors

Cloudflare worked with their vendors to enable specific tokens within the fixed "Boot Order Module." This forces the discovery of the Network Boot Interface during the boot sequence without requiring manual GUI interaction.

## Standardizing Network Interface Strings

Cloudflare is currently working with their UEFI vendors to standardize the network interface strings to only make use of the relevant information (e.g., protocol, transfer type, port number, and physical slot index) and drop the product details like the MAC address.

## Result: A More Dynamic System

By eliminating the guesswork from the network boot sequence, Cloudflare turned a four-hour ordeal back into a 3-minute process. The result is a system where changes are dynamic and no manual BIOS interactions are needed. A single BIOS firmware image serves all SKUs, configuration updates deploy at scale through the existing release pipeline, and the entire workflow operates from iPXE.

| Metric | Before Ordering Change | After Ordering Change |
| --- | --- | --- |
| Firmware Upgrade Automation | Nearly 4 hours | 3 minutes |
| Subsequent Single Boot | About 20 minutes | Less than a minute |

This optimization was only possible by digging deep into UEFI internals, collaborating closely with OEM vendors, and leveraging open-source tools like iPXE to build scalable automation.
