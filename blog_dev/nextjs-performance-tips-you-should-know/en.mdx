## The Optimization Toolbox

<Callout type="idea" title="The Mental Model">
  Every performance technique in Next.js maps to one of two goals: *get the first pixels on screen faster* or *send less JavaScript to the client*. Keep that in mind and the "why" becomes obvious.
</Callout>

<Stepper>
  <Step title="1. Image Optimization — The #1 Win for LCP">
    
    Use the `<Image>` component from `next/image`, not the raw `<img>` tag. It does three things automatically:
    
    <ul className="space-y-2 text-sm">
      <li><strong>Auto WebP/AVIF conversion</strong> — smaller files with no quality loss</li>
      <li><strong>Size/quality control</strong> — serve a 300px image on mobile, 1200px on desktop</li>
      <li><strong>Layout shift prevention</strong> — requires width/height so the browser reserves the right space</li>
    </ul>
    
    ```tsx filename="ImageDemo.tsx"
    import Image from 'next/image';

    export function Hero() {
      return (
        <Image
          src="/hero.jpg"
          alt="Dashboard overview"
          width={1200}
          height={675}
          priority
          className="rounded-xl"
        />
      );
    }
    ```

    <Alert type="warning" title="Critical: Always provide width and height">
      Without both, Next.js cannot reserve layout space, and CLS suffers. Use the real intrinsic dimensions of your image. If in doubt, check the image file in your OS file browser or use an image editor.
    </Alert>
    
  </Step>

  <Step title="2. Font Optimization — Eliminate FOIT">
    
    Use the `next/font` module. It downloads fonts at build time and serves them locally—zero third-party requests, zero layout shift.
    
    <Tabs defaultTab="Google Fonts">
      <Tab label="Google Fonts">
        
        ```tsx filename="fonts.ts"
import { Inter, Roboto_Mono } from 'next/font/google';

const inter = Inter({
  subsets: ['latin'],
  display: 'swap',     // ✅ Shows fallback text instantly
  variable: '--font-inter',
});

const robotoMono = Roboto_Mono({
  subsets: ['latin'],
  weight: ['400', '700'],
});
        ```
        
        <SuccessAlert title="What this gives you for free">
          <ul className="text-sm space-y-1">
            <li>Self-hosted font files — no external request to Google</li>
            <li>Automatic `font-display: swap` — text visible while font loads</li>
            <li>Subset by `subsets` — drop unused glyphs (e.g. omit `cyrillic` if unused)</li>
            <li>CSS variable for easy Tailwind integration</li>
          </ul>
        </SuccessAlert>
        
      </Tab>
      
      <Tab label="Local Fonts">
        
        ```tsx filename="local-font.ts"
import localFont from 'next/font/local';

const geist = localFont({
  src: './fonts/GeistVF.woff2',
  display: 'swap',
  variable: '--font-geist',
});
        ```
        
        <InfoAlert title="When to use local fonts">
          If you already have font files (purchased or self-hosted), `next/font/local` gives you the same zero-shift benefits without going through Google's CDN.
        </InfoAlert>
        
      </Tab>
    </Tabs>
    
  </Step>

  <Step title="3. Dynamic Imports — Split Heavy Code">
    
    Don't load components that aren't visible on initial paint. Use `next/dynamic` to code-split at the component level:
    
    ```tsx filename="dynamic-chart.tsx"
// Heavy chart library: ~200KB gzipped
import dynamic from 'next/dynamic';

// Load it only when needed (client-side, no SSR)
const RevenueChart = dynamic(
  () => import('@/components/RevenueChart'),
  {
    ssr: false,           // ✅ Skip server rendering for this component
    loading: () => <Skeleton />, // ✅ Show placeholder instantly
  }
);

export function Dashboard() {
  return <RevenueChart />;
}
    ```
    
    <Alert type="info" title="When to skip SSR (`ssr: false`)">
      <ul className="text-sm space-y-1">
        <li>Charts/graphs that rely on `window` or browser APIs</li>
        <li>Heavy animation libraries (GSAP, Framer Motion)</li>
        <li>Third-party widgets (Google Maps, Stripe Elements)</li>
        <li>Dark mode toggles — wait, this is cheap enough to render on server</li>
      </ul>
    </Alert>
    
    <Accordion type="multiple">
      <AccordionItem title="What does code splitting actually do?" defaultOpen>
        Without dynamic imports, bundlers may group everything into a single JS file. With dynamic imports, the component loads in a separate chunk fetched only when needed. This is the **single biggest bundle size reducer** for medium-to-large apps.
      </AccordionItem>
      <AccordionItem title="How much should I split?">
        Use the bundle analyzer (step 6) to find your heaviest components. Typically: rich text editors, charts, 3D viewers, auth widgets. Don't over-split—dynamic imports add a network round-trip. Split components that prevent meaningful initial JS download.
      </AccordionItem>
    </Accordion>
    
  </Step>

  <Step title="4. Server Components — Default to Not Shipping JS">
    
    In the App Router, all components are Server Components by default. This is your biggest performance lever—you get the results of rendering *without* sending that component's JS to the browser.
    
    ```tsx filename="server-component.tsx"
// This entire file runs on the server
// → Zero JavaScript sent to the browser for this component
async function BlogPost({ slug }: { slug: string }) {
  const post = await db.posts.findUnique({ where: { slug } });

  return (
    <article>
      <h1>{post.title}</h1>
      <MDXContent source={post.content} />
    </article>
  );
}
    ```
    
    <Tabs defaultTab="Decision Tree">
      <Tab label="Decision Tree" icon={<span>🌳</span>}>
        
        ```text filename="decision-flow.txt"
Need a useState, useEffect, onClick, or window?
    └── Yes → Add "use client" (Client Component)
    └── No  → It's already a Server Component (best default)

Server Component delivers:
  ✅ HTML directly to browser (fast FCP)
  ✅ Data fetching on server (no client fetch waterfalls)
  ✅ Zero JS bundle cost for that component
        ```
        
      </Tab>
      
      <Tab label="Common Patterns" icon={<span>🔧</span>}>
        
        <Table>
          <TableHeader>
            <TableCell>Scenario</TableCell>
            <TableCell>Should be</TableCell>
            <TableCell>Notes</TableCell>
          </TableHeader>
          <TableBody>
            <TableRow>
              <TableCell>Page layout shell</TableCell>
              <TableCell><Badge variant="success">Server</Badge></TableCell>
              <TableCell>No interactivity needed</TableCell>
            </TableRow>

</Stepper>

## Caching and Data Flow

<Callout type="info" title="Next.js Caching Layers">
  Next.js has multiple caching mechanisms. Understanding the difference prevents 90% of "why is my data stale?" confusion.
</Callout>

<Accordion type="single">
  <AccordionItem title="Cache Layers in Next.js (App Router)">
    
    <Timeline variant="default">
      <TimelineItem title="Request Memoization" status="completed">
        When you call `await fetch(...)` inside a Server Component, Next.js deduplicates identical requests *within a single render*. A second identical request in the same render returns the cached result *without hitting the network again*.
      </TimelineItem>
      <TimelineItem title="Data Cache" status="completed">
        Next.js extends `fetch` with a built-in data cache. Results are persisted (by default, forever-ish). Subsequent renders across requests read from the cache.
        ```tsx filename="fetch-options.ts"
// Force revalidation every hour
fetch('https://api.example.com/data', {
  next: { revalidate: 3600 } // seconds
});

// No revalidation = cache until you deploy
fetch('https://api.example.com/data');

// Always fresh (no caching)
fetch('https://api.example.com/data', {
  cache: 'no-store'
});
        ```
      </TimelineItem>
      <TimelineItem title="Full Route Cache" status="current">
        Static pages are rendered at build time and cached as HTML. A static page serves instantly—no server computation at request time.
      </TimelineItem>
      <TimelineItem title="Router Cache (Client-side)" status="pending">
        The client-side router cache stores rendered pages in memory (and localStorage for back/forward navigation). Navigating back feels instant because the page loads from cache.
      </TimelineItem>
    </Timeline>
    
  </AccordionItem>

  <AccordionItem title="Server Actions — Revalidation Made Simple">
    
    Server Actions blur the line between client and server. Call them from Client Components, but they execute on the server with full database access.
    
    ```tsx filename="server-action.ts"
'use server';
import { revalidatePath } from 'next/cache';
import { db } from '@/lib/db';

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string;
  await db.posts.create({ data: { title } });
  revalidatePath('/posts'); // Invalidate the posts cache
}
    ```
    
    <SuccessAlert title="Why Server Actions?">
      Before Server Actions, "create post" required building a full API route, handling auth in a separate layer, and revalidating on the client with messy cache-tag logic. Server Actions let you write server-side code inline — no API boilerplate, no client-side state management for the mutation itself.
    </SuccessAlert>
    
  </AccordionItem>

  <AccordionItem title="Streaming and Suspense — Progressive Rendering">
    
    Instead of waiting for all data to load, stream pieces of HTML as they become ready:
    
    ```tsx filename="streaming-page.tsx"
// Slow data component — this is the bottleneck
async function SlowChart() {
  const data = await fetch('/api/slow-data').then(r => r.json());
  return <Results data={data} />;
}

// Skeleton shows immediately, content streams in
export default async function DashboardPage() {
  return (
    <div>
      <header>Dashboard</header>          {/* ✅ Instant */}
      <RecentActivity />                   {/* ✅ Fast */}
      <Suspense fallback={<ChartSkeleton />}>  {/* Shows skeleton */}
        <SlowChart />                      {/* Streams when ready */}
      </Suspense>
    </div>
  );
}
    ```
    
    <InfoAlert title="How SSR + Streaming work together">
      Server Components still run on the server. The *difference* with Suspense is that the response streams chunks as they're ready. The shell sends immediately, slow data sends later. The user sees content progressively instead of waiting for one big HTML response.
    </InfoAlert>
    
  </AccordionItem>
</Accordion>

## Advanced Bundle Analysis

<Callout type="warning" title="Know What You're Shipping">
  A single unoptimized import can add 200KB+ to your client bundle. The bundle analyzer makes the invisible visible.
</Callout>

<Stepper>
  <Step title="Install @next/bundle-analyzer">
    
    Add it to your project. It generates an interactive treemap of every JS chunk:
    
    ```bash title="Install"
npm install -D @next/bundle-analyzer
    ```
    
  </Step>

  <Step title="next.config.js Setup">
    
    ```javascript filename="next.config.js"
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});
module.exports = withBundleAnalyzer({});
    ```
    
  </Step>

  <Step title="Run the Analyzer">
    
    ```bash title="Terminal"
ANALYZE=true npm run build
    ```
    
    Open the URL it prints (typically `http://localhost:3000/_next/analyze/...`). You'll see a treemap where bigger boxes = bigger bundles. Hover to see what's taking space.
    
  </Step>

  <Step title="Common Victims">
    
    <Table>
      <TableHeader>
        <TableCell>Anti-Pattern</TableCell>
        <TableCell>Problem</TableCell>
        <TableCell>Better approach</TableCell>
      </TableHeader>
      <TableBody>
        <TableRow>
          <TableCell><code>import {'{'} format {'}'} from 'date-fns'</code></TableCell>
          <TableCell>Pulls in the whole library (~100+ functions)</TableCell>
          <TableCell>Use <code>date-fns</code> tree-shaking + specific imports, or <code>dayjs</code></TableCell>
        </TableRow>
        <TableRow>
          <TableCell><code>import {'{' } debounce {'}'} from 'lodash'</code></TableCell>
          <TableCell>Whole lodash (~500KB)</TableCell>
          <TableCell>Use <code>lodash-es</code> or write a 3-line debounce</TableCell>
        </TableRow>
        <TableRow>
          <TableCell><code>import * as Icons from 'lucide-react'</code></TableCell>
          <TableCell>Pulls every icon but uses 1</TableCell>
          <TableCell>Use named imports: <code>import {'{' } ChevronRight {'}'} from 'lucide-react'</code></TableCell>
        </TableRow>
        <TableRow>
          <TableCell>Heavy chart libs at top level</TableCell>
          <TableCell>Never used above fold</TableCell>
          <TableCell>Dynamic import as shown in step 3</TableCell>
        </TableRow>
      </TableBody>
    </Table>
    
  </Step>
</Stepper>

## But Wait, There's More

<Accordion type="multiple">
  <AccordionItem title="Prefetching & Link">
    
    `<Link>` from `next/link` automatically prefetches pages linked in the viewport (when the user is on a fast connection):
    
    ```tsx filename="prefetch-link.tsx"
import Link from 'next/link';

<Link href="/blog/nextjs-performance">  {/* ✅ Prefetched when visible */}
  Read more
</Link>

<Link href="/heavy-page" prefetch={false}>  {/* ⬇️ Skip prefetch */}
  Heavy page (saves bandwidth on mobile)
</Link>
    ```
    
  </AccordionItem>

  <AccordionItem title="Parallel & Sequential Fetching">
    
    Fetching data in parallel cuts waterfalls dramatically:
    
    ```tsx filename="parallel-fetch.tsx"
// ❌ Sequential: 200ms + 300ms = 500ms total
async function SlowPage() {
  const user  = await fetch('/api/user').then(r => r.json());   // 200ms
  const posts = await fetch('/api/posts').then(r => r.json()); // 300ms
  return <Dashboard user={user} posts={posts} />;
}

// ✅ Parallel: max(200ms, 300ms) = 300ms total
async function FastPage() {
  const [user, posts] = await Promise.all([
    fetch('/api/user').then(r => r.json()),
    fetch('/api/posts').then(r => r.json()),
  ]);
  return <Dashboard user={user} posts={posts} />;
}
    ```
    
  </AccordionItem>

  <AccordionItem title="Route Segment Config">
    
    Control rendering behavior per-page with route segment config:
    
    ```tsx filename="route-config.tsx"
export const dynamic = 'force-static';  // Always static, even in dev
export const revalidate = 3600;           // Revalidate every hour
export const fetchCache = 'force-no-store'; // Never use data cache

// For one-off routes without creating a separate layout:
export const metadata = {
  title: 'My Page',
  robots: { index: false },  // Prevent indexing this page
};
    ```
    
  </AccordionItem>
</Accordion>

## TL;DR

<Callout type="idea" title="The top 4 things to do today">
  <ol>
    <li>Replace every `<img>` with `<Image>` — check for missing width/height</li>
    <li>Move font loading to <code>next/font</code></li>
    <li>Run <code>ANALYZE=true npm run build</code> and hunt the biggest chunks</li>
    <li>Find one dynamic candidate in your codebase and convert to <code>next/dynamic</code></li>
  </ol>
  <p>These four actions routinely shave 30-50% off bundle size and bring LCP from 6s down to under 2s.</p>
</Callout>
