Every web app needs to move data between the frontend and backend. Two dominant patterns have emerged over the last decade: **REST** and **GraphQL**. I've shipped production APIs with both, and the answer isn't "one is better." It depends on who's consuming the API and how complex the data relationships are.

---

## The Contenders

<Stepper>
  <Step title="REST (Representational State Transfer)">
    Standard HTTP methods (`GET`, `POST`, `PUT`, `DELETE`) across multiple endpoints. Each endpoint returns a server-defined data structure — you get what they give you.
  </Step>
  <Step title="GraphQL">
    A query language with a **single endpoint**. The client asks for exactly the data it needs — no more, no less.
  </Step>
</Stepper>

---

## The Key Differences

<Callout type="warning" title="1. Data Fetching (Over-fetching vs. Precision)">
**REST** returns whatever the server sends. Hit `/users/1` and you might get 50 fields back when all you needed was the user's `name`.
**GraphQL** lets you pick the fields you want. The response matches your query, nothing extra.
</Callout>

<Callout type="danger" title="2. The N+1 Problem (Number of Requests)">
Need a user, their posts, and their followers? With **REST** that's three separate requests. With **GraphQL** it's one.
</Callout>

/>

<Callout type="info" title="3. Versioning">
**REST** APIs manage versioning through URLs (`/api/v1/` vs `/api/v2/`). **GraphQL** sidesteps this — add new fields, deprecate old ones, no version bumps.
</Callout>

<Callout type="idea" title="My Take">
  If you're building something simple — a CRUD app, a public API for third-party devs, or an internal tool — start with REST. It's well-understood, caching works naturally, and you'll ship faster. If you're building a data-heavy app with multiple clients (web, iOS, Android) that each need different slices of the same data, GraphQL's upfront complexity pays for itself by the second client.
</Callout>
