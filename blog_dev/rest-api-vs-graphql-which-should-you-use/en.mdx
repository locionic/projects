## Introduction

When building a web application, one of the first decisions you'll face is how your frontend and backend communicate. Two popular options are **REST** and **GraphQL**. Both work well, but they shine in different situations.

In this post, I'll break down the key differences and help you choose the right one for your project.

---

## What Is REST?

**REST** (Representational State Transfer) is the traditional approach to building APIs. It uses standard HTTP methods and URLs to perform operations:
```
GET    /users          → get all users
GET    /users/1        → get user with id 1
POST   /users          → create a new user
PUT    /users/1        → update user with id 1
DELETE /users/1        → delete user with id 1
```

Each endpoint returns a fixed data structure defined by the server.

---

## What Is GraphQL?

**GraphQL** is a query language for APIs developed by Facebook. Instead of multiple endpoints, you have a single endpoint and the client specifies exactly what data it needs:
```graphql
query {
  user(id: 1) {
    name
    email
    posts {
      title
    }
  }
}
```

The server returns exactly what you asked for — nothing more, nothing less.

---

## Key Differences

### 1. Data Fetching

**REST** — fixed responses, may return too much or too little data:
```json
// GET /users/1 — returns everything even if you only need name
{
  "id": 1,
  "name": "Loc Truong",
  "email": "loc@example.com",
  "createdAt": "2026-01-01",
  "role": "admin",
  "preferences": { ... }
}
```

**GraphQL** — ask for exactly what you need:
```graphql
query {
  user(id: 1) {
    name  # only get name
  }
}
```

### 2. Number of Requests

**REST** — often requires multiple requests:
```
GET /users/1       → get user
GET /users/1/posts → get their posts
GET /users/1/followers → get their followers
```

**GraphQL** — one request gets everything:
```graphql
query {
  user(id: 1) {
    name
    posts { title }
    followers { name }
  }
}
```

### 3. Versioning

**REST** — requires versioning when API changes:
```
/api/v1/users
/api/v2/users
```

**GraphQL** — evolves without versioning by adding new fields.

---

## When to Use REST

REST is a great choice when:

- Building a **simple or small API**
- Your team is already familiar with REST
- You need **HTTP caching** out of the box
- You're building **public APIs** that others will consume
- You want **simplicity** and less tooling overhead

---

## When to Use GraphQL

GraphQL is a great choice when:

- You have a **complex data model** with many relationships
- Your frontend needs **flexible, custom queries**
- You're building for **multiple clients** (web, mobile) with different data needs
- You want to **reduce over-fetching** and improve performance
- You're building a **large-scale application**

---

## Real World Example

Imagine building a blog. With REST:
```
GET /posts          → list of posts
GET /posts/1        → single post
GET /posts/1/author → author details
GET /posts/1/comments → comments
```

With GraphQL:
```graphql
query {
  post(id: 1) {
    title
    content
    author { name }
    comments { text }
  }
}
```

GraphQL wins here with a single, clean request.

---

## Conclusion

Both REST and GraphQL are excellent choices — the best one depends on your project:

| | REST | GraphQL |
|---|---|---|
| Simplicity | ✅ Simpler | ❌ More setup |
| Flexibility | ❌ Fixed responses | ✅ Flexible queries |
| Caching | ✅ Built-in | ❌ Extra work |
| Multiple clients | ❌ Over/under-fetching | ✅ Perfect fit |
| Learning curve | ✅ Low | ❌ Higher |

For most small to medium projects, **REST is perfectly fine**. If you're building something complex with multiple clients and rich data relationships, **GraphQL is worth the investment**.
