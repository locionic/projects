# GitHub's Reliability Woes: How AI Load Broke the Platform
===========================================================

In recent months, GitHub has been experiencing reliability issues, with outages and data integrity incidents affecting its users. The platform's reliability has been measured at 86% this month, down from 90% last month. This article explores the reasons behind GitHub's struggles and how other vendors are managing to keep up with the increased load.

## Data Integrity Incident

On April 23, a bug was introduced that caused incorrect merge commits when using the squash merge method. This resulted in 2,092 pull requests being impacted, with companies like Modal and Zipline affected. The bug was eventually fixed, but not before causing significant disruptions to users.

## Outages Don't Stop

On April 27, pull requests and issues disappeared from GitHub's web UI due to an Elasticsearch outage on the backend. This was followed by other outages, including problems with GitHub Actions and incomplete pull requests in repositories.

## Security Issue

On April 28, a critical security issue was disclosed, allowing a bad actor to access all repositories on GitHub and GitHub Enterprise server using only a git push command. The issue was fixed within six hours, but GitHub Enterprise servers that were not updated remain vulnerable.

## Famous Open Source Contributor Quits GitHub

Mitchell Hashimoto, founder of HashiCorp, announced that he was quitting GitHub due to its reliability issues. He stated that the platform was no longer suitable for professional work and that he was moving to Ghostty, an open-source terminal.

## CTO Blames AI Agent-Fueled Load Spike

GitHub CTO Vlad Fedorov attributed the reliability issues to a sudden increase in load caused by AI agents. He stated that the load increase was 3.5x over two years, which is a significant challenge for the platform.

## Why GitHub Struggles with AI Load

GitHub's struggles with AI load can be attributed to several factors, including:

*   **Tech debt**: Many of GitHub's systems are over 10 years old and are patched up, making them difficult and risky to change.
*   **Organizational debt**: With around 4,000 employees, including 1,000 engineers, GitHub has a significant organizational overhead.
*   **Customer expectations**: GitHub cannot break customer workflows, even if it means making changes to its systems more slowly.

## Other Vendors Manage to Keep Up

Despite the challenges posed by AI load, other vendors like Vercel, Linear, Resend, Railway, and Sentry are managing to keep up with the increased load. This raises questions about why GitHub is struggling to cope with the same challenges.

## Conclusion

GitHub's reliability issues are a significant concern for its users, and the platform's struggles with AI load are a major factor. While other vendors are managing to keep up with the increased load, GitHub's tech debt, organizational overhead, and customer expectations are making it harder for the platform to adapt.
