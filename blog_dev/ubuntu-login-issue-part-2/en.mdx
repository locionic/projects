I sat staring at a frozen desktop for two full minutes after typing my password. Every single morning. For three days straight.

<Callout type="idea" title="Key Insight">
Boot time and login time are completely different problems. `systemd-analyze` measures boot speed — not what happens after you type your password.
</Callout>

---

## The First Clue: NetworkManager

Ran `systemd-analyze blame` and found `NetworkManager-wait-online.service` sitting at 5.7 seconds. 

<Callout type="info" title="The Fix">
This service waits for a full network connection before considering boot complete. On a desktop? Pointless.
```bash
sudo systemctl disable NetworkManager-wait-online.service
```
</Callout>

Boot got faster. The login freeze didn't budge.

---

## The Second Clue: Duplicate Keyring

`journalctl -b` showed a cluster of errors exactly two minutes after login:

```
Failed to start app-gnome-gnome-keyring-secrets.scope
```

<Stepper>
<Step title="The Problem">
GNOME was starting the keyring twice — once via systemd, again via `.desktop` autostart files. The second attempt crashed and left systemd confused.
</Step>

<Step title="The Fix">
Disable the duplicate autostart entries locally:
```bash
mkdir -p ~/.config/autostart
for f in gnome-keyring-pkcs11.desktop gnome-keyring-secrets.desktop gnome-keyring-ssh.desktop; do
    cp /etc/xdg/autostart/$f ~/.config/autostart/$f
    echo "X-GNOME-Autostart-enabled=false" >> ~/.config/autostart/$f
done
```
</Step>
</Stepper>

The cgroup errors disappeared. The freeze? Still there.

---

## The Real Culprit: snapd's Math

Digging deeper into the journal revealed the actual culprit:

```
snapd[2242]: adjusting startup timeout by 2m0s (pessimistic estimate of 30s plus 5s per snap)
```

<Callout type="warning" title="Snapd's Math Problem">
Snapd calculates startup timeout as **30 seconds + 5 seconds per installed snap**. I had 18 snaps installed. 30 + (5 × 18) = 120 seconds exactly.
</Callout>

The `snapd-desktop-integration` snap was failing at login. Snapd waited the full two minutes before giving up.

### The Nuclear Option

`snapd-desktop-integration` only handles theme and font syncing. Purely cosmetic.

```bash
sudo snap remove snapd-desktop-integration
```

Desktop loaded instantly after that.

---

<Callout type="idea" title="Diagnostic Cheat Sheet">
**Boot breakdown:** `systemd-analyze blame | head -20`  
**User session breakdown:** `systemd-analyze --user blame | head -20`  
**Login failures:** `journalctl -b | grep -iE "timeout|timed out|failed"`
</Callout>

---

Three separate bugs. Two minutes of waiting. One `snap remove` command fixed it.

Why does snapd add five seconds per snap? No idea. But removing that one integration snap bought me back two minutes every morning. Worth it.
