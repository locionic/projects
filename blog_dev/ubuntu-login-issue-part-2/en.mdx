Staring at a frozen Ubuntu desktop for two full minutes after entering your password? This post documents how I unraveled a web of **multiple overlapping bugs** causing this exact issue.

<Callout type="idea" title="Key Insight">
Boot time and login time are entirely separate problems! `systemd-analyze` measures boot speed, not what happens *after* you log in.
</Callout>

---

## 1. The NetworkManager Timeout

First, `systemd-analyze blame` revealed `NetworkManager-wait-online.service` was stalling the boot for 5.7 seconds.

<Callout type="info" title="The Fix">
This service waits for a full network connection before booting. Completely unnecessary for a desktop!
```bash
sudo systemctl disable NetworkManager-wait-online.service
```
*Result: Faster boot, but the post-login freeze remained.*
</Callout>

---

## 2. Duplicate `gnome-keyring` Autostart

Running `journalctl -b` exposed a critical cluster of errors exactly 2 minutes after login:
`Failed to start app-gnome-gnome-keyring-secrets.scope`

<Stepper>
  <Step title="The Problem">
    GNOME was starting the keyring **twice** — once via systemd, and again via `.desktop` autostart files. The second attempt crashed and confused systemd.
  </Step>
  <Step title="The Fix">
    Disable the redundant autostart entries locally without breaking the system:
    ```bash
    mkdir -p ~/.config/autostart
    for f in gnome-keyring-pkcs11.desktop gnome-keyring-secrets.desktop gnome-keyring-ssh.desktop; do
        cp /etc/xdg/autostart/$f ~/.config/autostart/$f
        echo "X-GNOME-Autostart-enabled=false" >> ~/.config/autostart/$f
    done
    ```
  </Step>
</Stepper>

*Result: Cgroup errors vanished, but incredibly, the freeze STILL persisted!*

---

## 3. The `snapd` Pessimistic Timeout

Diving deeper into the journals, I found the final boss:
`snapd[2242]: ... adjusting startup timeout by 2m0s (pessimistic estimate of 30s plus 5s per snap)`

<Callout type="warning" title="Snapd's Math Problem">
Snapd calculates startup timeouts based on installed snaps: **30 seconds + 5s per snap**. With 18 snaps, that's exactly 2 minutes! 
The `snapd-desktop-integration` snap was failing at login, triggering snapd to hold the full timeout before giving up.
</Callout>

### The Final Fix
Since `snapd-desktop-integration` is purely cosmetic (syncing themes/fonts), nuking it was the easiest solution:
```bash
sudo snap remove snapd-desktop-integration
```

**And boom.** The desktop loaded instantly. 

<Callout type="idea" title="Pro Tip: Diagnostic Cheat Sheet">
- **Boot Breakdown**: `systemd-analyze blame | head -20`
- **User Session Breakdown**: `systemd-analyze --user blame | head -20`
- **Login Failures**: `journalctl -b | grep -iE "timeout|timed out|failed"`
</Callout>
