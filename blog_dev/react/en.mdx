# React in Practice

I've been building React apps for a few years now, and the things that tripped me up weren't the concepts themselves — they were the small misunderstandings that caused subtle bugs. This guide covers the patterns I use every day and the gotchas I've learned the hard way.

## State With `useState`

The most common bug I see (and used to write) is calling `setCount(count + 1)` multiple times and expecting it to stack:

```jsx
const [count, setCount] = useState(0);

function handleClick() {
  setCount(count + 1);
  setCount(count + 1); // Still increments by 1, not 2
}
```

React batches state updates. Both calls see the same `count` value (0), so both set it to 1. The fix is using the callback form:

```jsx
setCount(prev => prev + 1);
setCount(prev => prev + 1); // Now increments to 2
```

This matters most in event handlers and effects where multiple state updates happen in sequence.

## Side Effects With `useEffect`

The cleanup function in `useEffect` confused me at first. It's not sequential code — you're handing React a function to call later, when the component unmounts or before re-running the effect.

```jsx
useEffect(() => {
  const timer = setTimeout(() => console.log('Ping!'), 1000);
  return () => clearTimeout(timer); // React calls this on cleanup
}, []);
```

I've seen a pattern where people try to make cleanup conditional:

```jsx
return () => {
  if (something) clearTimeout(timer); // Don't do this
};
```

The cleanup function should always clean up, unconditionally. React runs it reliably — make sure it handles the case where the effect may not have fully initialized.

## Avoiding Prop Drilling With Context

Context is useful but has a sharp edge: every component consuming a context re-renders when any value in it changes, not just the value it reads.

```jsx
const UserContext = createContext({ user: null, settings: {} });

// In a component:
const { user } = useContext(UserContext);
// This re-renders even if only `settings` changed
```

For simple cases this doesn't matter. But for global state that changes frequently, I use Zustand instead — it uses subscriptions so only the components reading the changed value re-render.

<Callout type="info" title="My State Management Decision Tree">
- Local UI state → `useState`
- App-wide state that changes rarely → React Context
- App-wide state that changes often → Zustand
- Server data with caching → TanStack Query (React Query)
- URL state → Next.js searchParams or useRouter
</Callout>

## Next.js App Router

The App Router flips the mental model: everything is a Server Component by default. Server Components can't use hooks, can't handle user interactions, and don't ship JavaScript to the browser.

When you need interactivity, add `"use client"` at the top. The rule I follow: keep components server by default and only add `"use client"` when you specifically need `useState`, `useEffect`, event handlers, or browser APIs.

```tsx
// Server Component by default
async function Post({ slug }) {
  const post = await getPost(slug);
  return <article>{post.content}</article>;
}

// Client Component when interactivity is needed
"use client";
function LikeButton({ postId }) {
  const [liked, setLiked] = useState(false);
  return <button onClick={() => setLiked(!liked)}>Like</button>;
}
```

This pattern alone reduced my client-side JavaScript bundle by about 40% compared to the old Pages Router where everything was a client component.
