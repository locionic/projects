# Migrating from Vim to Helix: A Three-Month Review

As a long-time Vim user, I recently decided to give Helix a try, and after three months of use, I'm excited to share my thoughts on the experience.

## Why Helix?

One of the primary reasons I switched to Helix was its built-in language server support. After 20 years of using Vim and Neovim, I found that setting up a working language server configuration was too much work. Helix's out-of-the-box language server support has been a game-changer, allowing me to easily rename symbols, go to definitions, and more.

## Standout Features

### Excellent Search

Helix's search functionality is one of its strongest features. When searching for a string across multiple files, Helix provides a preview of the surrounding context, making it easier to identify the relevant results. This is a significant improvement over the vim-ripgrep plugin I previously used.

### Quick Reference

Helix's quick reference feature, accessible by pressing `g`, provides a helpful popup with suggestions for where to go next. This has been particularly useful for me, as I don't often use the "go to definition" or "go to reference" features and tend to forget the keyboard shortcuts.

## Vim to Helix Translations

While Helix has a different set of keybindings and features than Vim, I've found some useful translations:

* Instead of using marks (`ma`, `'a`), I use `Ctrl+O` and `Ctrl+I` to navigate to previous or next cursor locations.
* I've replaced macros with multiple cursors, which I find more intuitive and efficient.
* To batch change text, I use the `%` key to highlight everything, then `s` to select the desired text using a regex.
* Helix's buffer switcher (`<space>b`) is a great alternative to Neovim-style tabs.

## Annoyances and Limitations

While Helix has been a great experience overall, there are a few areas where it falls short:

* The `:reflow` command doesn't work as well as Vim's `gq` for reflowing text, particularly with lists.
* There's no persistent undo feature, which means I have to use a workaround to maintain my undo history.
* Files don't autoreload after changes on disk, requiring a manual reload using `:reload-all`.
* Occasionally, Helix panics, which can be frustrating.

## Switching Was Easier Than Expected

Despite my initial concerns about relearning 20 years of Vim muscle memory, I found the transition to Helix to be relatively smooth. Starting with a low-stakes project helped me get accustomed to the new keybindings and features. While there are still some differences that throw me off, I've found that learning the "Helix way" has been more efficient than trying to force Vim-like keybindings.

## Terminal-Based Workflow

After years of using a GUI version of Vim, I decided to switch to a terminal-based workflow with Helix. This has involved:

* Creating a separate terminal window for each project
* Making the Helix tab the first tab in the terminal window
* Using the same working directory for all tabs in the window

This setup has worked well for me, and I might even prefer it to my previous workflow.

## Configuration

One of the benefits of Helix is its simple configuration process. My configuration consists of just a few keyboard shortcuts and a theme setting. I've also set up language preferences in a separate `languages.toml` file.

```toml
theme = "solarized_light"

[editor]
default-yank-register = "+"

[keys.normal]
"^" = "goto_first_nonwhitespace"
"$" = "goto_line_end"

[keys.select]
"^" = "goto_first_nonwhitespace"
"$" = "goto_line_end"

[keys.normal.space]
l = ":reflow"
```

```toml
[[language]]
name = "python"
formatter = { command = "black", args = ["--stdin-filename", "%{buffer_name}", "-"] }
language-servers = ["pyright"]
auto-format = false
```

## Conclusion

While it's only been three months, I'm excited about the potential of Helix as a modern text editor. While there are some areas where it falls short, the benefits of its streamlined configuration process, excellent search functionality, and built-in language server support make it a compelling choice for developers.
