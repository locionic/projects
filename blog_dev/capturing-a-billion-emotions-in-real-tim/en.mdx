# Capturing a Billion Emotions in Real-time

In this article, we will explore how Hotstar built a scalable system to capture and display real-time user emotions through emojis, handling billions of submissions during a tournament. We will discuss the architecture, design principles, and implementation details of the system.

## High-Level Design

The system consists of the following components:

*   Client: Users submit emojis through the Hotstar app or website.
*   Load Balancer: Distributes incoming traffic across multiple servers.
*   API Server: Handles client requests, writes data to a message queue, and responds to clients.
*   Message Queue (Kafka): Stores data from API servers and feeds it to processing applications.
*   Processing Application (Spark): Consumes data from Kafka, computes aggregates, and writes results to another Kafka queue.
*   Delivery Mechanism (PubSub): Delivers processed data to clients in real-time.

## Key Design Principles

1.  **Scalability**: The system should be horizontally scalable to support increasing traffic.
2.  **Decomposition**: Break down the system into smaller components, each performing a specific task, allowing for independent scaling.
3.  **Asynchronous Processing**: Enable execution without blocking resources, supporting higher concurrency.

## Implementation

### Handling Client Requests

*   Clients send user-submitted emojis via HTTP API.
*   API servers write data to a message queue (Kafka) using asynchronous processing to prevent blocking client connections.
*   Golang is used for API server implementation, leveraging Goroutines and Channels for concurrent message writing to Kafka.

### Message Queue

*   Kafka is chosen for its high throughput, availability, low latency, and support for consumer groups.
*   To simplify Kafka management, Hotstar's data platform (Knol) is used, built on top of Kafka.

### Processing Application

*   Spark is selected for its support for micro-batching and aggregations, essential for real-time processing.
*   A Spark streaming job computes aggregates over a batch of 2 seconds and writes results to another Kafka queue.

### Delivery Mechanism

*   PubSub is used as the delivery mechanism, a real-time messaging infrastructure built at Hotstar.
*   A simple Kafka consumer in Python is used to receive processed data, perform normalization, and send top emojis to PubSub.
*   Clients receive messages from PubSub and display emoji animations to users.

## Impact

*   The system captured over 5 billion emojis from 55.83 million users during the ICC Cricket World Cup 2019.
*   The system has captured more than 6.5 billion emojis to date.
*   The infrastructure built for emojis is extended to support voting, polls, and trivia contests, handling over 3 billion votes.

By building this system in-house, Hotstar achieved scalability, performance, and cost-effectiveness, enabling the company to provide a seamless and engaging user experience.
