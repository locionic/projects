まだ見かけます。Pythonの「アンダースコア」をアクセス修飾子みたいに扱う人たち。しかもだいたい最後は同じで、ややこしいエラーが出て、「本当はprivateにしたかったのに、できてないじゃん」みたいなコードになる。

結論から言うと、PythonはJavaやC#みたいにアクセス制御を強制しません。`private` / `protected` / `public` みたいな本物の壁がない。代わりに、アンダースコアは主に「名前のシグナル」です。そして例外的に、ダブルアンダースコアだけ特別な挙動があります。

### シングルアンダースコア：基本「君のためじゃない」（たぶん）
例えばこんなふうに書くとします。

```python
class User:
    def _helper(self):
        ...
```

先頭のアンダースコアは、`_helper` を内部用途として想定している、というかなり強いヒントになります。Pythonは呼ぶのを止めません。

```python
u = User()
u._helper()
```

例外も出ないし、ロックもかからないし、魔法みたいなことも起きない。ただの「慣習」であって、他の人間（と一部のツール）が尊重してくれる、というだけです。

### ダブルアンダースコア：プライバシーというより名前修飾（name mangling）
次はこれ。

```python
class User:
    def __secret(self):
        ...
```

これは単なる「ヒント」ではありません。Pythonは名前修飾を行って、メソッドが同じ属性名として保存されないようにします。概念的には、`__secret` はだいたい `_User__secret` みたいな別名に化ける感じです。

だからこれは動きません。

```python
u = User()
u.__secret()  # AttributeError
```

でも、外部から完全にブロックされているわけではありません。変換後の名前を知っていれば、普通に到達できます。

```python
u._User__secret()
```

要するに、「アクセスしにくくなる」だけで、「アクセス不能になる」わけではないんですよね。

### この誤解が残り続ける理由
混乱の原因は、「言語に強制されたカプセル化」があると期待してしまうことです。Pythonではアンダースコアは主にコミュニケーションのためのもの。

- 「内部実装の都合」なのか  
- 「サポートされている公開インターフェース」なのか  

その違いを伝えるために使われます。

つまり、`__name` みたいなものがあっても、Pythonは本当の意味でのプライバシーとは別物をやってます。事故を減らすための“名前の衝突回避”や、サブクラス化を安全にするための仕組みが中心で、「実行時のアクセスゲート」として働くものではない。

### じゃあどうすればいい？
「どういう使われ方を想定してるか」を、アンダースコアで正直に伝えるのが一番です。

- `_name`: 内部寄り。よほど意図があるのでない限り、頼らないでね  
- `__name`: 名前修飾で衝突を避ける。ただし“本当の秘匿”ではない  
- アンダースコアなし：通常の利用を想定してる

そして、「Pythonでこれをどうやってprivateにできる？」と自分が思ってしまったら、たいていは次の問いに言い換える方がいいです。

「これは安定したAPI機能なの？それとも、後で変えるかもしれない実装の都合？」

その切り替えだけで、かなりの摩擦が消えます。
