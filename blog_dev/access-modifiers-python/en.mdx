I still see people treating Python “underscores” like they’re access modifiers, and it always ends the same way: confusing errors, and code that feels like it should’ve been private—but isn’t.

Here’s the deal: Python doesn’t enforce access control the way Java or C# does. There’s no real `private` / `protected` / `public` barrier. Instead, underscores are mostly naming signals—with one special behavior for double underscores.

### Single underscore: “not for you” (mostly)
When you write something like:

```python
class User:
    def _helper(self):
        ...
```

that leading underscore is a strong hint that `_helper` is meant for internal use. Python won’t stop you from calling it:

```python
u = User()
u._helper()
```

No exception, no lock, no magic. It’s a convention that other humans (and some tooling) will respect.

### Double underscore: name mangling, not privacy
Then there’s:

```python
class User:
    def __secret(self):
        ...
```

This isn’t just “a hint.” Python performs name mangling so the method isn’t stored under the same attribute name. Conceptually, `__secret` becomes something like `_User__secret`.

So this won’t work:

```python
u = User()
u.__secret()  # AttributeError
```

But it’s not actually blocked from the outside. If you know the mangled name, you can still reach it:

```python
u._User__secret()
```

Bottom line: it’s “harder to access,” not “blocked.”

### Why this misunderstanding sticks
The confusion comes from expecting language-enforced encapsulation. In Python, underscores are primarily about communication: “internal implementation detail” versus “part of the supported interface.”

So even with `__name`, Python is doing something different than true privacy. It’s mostly preventing accidental collisions and making subclassing safer—not acting like a runtime access gate.

### What to do instead
Use underscores to tell the truth about how you expect the code to be used:

- `_name`: internal-ish; don’t rely on it unless you really mean to  
- `__name`: avoid name collisions via mangling; still not true privacy  
- no underscore: intended for normal use

And if you catch yourself asking “how do I make this private in Python?”, it’s usually a better question to ask “is this a stable API feature, or an implementation detail that I might change later?” That shift clears up a ton of friction.
