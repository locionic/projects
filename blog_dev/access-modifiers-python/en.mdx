<Callout type="warning" title="Common Misconception">
  I still see people treating Python "underscores" like they're access modifiers. This always ends in confusing errors and code that feels like it should've been private—but isn't.
</Callout>

Here's the deal: Python doesn't enforce access control like Java or C#. There's no real `private`, `protected`, or `public` barrier. Instead, underscores are **naming signals**—with one special behavior for double underscores.

<Stepper>
  <Step title="Single Underscore: 'Not for you'">
    When you write `_helper(self):`, it's a strong hint that the method is for internal use. Python won't stop you from calling it (`u._helper()`), but it's a convention other devs should respect.
  </Step>
  <Step title="Double Underscore: Name Mangling">
    When you write `__secret(self):`, Python performs name mangling, conceptually changing the attribute name to `_User__secret`. It's harder to access, but definitely not blocked! 
  </Step>
</Stepper>

<Callout type="info" title="Why the Misunderstanding?">
  Developers coming from other languages expect language-enforced encapsulation. In Python, underscores simply communicate: *"internal implementation detail"* versus *"part of the supported interface."*
</Callout>

### What To Do Instead
Use underscores to tell the truth about how your code should be used:
- `_name`: Internal-ish; don't rely on it.
- `__name`: Avoid name collisions via mangling.
- **no underscore**: Intended for normal use!

<Callout type="idea" title="Pro Tip">
  Instead of asking, "How do I make this private?", ask, "Is this a stable API feature or an implementation detail?" That shift clears up a ton of friction!
</Callout>
