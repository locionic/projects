<Alert type="warning" title="Common Misconception">
  I still see people treating Python "underscores" like they're access modifiers from Java or C#. This always ends in confusing errors and code that feels like it should've been private—but isn't.
</Alert>

Here's the deal: **Python doesn't enforce access control**. There's no real `private`, `protected`, or `public` barrier. Instead, underscores are **naming signals**—with one special behavior for double underscores that trips up almost everyone.

<Stepper>
  <Step title="No Underscore → Public API">
    This is your contract with users. Changing it is a breaking change.
    
    ```python
    class User:
        def get_name(self) -> str:      # ✅ Public: safe to use
            return self.name
        
        def calculate_age(self) -> int: # ✅ Public: part of API
            return 2024 - self.birth_year
    ```
  </Step>

  <Step title="Single Underscore `_name` → Internal Convention">
    A *gentleman's agreement* that this is for internal use. Python won't stop you, but linters and IDEs will warn you.
    
    ```python
    class User:
        def _validate_email(self) -> bool:  # 🔒 Internal: don't call directly
            return "@" in self.email
        
        def _cache_key(self) -> str:        # 🔒 Internal: implementation detail
            return f"user:{self.id}"
    ```
    
    <Alert type="info" title="What single underscore actually does">
      <ul>
        <li>Signals "internal use only" to other developers</li>
        <li>Excluded from `import *` by default</li>
        <li>Linters (ruff, pyflakes) will warn on external access</li>
        <li><strong>Zero runtime enforcement</strong> — you can still call it!</li>
      </ul>
    </Alert>
    
    <Quiz
      question="What does a single underscore prefix (`_name`) actually enforce at runtime?"
      options={[
        "Prevents external access completely",
        "Only signals internal use — no runtime enforcement",
        "Makes the attribute private to the class",
        "Triggers an error if accessed from outside"
      ]}
      correctIndex={1}
      explanation="A single underscore is purely a naming convention. Python doesn't enforce any access restrictions — it's a signal to developers and linters, but `obj._internal` works perfectly fine at runtime."
    />
  </Step>

  <Step title="Double Underscore `__name` → Name Mangling">
    This *actually changes the name* at runtime to prevent accidental collisions in inheritance—not to enforce privacy.
    
    ```python
    class Base:
        def __secret(self) -> str:
            return "base secret"
        
        def call_secret(self) -> str:
            return self.__secret()  # Works! Calls Base.__secret
    
    class Derived(Base):
        def __secret(self) -> str:     # Different name: _Derived__secret
            return "derived secret"
        
        def call_both(self) -> tuple:
            return (self.__secret(), super().call_secret())
            # Returns: ("derived secret", "base secret")
    ```
    
    <Alert type="danger" title="Name mangling is NOT privacy">
      You can still access it: `obj._Base__secret()` works perfectly fine. It's just harder to do *by accident*.
    </Alert>
    
    <Quiz
      question="What happens when a subclass defines `__secret` with the same name as a parent class?"
      options={[
        "The parent's __secret is completely overridden",
        "Both exist separately due to name mangling (_Parent__secret vs _Child__secret)",
        "Python raises a NameError",
        "The child inherits the parent's __secret automatically"
      ]}
      correctIndex={1}
      explanation="Name mangling renames `__secret` to `_ClassName__secret` in each class. So `Parent.__secret` becomes `_Parent__secret` and `Child.__secret` becomes `_Child__secret` — they're completely separate attributes."
      code={`class Parent:
    def __secret(self): return "parent"

class Child(Parent):
    def __secret(self): return "child"

c = Child()
print(c._Parent__secret())  # "parent" - still accessible!
print(c._Child__secret())   # "child"`}
    />
  </Step>
</Stepper>

    <Flashcard term="Name Mangling" definition="Python renames __private to _ClassName__private at runtime to prevent accidental name collisions in inheritance chains. It's not for privacy — it's for avoiding conflicts." category="Python Internals" />
    
    <Flashcard term="Single Underscore" definition="A convention meaning 'internal use only'. Zero runtime enforcement — just a signal to developers and linters. Excluded from import * by default." category="Python Conventions" />
    
    <Flashcard term="Magic Methods" definition="Methods like __init__, __str__, __len__ that implement Python protocols. Called automatically by Python internals (e.g., str(obj) calls __str__)." category="Python Protocols" />
    
    ```python title="Name Mangling Demo" filename="mangling_demo.py" showLineNumbers
class Demo:
    def __init__(self):
        self.public = "public"
        self._internal = "internal"
        self.__private = "private"

d = Demo()

# These all work:
print(d.public)           # "public"
print(d._internal)        # "internal" (with linter warning)
print(d._Demo__private)   # "private" — name mangled!

# This fails:
# print(d.__private)      # AttributeError!
    ```
    
    <Alert type="note" title="Try it yourself">
      Run this in a Python REPL to see the mangled names: <code>dir(d)</code> will show <code>_Demo__private</code>.
    </Alert>
    
  </Tab>

  <Tab label="Inheritance Pitfalls" icon={<span>🌳</span>}>
    
    ```python title="The Classic Inheritance Bug" filename="inheritance_bug.py" showLineNumbers
class Parent:
    def __init__(self):
        self.__value = 42
    
    def get_value(self):
        return self.__value

class Child(Parent):
    def __init__(self):
        super().__init__()
        self.__value = 100  # Creates _Child__value, separate!
    
    def get_both(self):
        return (self.__value, super().get_value())
        # Returns (100, 42) — two different variables!

c = Child()
print(c.get_both())  # (100, 42)
print(c._Parent__value)  # 42 — still accessible!
print(c._Child__value)   # 100
    ```
    
    <Alert type="warning" title="When to actually use double underscore">
      Only use <code>__name</code> when you <strong>specifically</strong> need to avoid name collisions in a class hierarchy. For "private" code, single underscore is the right choice.
    </Alert>

## Real-World Patterns

<Accordion type="multiple">
  <AccordionItem title="The `_protected` Pattern for Extensible Classes" defaultOpen>
    
    Use single underscore for methods subclasses *should* override:
    
    ```python filename="extensible.py" showLineNumbers
class DatabaseConnection:
    def connect(self) -> Connection:
        """Public API: establish connection."""
        self._validate_config()
        conn = self._create_connection()
        self._post_connect_setup(conn)
        return conn
    
    # Override these in subclasses:
    def _validate_config(self) -> None:
        """Check required config exists."""
        if not self.host:
            raise ValueError("Host required")
    
    def _create_connection(self) -> Connection:
        """Actual connection logic — override for different DBs."""
        return pg_connect(self.host, self.user, self.password)
    
    def _post_connect_setup(self, conn: Connection) -> None:
        """Hook for subclasses to run after connect."""
        conn.set_timezone("UTC")
```
    
    <SuccessAlert title="Why this works">
      Subclasses override `_create_connection` for MySQL, MongoDB, etc. without touching the public `connect()` method. The single underscore says "override me" without enforcing it.
    </SuccessAlert>
    
    <Quiz
      question="Why use single underscore (`_create_connection`) instead of double underscore for methods subclasses should override?"
      options={[
        "Double underscore prevents overriding entirely",
        "Single underscore signals 'override me' without name mangling complications",
        "Double underscore is only for magic methods",
        "Single underscore makes it truly private"
      ]}
      correctIndex={1}
      explanation="Single underscore is a convention saying 'this is for internal use/overriding'. Double underscore would mangle the name in each subclass, making the override pattern confusing and breaking the super() chain."
    />
  </AccordionItem>

  <AccordionItem title="The `__slots__` Optimization">
    
    Double underscore in `__slots__` is a special case — it's a class-level declaration, not name mangling:
    
    ```python filename="slots_example.py" showLineNumbers
class Point:
    __slots__ = ('x', 'y', '_cached_hash')  # No __dict__ created!
    
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y
        self._cached_hash = None  # Internal cache
    
    def __hash__(self) -> int:
        if self._cached_hash is None:
            self._cached_hash = hash((self.x, self.y))
        return self._cached_hash

# Memory efficient: no __dict__ per instance
p = Point(1, 2)
# p.z = 3  # AttributeError — can't add arbitrary attributes
```
    
    <InfoAlert title="Performance note">
      <code>__slots__</code> saves ~40-50% memory per instance by avoiding <code>__dict__</code>. Use for classes with many instances (dataclasses, game entities, data rows).
    </InfoAlert>
    
  </AccordionItem>

  <AccordionItem title="Testing Private Methods">
    
    <Tabs defaultTab="Don't Test Private">
      <Tab label="Don't Test Private" icon={<span>❌</span>}>
        
        <WarningAlert title="Testing `_private` methods couples tests to implementation">
          If you test `_validate_email()` directly, refactoring it breaks tests even if behavior is correct.
        </WarningAlert>
        
      </Tab>
      
      <Tab label="Test via Public API" icon={<span>✅</span>}>
        
        ```python filename="test_public_api.py" showLineNumbers
import pytest
from myapp import UserService

class TestUserService:
    def test_create_user_validates_email(self):
        # Test THROUGH public API, not _validate_email directly
        service = UserService()
        
        with pytest.raises(ValueError, match="Invalid email"):
            service.create_user(email="not-an-email")
        
        # Valid email works
        user = service.create_user(email="valid@example.com")
        assert user.email == "valid@example.com"
```
        
        <SuccessAlert title="Benefits">
          <ul>
            <li>Tests survive refactoring of internal implementation</li>
            <li>Tests document expected behavior, not internal structure</li>
            <li>You can change `_validate_email` → `_check_email_format` freely</li>
          </ul>
        </SuccessAlert>
        
      </Tab>
      
      <Tab label="When You Must" icon={<span>⚠️</span>}>
        
        <Alert type="note" title="Rare exceptions where testing private is OK">
          <ul>
            <li>Complex algorithms with many edge cases (cryptography, compression)</li>
            <li>Legacy code you can't refactor yet</li>
            <li>Performance-critical paths where public API adds overhead</li>
          </ul>
          <p>In these cases, import the module and test directly: <code>from mymodule import _internal_helper</code></p>
        </Alert>
        
      </Tab>
    </Tabs>
    
    <Quiz
      question="What's the main problem with testing private methods directly?"
      options={[
        "It makes tests run slower",
        "It couples tests to implementation details — refactoring breaks tests even when behavior is correct",
        "Private methods can't be imported",
        "It violates encapsulation"
      ]}
      correctIndex={1}
      explanation="When you test `_validate_email()` directly, any internal refactoring (renaming, changing signature, splitting into multiple helpers) breaks the test even though the public behavior is unchanged. Tests should verify behavior through the public API."
    />
  </AccordionItem>
</Accordion>

## TL;DR

<Callout type="idea" title="Remember">
  <strong>Python trusts you.</strong> Underscores are communication tools, not walls.
  <ul>
    <li><code>name</code> → "Use this freely"</li>
    <li><code>_name</code> → "Internal detail, may change"</li>
    <li><code>__name</code> → "Won't clash with subclass names"</li>
    <li><code>__name__</code> → "Magic method for Python protocols"</li>
  </ul>
</Callout>
