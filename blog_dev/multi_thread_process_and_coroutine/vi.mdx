Ba khái niệm này hay gây nhầm lẫn vì tên nghe có vẻ giống nhau, nhưng bản chất rất khác nhau. Bài này sẽ giải thích từng cái theo thứ tự, bắt đầu từ vấn đề mà chúng giải quyết.

## Vấn đề cốt lõi: GIL

Trước hết cần hiểu **GIL** (Global Interpreter Lock) — cái khóa của Python chỉ cho **một thread chạy tại một thời điểm**.

Đây là thiết kế có chủ đích để bảo vệ bộ nhớ nội bộ của Python khỏi bị hỏng khi nhiều thread cùng truy cập. Nhưng hệ quả là: dù máy bạn có 8 core, Python mặc định chỉ dùng 1 core tại một lúc.

Ba công cụ dưới đây là ba cách khác nhau để đối phó với giới hạn này.

## Threading

### Ý tưởng

Threading tạo ra nhiều thread trong cùng một process, chia sẻ chung bộ nhớ và GIL. Nghe có vẻ không hiệu quả — nhưng có một ngoại lệ quan trọng.

**Khi thread chờ I/O (đọc file, gọi API, query database), nó nhả GIL ra.** Lúc đó thread khác có thể chạy.

### Tại sao hiệu quả cho I/O-bound?

Giả sử bạn cần tải 3 file từ internet, mỗi file mất 3 giây:

- **Sequential**: tổng 9 giây — trong lúc chờ file 1 về, CPU ngồi không.
- **Threading**: tổng ~3 giây — trong lúc thread 1 chờ, thread 2 và 3 gửi request của họ.

CPU không hề làm thêm việc gì — nó chỉ ngừng ngồi chờ vô ích.

```python
import threading
import requests

def fetch(url):
    r = requests.get(url)   # I/O → nhả GIL
    print(r.status_code)

urls = ["https://a.com", "https://b.com", "https://c.com"]
threads = [threading.Thread(target=fetch, args=(u,)) for u in urls]
[t.start() for t in threads]
[t.join()  for t in threads]
```

### Khi nào KHÔNG dùng threading?

Nếu task của bạn là **CPU-bound** (tính toán nặng, vòng lặp số học, xử lý ảnh), threading không giúp ích gì — GIL vẫn chỉ cho 1 thread chạy, kết quả không khác gì chạy tuần tự.

> **Quy tắc nhớ nhanh:** Task hay *chờ* (mạng, disk, DB)? → Threading. Task hay *tính* (số học, vòng lặp nặng)? → Không phải threading.

## Multiprocessing

### Ý tưởng

Thay vì nhiều thread trong 1 process, multiprocessing tạo ra **nhiều process độc lập**. Mỗi process có Python interpreter riêng, bộ nhớ riêng — và **GIL riêng**.

Kết quả: mỗi process có thể chạy trên một CPU core khác nhau, hoạt động **thực sự song song**.

### Tại sao threading thất bại với CPU-bound?

Giả sử bạn xử lý 4 ảnh, mỗi ảnh mất 2 giây:

- **Threading**: GIL vẫn chỉ cho 1 thread chiếm CPU. Các thread còn lại xếp hàng chờ. Tổng vẫn ~8 giây.
- **Multiprocessing**: 4 process chạy trên 4 core, cùng lúc. Tổng ~2 giây.

### Nhược điểm

- **Khởi tạo chậm hơn**: mỗi process tạo mới cần ~100ms, trong khi thread chỉ vài ms.
- **Bộ nhớ tốn hơn**: mỗi process có bộ nhớ riêng, không chia sẻ được.
- **Giao tiếp phức tạp hơn**: phải dùng Queue, Pipe, hoặc shared memory để truyền dữ liệu giữa các process.

> **Dùng multiprocessing khi:** xử lý ảnh/video, tính toán số học nặng, machine learning, nén file — bất cứ thứ gì mà CPU là bottleneck.

## Coroutine (asyncio)

### Ý tưởng

Coroutine là cách thứ ba — và thực ra **nhẹ hơn threading nhiều**. Thay vì để OS quyết định khi nào đổi thread, chính code tự nói: *"tao đang chờ, mày chạy đi"*.

Chỉ có **1 thread duy nhất**. Event loop liên tục hỏi: "coroutine nào đang sẵn sàng chạy?" Khi một coroutine gặp `await`, nó nhường quyền, event loop chạy coroutine khác.

```python
import asyncio

async def fetch(url):          # "async" → đây là coroutine
    print(f"Gửi request {url}")
    await asyncio.sleep(2)     # "await" → tạm dừng, nhường chỗ
    print(f"Nhận xong {url}")

async def main():
    await asyncio.gather(       # chạy cả 3 cùng lúc
        fetch("a.com"),
        fetch("b.com"),
        fetch("c.com"),
    )

asyncio.run(main())
```

### Coroutine vs Threading: cùng kết quả, khác cách

Cả hai đều hiệu quả cho I/O-bound. Nhưng:

| | Threading | Coroutine |
|---|---|---|
| Số thread | nhiều | 1 duy nhất |
| Ai điều phối | OS | chính code (`await`) |
| Overhead | trung bình | rất thấp |
| Tạo 10.000 cái | khó | bình thường |

> **Tạo 10.000 threads** → crash hoặc chậm kinh khủng. **Tạo 10.000 coroutines** → bình thường. Đó là lý do các web framework hiện đại (FastAPI, aiohttp) dùng asyncio thay vì threading.

### Giống như đầu bếp phục vụ nhiều bàn

Đặt món bàn 1 lên bếp (`await`), quay sang bàn 2 pha nước, quay sang bàn 3 bưng đồ — không cần thêm người, chỉ cần không đứng chờ.

## Tổng kết: Chọn cái nào?

| | Threading | Coroutine | Multiprocessing |
|---|---|---|---|
| **Tốt cho** | I/O đơn giản | I/O nhiều, đồng thời cao | CPU nặng |
| **Số thread/process** | nhiều thread | 1 thread | nhiều process |
| **Overhead** | trung bình | rất thấp | cao |
| **Chia sẻ bộ nhớ** | có | có | không |
| **Module** | `threading` | `asyncio` | `multiprocessing` |
| **Cách hiện đại** | `concurrent.futures` | `async/await` | `concurrent.futures` |

### Quy tắc chọn nhanh

```
Task của bạn làm gì phần lớn thời gian?

├── Chờ mạng / disk / DB (I/O-bound)
│   ├── Cần xử lý hàng nghìn kết nối? → asyncio (coroutine)
│   └── Vài chục task đơn giản?       → threading
│
└── Tính toán liên tục (CPU-bound)    → multiprocessing
```

Nếu không chắc, `concurrent.futures` là module dễ dùng nhất — cùng API cho cả ThreadPoolExecutor và ProcessPoolExecutor:

```python
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# I/O-bound
with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(fetch_url, urls))

# CPU-bound
with ProcessPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(heavy_compute, data))
```
