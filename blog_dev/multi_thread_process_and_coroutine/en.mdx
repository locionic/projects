# Python Concurrency: Threads, Processes, and Coroutines

<Callout type="info" title="The Core Issue: The GIL">
The **Global Interpreter Lock (GIL)** allows only one thread to run at any given moment to protect Python's memory. Even with 8 cores, Python defaults to using just one! Here are 3 ways to bypass this limit.
</Callout>

<Flashcard term="Global Interpreter Lock (GIL)" definition="A mutex that allows only one thread to execute Python bytecode at a time. Protects CPython's memory management but prevents true parallelism in CPU-bound threaded code." category="Python Internals" />
<Flashcard term="Thread" definition="Lightweight unit of execution within a process. Shares memory space with other threads in the same process. Good for I/O-bound tasks where the GIL is released during waiting." category="Concurrency" />
<Flashcard term="Process" definition="Independent memory space with its own Python interpreter and GIL. True parallelism across CPU cores. Higher overhead than threads but no GIL contention." category="Concurrency" />
<Flashcard term="Coroutine" definition="Function that can pause (await) and resume. Runs on a single thread via an event loop. Enables high-concurrency I/O without thread overhead." category="Concurrency" />

## 1. Threading

<Callout type="idea" title="Best for I/O Tasks">
When a thread waits for I/O (like an API call or DB query), it releases the GIL. That's when other threads can jump in and run!
</Callout>

<ConceptCompare
  columns={["Aspect", "Threading"]}
  rows={[
    { feature: "Parallelism", cells: [{ text: "None (GIL)", type: "negative" }] },
    { feature: "Memory", cells: [{ text: "Shared (low overhead)", type: "positive" }] },
    { feature: "Best for", cells: [{ text: "Network requests, disk I/O, DB queries", type: "positive" }] },
    { feature: "Avoid for", cells: [{ text: "Heavy computation, CPU-bound work", type: "negative" }] }
  ]}
/>

<Stepper>
  <Step title="How it works">Creates multiple threads in a single process. They share memory and the same GIL.</Step>
  <Step title="When to use">Perfect for network requests, disk operations, or waiting on databases.</Step>
  <Step title="When to avoid">Don't use for heavy math or CPU-bound tasks (the GIL will block true parallelism).</Step>
</Stepper>

## 2. Multiprocessing

<Callout type="warning" title="True Parallelism">
Multiprocessing creates **multiple independent processes**, each with its own Python interpreter, memory, and GIL. This achieves true parallelism across CPU cores!
</Callout>

<ConceptCompare
  columns={["Aspect", "Multiprocessing"]}
  rows={[
    { feature: "Parallelism", cells: [{ text: "True (multiple GILs)", type: "positive" }] },
    { feature: "Memory", cells: [{ text: "Isolated (high overhead)", type: "warning" }] },
    { feature: "Best for", cells: [{ text: "Image processing, ML inference, data crunching", type: "positive" }] },
    { feature: "Tradeoff", cells: [{ text: "Slower startup, higher memory", type: "negative" }] }
  ]}
/>

<Stepper>
  <Step title="How it works">Spawns independent processes that run simultaneously on different cores.</Step>
  <Step title="When to use">Heavy computation, image processing, machine learning, and data crunching.</Step>
  <Step title="The tradeoff">Slower startup time and much higher memory usage (no shared memory).</Step>
</Stepper>

<Quiz
  question="What's the main memory disadvantage of multiprocessing compared to threading?"
  options={[
    "Processes share no memory — each has its own copy of data",
    "Processes use a global memory pool that fills up quickly",
    "Processes can't use virtual memory",
    "Processes have a fixed 1GB memory limit"
  ]}
  correctIndex={0}
  explanation="Each process gets its own memory space. If you spawn 8 processes with a 100MB data structure, you use 800MB total. Threads would share that 100MB. This is why multiprocessing has 'much higher memory usage' — no shared memory."
/>

## 3. Coroutines (asyncio)

<Callout type="idea" title="The Modern Way">
Coroutines are incredibly lightweight. Instead of the OS switching threads, your code explicitly says: *"I'm waiting, go run something else"* using `await`.
</Callout>

<ConceptCompare
  columns={["Aspect", "Coroutines (asyncio)"]}
  rows={[
    { feature: "Parallelism", cells: [{ text: "Single-threaded (cooperative)", type: "neutral" }] },
    { feature: "Memory", cells: [{ text: "Very low (KB per task)", type: "positive" }] },
    { feature: "Best for", cells: [{ text: "10k+ concurrent connections, WebSockets", type: "positive" }] },
    { feature: "Scaling", cells: [{ text: "10,000 tasks is easy", type: "positive" }] }
  ]}
/>

<Stepper>
  <Step title="How it works">A single thread uses an event loop to jump between tasks whenever one yields control.</Step>
  <Step title="When to use">High-concurrency I/O like WebSockets or modern web frameworks (FastAPI).</Step>
  <Step title="The advantage">Creating 10,000 coroutines is easy and efficient, unlike 10,000 threads which would crash the system.</Step>
</Stepper>

<Quiz
  question="How does the event loop achieve concurrency without threads?"
  options={[
    "It creates hidden kernel threads for each coroutine",
    "It runs all coroutines in parallel on multiple cores",
    "It switches between tasks at await points on a single thread",
    "It compiles coroutines to C for speed"
  ]}
  correctIndex={2}
  explanation="The event loop runs on one thread. When a coroutine hits `await`, it yields control back to the loop, which then runs the next ready coroutine. This cooperative scheduling means thousands of coroutines can run on one thread without thread overhead."
/>

<Callout type="idea" title="The Verdict">
For most small to medium projects, **threading** is perfectly fine for I/O. But if you're building a massive, data-rich application with thousands of concurrent connections, **asyncio is entirely worth the investment.** For CPU-heavy work, **multiprocessing** is the only way to use all cores.
</Callout>

If you're unsure, `concurrent.futures` is a great unified API for both threading and multiprocessing!
