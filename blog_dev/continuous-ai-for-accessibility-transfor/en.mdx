# Continuous AI for Accessibility: Transforming User Feedback into Inclusive Software Development

For years, GitHub struggled to effectively manage accessibility feedback, as it didn't fit neatly into a single team or process. However, with the help of AI, GitHub transformed its approach to accessibility, ensuring every piece of user feedback becomes a tracked, prioritized issue.

## The Challenge of Accessibility Feedback

Accessibility issues don't belong to any single team; they cut across the entire ecosystem. For example, a screen reader user might report a broken workflow that touches navigation, authentication, and settings. A keyboard-only user might hit a trap in a shared component used across dozens of pages. A low vision user might flag a color contrast issue that affects every surface using a shared design element.

## Centralizing Feedback and Creating a Foundation

To address this challenge, GitHub centralized scattered reports, created templates, and triaged years of backlog. This foundation enabled the development of an internal workflow powered by GitHub Actions, GitHub Copilot, and GitHub Models.

## Continuous AI for Accessibility

Continuous AI for accessibility weaves inclusion into the fabric of software development. It's not a single product or a one-time audit—it's a living methodology that combines automation, artificial intelligence, and human expertise.

### Designing for People First

Before building the system, GitHub stepped back to understand who it needed to serve:

* Issue submitters: Community managers, support agents, and sales reps submit issues on behalf of users and customers.
* Accessibility and service teams: Engineers and designers responsible for fixes need structured, actionable data.
* Program and product managers: Leadership needs visibility into pain points by category, trends, and progress over time.

### How Feedback Flows

The workflow reacts to key events: issue creation launches GitHub Copilot analysis, status changes initiate hand-offs between teams, and resolution triggers submitter follow-up with the user.

### Actioning Intake

Feedback can come from anywhere, but most users choose the GitHub accessibility discussion board. Posts are public, allowing other users to confirm the problem, add context, or suggest workarounds.

### GitHub Copilot Analysis

A GitHub Action workflow programmatically calls the GitHub Models API to analyze the report. GitHub Copilot populates approximately 80% of the issue's metadata automatically, including issue type, user segment, original source, affected components, and enough context to understand the user's experience.

### Submitter Review

The submitter attempts to replicate the problem the user reported, using a checklist provided by GitHub Copilot to guide them through expert-level testing procedures.

### Accessibility Team Review

Once the submitter marks the issue as reviewed, a GitHub Action updates its status on the workflow project board and adds it to a separate accessibility first responder board. The accessibility team validates GitHub Copilot's analysis and corrects anything the AI got wrong.

### Linking to Audits

The team connects user and customer feedback to the formal accessibility audit system, prioritizing based on real-world impact.

### Closing the Loop

The submitter communicates the plan to the user, letting them know what's being done and what to expect. This closes the loop, ensuring that every piece of accessibility feedback is tracked, prioritized, and acted on.
