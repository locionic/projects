I still remember the shock when Ichecked my OpenAI bill. Two hundred dollars. For a solo project I was barely making money from.

The API costs had crept up on me. What started as a few cents for testing had become hundreds of dollars in production. I'd been building a documentation tool that auto-explained code, and every time a user ran a query, my costs climbed.

That's when I decided to go local.

<Callout type="info">
If you're spending more than $50/month on AI APIs for development work, local models can cut that to zero. The trade-off is setup time, not money.
</Callout>

## What Even Is Ollama?

Ollama is a command-line tool that runs AI models on your own hardware. No servers. No cloud. No monthly invoices.

I first heard about it from a colleague who was running Llama 2 on his MacBook. I didn't believe him. "You can't run a useful language model on a laptop," I said. I was wrong.

The setup took me about twenty minutes. I downloaded the app, ran one command, and suddenly Mistral 7B was living on my machine, ready to answer questions without calling home.

Does that sound too good to be true? I thought so too, until I saw it working.

## Getting Started Takes Less Than an Hour

Installation varies by operating system. On macOS, you download the app from ollama.ai and you're done. On Linux, it's a single curl command:

```bash
curl -fsSL https://ollama.ai/install.sh | sh
```

Windows users need WSL2, which adds some friction. I won't lie about that. But once it's running, the experience is the same across platforms.

<Stepper>
  <Step title="Download Ollama from ollama.ai">
    The installer is about 200MB. It takes longer to download than to install.
  </Step>
  <Step title="Pull your first model">
    Run `ollama pull mistral` and wait ten to fifteen minutes depending on your internet speed. The model downloads once and stays on your machine.
  </Step>
  <Step title="Start chatting">
    Run `ollama run mistral` and you're in a REPL. Type a question, press enter, get an answer. It's that simple.
  </Step>
</Stepper>

My first conversation with Mistral felt like magic. I asked it to explain a complex regex pattern I'd been wrestling with, and it broke it down in seconds. No latency from a distant server. No token limits. Just answers.

## Which Model Should You Start With?

This confused me at first. There are dozens of models available. Llama 3, Mistral, Phi, Gemma, Code Llama. Which one do you actually want?

Here's what I've learned through trial and error.

**For general coding questions, Mistral 7B is my daily driver.** It's fast, accurate, and doesn't eat all your RAM. I run it alongside VS Code and forget it's there.

**For explaining unfamiliar codebases, Llama 3 8B is surprisingly good.** It's better at following complex instructions and maintaining context across longer conversations.

**For generating code, Code Llama 7B earns its name.** It's not perfect, but it understands languages that general models stumble on. I've used it to scaffold entire functions in Rust and Haskell where ChatGPT often hallucinated syntax.

What about the bigger models? Llama 3 70B lives in a different category. You'd need serious hardware to run it comfortably. I've got 32GB of RAM and I still feel the strain.

<Callout type="idea">
Start with Mistral 7B. It's the best balance of capability and resource usage I've found. You can always experiment with larger models once you've got the basics working.
</Callout>

## The Hardware Reality

I won't pretend any laptop can run these models. My setup is a MacBook Pro with an M2 Max and 32GB of unified memory. I know not everyone has this, and I'm not here to brag.

Here's what I've observed: most modern machines can run the 7B parameter models. The 13B models are hit or miss. Anything above that really needs better hardware or serious patience.

The bottleneck is almost always RAM. Ollama shows you the memory usage when you start a model. Watch that number. If it gets close to your available memory, you'll feel your machine slow to a crawl.

I've talked to developers running Ollama on Linux boxes with RTX 3060s. The GPU acceleration makes a huge difference. If you're serious about this, an NVIDA GPU in your build is worth considering.

## My Actual Workflow After Six Months

Using Ollama in practice is different from the marketing. I've developed habits that make it genuinely useful rather than just a novelty.

I wrapped Ollama with a simple HTTP server so my existing tools could talk to it. Most AI-powered plugins support OpenAI-compatible endpoints. Ollama speaks that protocol out of the box. I pointed my tools at localhost and they just worked.

<tabs>
  <tab title="Documentation Helper">

I built a small script that reads a file and asks Mistral to explain it. I pipe the file path, get back a plain-English summary. It's saved me hours of reverse-engineering old code.

  </tab>
  <tab title="Code Review Buddy">

Before committing, I ask Code Llama to check for obvious bugs. It's not a substitute for real review, but it catches things I miss at 11pm. Last week it flagged an off-by-one error I would have spent an hour debugging.

  </tab>
  <tab title="Learning Partner">

I read unfamiliar codebases by asking questions. "Why does this function take a closure?" Mistral explains in context, not in generic terms. The answers feel hand-tailored to what I'm looking at.

  </tab>
</tabs>

## The Privacy Angle Nobody Talks About Enough

Here's something that convinced me more than any cost argument: my code stays on my machine.

When I send a prompt to an API, I'm handing a third party my unfinished, possibly buggy, potentially confidential code. I've signed NDAs that probably cover this. I still felt uneasy.

With Ollama, the conversation never leaves my laptop. That matters when you're working on proprietary systems or anything even adjacent to sensitive data.

I once helped a healthcare startup evaluate local AI options. They couldn't send patient records through external APIs due to HIPAA concerns. Ollama running on their own servers solved a problem that API providers simply couldn't touch.

Is privacy a concern for your project? If it is, local models might be the answer you've been looking for.

## What's Broken and What Needs Work

I've been critical so far because I want you to have realistic expectations. Ollama isn't without issues.

**Context windows are smaller than commercial APIs.** Mistral 7B handles about 8,000 tokens comfortably. That's fine for most tasks. It falls short if you're trying to analyze a massive codebase in one conversation.

**Model quality varies.** I've had Code Llama produce nonsense that Mistral would never suggest. You don't get the consistency of GPT-4. There's calibration involved in knowing which model to trust for which task.

**No built-in chat interface until recently.** The new Ollama app on macOS helps, but most interactions still happen in the terminal. If you're not comfortable there, this might not be for you.

<Callout type="warning">
Ollama models are less capable than GPT-4 on hard reasoning tasks. They're not replacements for commercial APIs in every scenario. Know your use case before committing.
</Callout>

## Should You Switch?

I've cut my API bills to nearly zero. That alone was worth the migration effort.

But the bigger win for me was changing how I think about AI tools. Local models have different strengths than cloud ones. They're private, fast, and free at scale. They also require more tweaking and tolerance for imperfection.

Are you building something where cost and privacy matter more than cutting-edge capability? Then yes, switch.

Are you solving hard problems that require the best reasoning available? Maybe keep cloud APIs for those, and use Ollama for everything else.

I use both now. The cloud for the hard stuff, local for everything I can get away with. It's the best of both worlds, and it turns out I can afford both.
