# Introduction to Test-Driven Development (TDD)

What if you wrote your tests **before** writing the actual code? That's the core idea behind TDD — a practice that leads to cleaner, more reliable, and better-designed code.

<Callout type="idea" title="A Mindset Shift">
TDD might sound counterintuitive at first, but once you embrace it, it completely transforms the way you approach software architecture.
</Callout>

TDD follows a simple, repeating three-step rhythm:

<Stepper>
  <Step title="🔴 Red">Write a test that fails because the implementation doesn't exist yet.</Step>
  <Step title="🟢 Green">Write the minimum amount of code required to make the test pass.</Step>
  <Step title="🔵 Refactor">Clean up the code, remove duplication, and optimize while keeping tests passing.</Step>
</Stepper>

## A Simple Example: Multiplication

Let's build a `multiply` function using TDD!

<Stepper>
  <Step title="Step 1 — Red (Failing Test)">
    Write the test first. It fails expectedly.
    ```js
    test('multiplies two numbers', () => {
      expect(multiply(3, 4)).toBe(12);
    });
    ```
  </Step>
  <Step title="Step 2 — Green (Make it Pass)">
    Write just enough code to satisfy the test.
    ```js
    function multiply(a, b) {
      return a * b;
    }
    ```
  </Step>
  <Step title="Step 3 — Refactor">
    The code is already clean! Nothing to change here, but as complexity grows, refactoring becomes essential.
  </Step>
</Stepper>

## Why Use TDD?

<Callout type="info" title="The Benefits">
- **Fewer Bugs:** Issues are caught instantly.
- **Better Design:** Forces you to define the interface before implementation.
- **Fearless Refactoring:** A solid test suite gives you confidence to change code safely.
- **Living Documentation:** Tests describe exactly how the code is supposed to behave.
</Callout>

<Callout type="warning" title="Common Mistakes to Avoid">
- Writing too much code before running tests.
- Skipping the **Refactor** step.
- Testing internal implementation details instead of external behavior.
- Writing tests *after* the code (that's just testing, not TDD!).
</Callout>

## Conclusion

TDD is not just a testing technique — it's a **design practice**. Start small: pick a simple utility function, write the test first, watch it fail, and make it pass. You'll quickly see why developers love it!
