## Introduction

What if you wrote your tests **before** writing the actual code? That's the core idea behind **Test-Driven Development (TDD)** — a practice that leads to cleaner, more reliable, and better-designed code.

TDD might sound counterintuitive at first, but once you try it, it changes the way you think about writing software.

---

## The TDD Cycle: Red, Green, Refactor

TDD follows a simple three-step cycle:

1. **Red** — write a test that fails (because the code doesn't exist yet)
2. **Green** — write the minimum code to make the test pass
3. **Refactor** — clean up the code while keeping tests passing

Repeat this cycle for every new feature or behavior.

---

## A Simple TDD Example

Let's build a `multiply` function using TDD.

### Step 1 — Red (write a failing test)
```js
// multiply.test.js
test('multiplies two numbers', () => {
  expect(multiply(3, 4)).toBe(12);
});
```

Run the test — it fails because `multiply` doesn't exist yet. That's expected.

### Step 2 — Green (make it pass)
```js
// multiply.js
function multiply(a, b) {
  return a * b;
}

module.exports = { multiply };
```

Run the test again — it passes. ✅

### Step 3 — Refactor

The code is already clean, so nothing to change here. But as complexity grows, this step becomes important for removing duplication and improving readability.

## A Real-World TDD Example

Let's build a shopping cart with TDD.

### Test 1 — empty cart has zero total
```js
test('empty cart has total of 0', () => {
  const cart = new Cart();
  expect(cart.total()).toBe(0);
});
```
```js
class Cart {
  total() {
    return 0;
  }
}
```

### Test 2 — adding item increases total
```js
test('adding item increases total', () => {
  const cart = new Cart();
  cart.add({ name: 'Book', price: 15 });
  expect(cart.total()).toBe(15);
});
```
```js
class Cart {
  constructor() {
    this.items = [];
  }

  add(item) {
    this.items.push(item);
  }

  total() {
    return this.items.reduce((sum, item) => sum + item.price, 0);
  }
}
```

### Test 3 — multiple items
```js
test('calculates total for multiple items', () => {
  const cart = new Cart();
  cart.add({ name: 'Book', price: 15 });
  cart.add({ name: 'Pen', price: 3 });
  expect(cart.total()).toBe(18);
});
```

No code changes needed — it already passes! ✅

Notice how TDD guided us to a clean, well-structured `Cart` class naturally.

## Benefits of TDD

**1. Fewer bugs** — tests catch issues immediately as you write code.

**2. Better design** — writing tests first forces you to think about the interface before the implementation.

**3. Confidence to refactor** — with full test coverage, you can change code without fear.

**4. Living documentation** — tests describe exactly what your code does.

**5. Faster debugging** — when something breaks, failing tests point directly to the problem.

---

## Common TDD Mistakes

**Writing too much code at once** — write only enough code to make the current test pass.

**Skipping the refactor step** — clean code matters, don't rush past it.

**Testing implementation details** — test behavior, not how the code works internally.

**Writing tests after the code** — that's not TDD, it's just testing.

---

## When TDD Works Best

TDD is especially valuable for:

- **Business logic** — calculations, validation, data transformations
- **Utility functions** — pure functions with clear inputs and outputs
- **Bug fixes** — write a test that reproduces the bug, then fix it

TDD is less natural for:
- UI components (though possible)
- Exploratory code where you're not sure what you're building yet

---

## Getting Started

If TDD is new to you, start small:

1. Pick one simple utility function in your project
2. Delete the implementation (keep a backup)
3. Write tests for the expected behavior
4. Rewrite the implementation to make tests pass
5. Refactor

After a few rounds of this, the Red-Green-Refactor rhythm becomes natural.

---

## Conclusion

TDD is not just a testing technique — it's a design practice. By writing tests first you:

1. Think clearly about what your code should do
2. Write only the code you need
3. Build a safety net that catches regressions
4. End up with cleaner, more maintainable code

Start with one function today. Write the test first, watch it fail, then make it pass. You'll quickly see why so many developers swear by TDD.
