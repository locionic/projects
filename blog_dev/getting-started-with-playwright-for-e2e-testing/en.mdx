## Introduction

End-to-end (E2E) testing verifies that your entire application works correctly from the user's perspective — clicking buttons, filling forms, navigating pages. While unit tests check individual functions, E2E tests check the whole flow.

**Playwright** is Microsoft's open-source E2E testing framework that supports Chromium, Firefox, and WebKit. It's fast, reliable, and has become the go-to choice for modern web testing.

In this guide, you'll learn how to set up Playwright and write your first E2E tests from scratch.

---

## Why Playwright?

There are several E2E testing tools available — Selenium, Cypress, Puppeteer. Here's why Playwright stands out:

- **Cross-browser** — test on Chrome, Firefox, and Safari with one codebase
- **Auto-wait** — automatically waits for elements to be ready before interacting
- **Fast** — runs tests in parallel by default
- **Modern** — supports TypeScript, async/await, and modern web features
- **Free** — fully open source with no paid tier needed

## 1. Installation

Start by installing Playwright in your project:
```bash
npm init playwright@latest
```

This will prompt you to:
- Choose between TypeScript or JavaScript (recommend TypeScript)
- Choose a folder for tests (default: `tests`)
- Add a GitHub Actions workflow (optional but recommended)
- Install browsers

Playwright will download Chromium, Firefox, and WebKit automatically.

---

## 2. Project Structure

After installation your project will look like this:
```
playwright.config.ts   ← configuration file
tests/
  example.spec.ts      ← sample test file
test-results/          ← test output
```

## 3. Your First Test

Open `tests/example.spec.ts` and replace it with this:
```ts
import { test, expect } from '@playwright/test';

test('homepage loads correctly', async ({ page }) => {
  // Navigate to your site
  await page.goto('https://www.locionic.com');

  // Check the title
  await expect(page).toHaveTitle(/Portfolio/);

  // Check a heading is visible
  await expect(page.getByRole('heading', { name: /Portfolio & Blog/i })).toBeVisible();
});
```

Run the test:
```bash
npx playwright test
```

You'll see output like:
```
✓ homepage loads correctly (1.2s)
1 passed
```

## 4. Common Actions

Here are the most common things you'll do in Playwright tests:

### Clicking elements
```ts
await page.getByRole('button', { name: 'Submit' }).click();
await page.getByText('Read more').click();
await page.locator('#login-btn').click();
```

### Filling forms
```ts
await page.getByLabel('Email').fill('test@example.com');
await page.getByLabel('Password').fill('password123');
```

### Navigating
```ts
await page.goto('https://example.com/about');
await page.goBack();
await page.reload();
```

### Assertions
```ts
await expect(page).toHaveURL(/dashboard/);
await expect(page.getByText('Welcome')).toBeVisible();
await expect(page.getByRole('button')).toBeDisabled();
```

## 5. Testing a Login Flow

Here's a real-world example — testing a login flow:
```ts
test('user can log in successfully', async ({ page }) => {
  // Go to login page
  await page.goto('/login');

  // Fill in credentials
  await page.getByLabel('Email').fill('user@example.com');
  await page.getByLabel('Password').fill('correctpassword');

  // Click login button
  await page.getByRole('button', { name: 'Log in' }).click();

  // Verify redirect to dashboard
  await expect(page).toHaveURL(/dashboard/);
  await expect(page.getByText('Welcome back')).toBeVisible();
});
```

## 6. Running Tests in Different Modes
```bash
# Run all tests
npx playwright test

# Run a specific file
npx playwright test tests/login.spec.ts

# Run with browser visible (headed mode)
npx playwright test --headed

# Run in a specific browser
npx playwright test --project=firefox

# Show test report
npx playwright show-report
```

## 7. Playwright Config

Your `playwright.config.ts` controls how tests run. Here's a practical setup:
```ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  retries: 1,
  reporter: 'html',
  use: {
    baseURL: 'https://www.locionic.com',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },
  projects: [
    { name: 'chromium', use: { browserName: 'chromium' } },
    { name: 'firefox', use: { browserName: 'firefox' } },
  ],
});
```

Key settings:
- `retries: 1` — retry failed tests once to reduce flakiness
- `screenshot: 'only-on-failure'` — capture screenshots when tests fail
- `video: 'retain-on-failure'` — record video of failed tests for debugging

## 8. Best Practices

**Use role-based selectors** — they're more resilient than CSS selectors:
```ts
// ✅ Good
page.getByRole('button', { name: 'Submit' })

// ❌ Fragile
page.locator('.btn-primary-submit-v2')
```

**Keep tests independent** — each test should work on its own without relying on previous tests.

**Use `beforeEach` for repeated setup:**
```ts
test.beforeEach(async ({ page }) => {
  await page.goto('/');
});
```

**Group related tests** with `test.describe`:
```ts
test.describe('Blog page', () => {
  test('shows latest posts', async ({ page }) => { ... });
  test('can search posts', async ({ page }) => { ... });
});
```
