# Browser Run: Rebuilt on Cloudflare Containers for Enhanced Performance and Scalability

Browser Run, a platform for programmatically controlling and interacting with headless browser instances, has been rebuilt on top of Cloudflare's Containers. This upgrade brings significant improvements in performance, reliability, and scalability, enabling developers to leverage the power of browsers at scale.

## What is Browser Run?

Browser Run is a Cloudflare platform that allows developers to programmatically control and interact with headless browser instances running on Cloudflare's global network. This is useful for various use cases, such as:

* End-to-end testing of web applications
* Securely investigating suspicious URLs
* Leveraging browser capabilities for tasks like PDF rendering, screenshot capture, and content extraction
* Enabling AI agents to interact with the web

## Outgrowing Shared Infrastructure

Prior to the upgrade, Browser Run shared infrastructure with Browser Isolation (BISO). However, this shared infrastructure presented several challenges, including:

* Slower startup and development times due to larger container images
* Suboptimal global distribution of browsers, compromising resiliency and latency
* Conflicting usage patterns between BISO and Browser Run, creating scaling bottlenecks and availability delays

## The Migration to Containers

To address these challenges, the Browser Run team migrated to Cloudflare Containers, which provided a more scalable and flexible infrastructure. The migration involved:

* Gradually introducing Container-powered browsers alongside BISO browsers
* Comparing performance and isolating implementation bugs during the transition
* Ramping up adoption by using Container browsers for Quick Actions endpoints, Workers browser binding, and eventually all remaining contract customers

## Overcoming Performance and Scale Bottlenecks

During the migration, the team encountered several challenges, including:

* Adapting to a novel, unstable early-stage Containers platform interface
* Overcoming friction in the new technical environment
* Managing the increased latency introduced by establishing WebSockets between DOs and containers

To address these challenges, the team implemented several solutions, including:

* Creating regional pools of pre-warmed DO-backed browser containers to constrain latency
* Using Workers KV to manage container state, but eventually migrating to D1 + Queues to achieve better scalability and performance

## Migrating from KV to D1 + Queues

The team migrated from using KV to store container state to using D1 instances, which provided a more transactional and scalable solution. This involved:

* Using SQLite transactions to ensure atomic assignment and prevent race conditions
* Implementing batch writes using Queues to achieve higher throughput and reduce lag times

## Additional Perks for Quick Actions

The upgrade also enabled several optimizations for quick actions, including:

* Sending all parameters in a single HTTP request directly to the container, reducing back-and-forth between the worker and browser
* Executing the entire flow internally without any WebSocket communication

## Results and Benefits

The upgrade resulted in significant improvements in performance, reliability, and scalability, including:

* A sharp decrease in average quick-action response time
* Increased limits for concurrent browser instances and usage
* Faster upgrades and better browser flexibility
* Improved support for agentic development and AI agents

## Getting Started with Browser Run

Browser Run is available on all Workers plans. Developers can get started by:

* Checking out the quick start guide
* Exploring the Quick Actions
* Trying the /crawl endpoint to deeply extract data from any webpage
* Building AI agents with the Agents SDK, which includes built-in Browser Run support.
