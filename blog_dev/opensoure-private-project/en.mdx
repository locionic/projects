# Fork It, Keep It Private, Stay in Sync

You found an open-source project that's 80% of what you need. You want to add custom features, keep them private, but still pull in upstream bug fixes. Here is the cleanest way to do it.

<Callout type="info" title="The Goal">
Maintain a private fork with your custom commits cleanly applied on top of the latest upstream changes.
</Callout>

---

## The Workflow

<Stepper>
  <Step title="Clone & Setup Remotes">
    Clone the original project and set your private repo as `origin`, while keeping the original as `upstream`.
    ```bash
    git clone https://github.com/original-author/some-project
    cd some-project
    git remote set-url origin https://github.com/you/your-private-repo
    git remote add upstream https://github.com/original-author/some-project
    git push origin main
    ```
  </Step>
  <Step title="Fetch Upstream Changes">
    When the original project updates, grab the latest changes.
    ```bash
    git fetch upstream
    ```
  </Step>
  <Step title="Rebase Your Work">
    Cleanly stack your custom commits on top of the upstream's latest code.
    ```bash
    git checkout main
    git rebase upstream/main
    ```
  </Step>
  <Step title="Push to Private Repo">
    Force push your rewritten history safely.
    ```bash
    git push origin main --force-with-lease
    ```
  </Step>
</Stepper>

---

## Why Rebase Instead of Merge?

<Callout type="idea" title="Linear History">
Using `git merge` creates noisy merge commits every time you sync. `git rebase` keeps your custom features stacked cleanly on top of the upstream commits, creating a readable, linear history.
</Callout>

## Why `--force-with-lease`?

<Callout type="warning" title="Safety First">
Rebasing rewrites commit hashes. A plain `--force` pushes unconditionally and can overwrite collaborators' work. `--force-with-lease` guarantees you only push if no one else has updated the remote branch since your last fetch.
</Callout>

## License Check

<Callout type="danger" title="Know Your Licenses">
Before keeping a fork private, check the upstream license!
- **MIT / Apache 2.0:** You can freely use, modify, and keep your fork private.
- **GPL / AGPL:** Requires you to publish your modifications if you distribute or host the software.
</Callout>
