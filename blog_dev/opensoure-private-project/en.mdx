# How to Fork an Open Source Project into a Private Repo and Stay in Sync

You found an open source project that's 80% of what you need. You want to build on top of it — add your own features, tweak internals, keep your changes private — while still pulling in upstream bug fixes and improvements. Here's the clean way to do it.

---

## Initial Setup

Clone the upstream repo and push it to your own private remote:

```bash
git clone https://github.com/original-author/some-project
cd some-project

# Replace origin with your private repo
git remote set-url origin https://github.com/you/your-private-repo

# Add upstream so you can pull future changes
git remote add upstream https://github.com/original-author/some-project

git push origin main
```

Verify your remotes:

```bash
git remote -v
# origin    https://github.com/you/your-private-repo (fetch)
# origin    https://github.com/you/your-private-repo (push)
# upstream  https://github.com/original-author/some-project (fetch)
# upstream  https://github.com/original-author/some-project (push)
```

Now you have two remotes: `origin` (your private repo) and `upstream` (the source of truth).

---

## The Sync Workflow

Every time upstream ships changes you want:

```bash
git fetch upstream
git checkout main
git rebase upstream/main
git push origin main --force-with-lease
```

That's it. Four commands.

---

## Why Rebase Instead of Merge?

You could use `git merge upstream/main` and it would work. But every sync creates a merge commit:

```
* Merge remote-tracking branch 'upstream/main'   ← noise
* your feature
* Merge remote-tracking branch 'upstream/main'   ← noise
* another feature
```

With rebase, your commits always sit cleanly on top of upstream:

```
* your feature B
* your feature A
* [upstream] fix auth bug
* [upstream] bump dependency
```

Linear, readable, no noise.

---

## Why `--force-with-lease` Instead of `--force`?

Rebase rewrites commit hashes. Even if the content is identical, git sees your local `main` and `origin/main` as diverged — a plain `git push` will be rejected.

`--force` would push anyway, unconditionally. `--force-with-lease` pushes only if the remote hasn't changed since your last fetch. On a solo private repo the difference rarely matters, but it's the correct habit: it prevents accidentally overwriting commits you forgot about or that a collaborator pushed.

---

## Handling Conflicts

When upstream changes overlap with yours, rebase will pause:

```bash
git rebase upstream/main
# CONFLICT (content): Merge conflict in src/config.ts
```

Fix the conflict in your editor, then:

```bash
git add src/config.ts
git rebase --continue
```

Repeat for each conflicting commit. If things go sideways:

```bash
git rebase --abort  # returns you to pre-rebase state
```

---

## License Check

Before keeping a fork private, verify the upstream license. MIT and Apache 2.0 let you use, modify, and distribute (or not distribute) freely — private forks are fine. GPL and AGPL are different: they require you to publish your modifications if you distribute the software, and AGPL extends this to network use.

If the project is MIT-licensed, you have no obligations beyond attribution in the license file.

---

## Summary

| Step | Command |
|------|---------|
| One-time: add upstream | `git remote add upstream <url>` |
| Fetch changes | `git fetch upstream` |
| Rebase your work on top | `git rebase upstream/main` |
| Push | `git push origin main --force-with-lease` |

Keep your features on `main` (or a dedicated branch), sync regularly to avoid large conflict batches, and you have a maintainable private fork that stays current with upstream indefinitely.
