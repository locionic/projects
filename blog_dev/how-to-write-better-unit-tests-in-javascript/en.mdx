## Introduction

Writing code is only half the job. Making sure that code keeps working as your project grows is the other half. Unit tests are an incredibly powerful tool, yet many developers either skip them or write tests that don't actually help.

Let's look at practical techniques to write better unit tests using **Jest**.

<Callout type="info" title="What is a Unit Test?">
  A unit test verifies a single, isolated piece of code. It should **never** depend on databases, APIs, or external systems. It must be fast and predictable!
</Callout>

---

## Best Practices

<Stepper>
  <Step title="Follow the AAA Pattern">
    Structure every test with **Arrange** (set up data), **Act** (call the function), and **Assert** (verify the result). This makes tests instantly readable.
  </Step>
  <Step title="Test One Thing at a Time">
    Each test should verify exactly one behavior. Don't write a mega-test that checks 15 different scenarios. Break them out!
  </Step>
  <Step title="Write Descriptive Names">
    Avoid vague names like `test1`. Instead, explain the expectation: `test('should return false when age is negative')`.
  </Step>
</Stepper>

---

## Mocking & Edge Cases

When your function interacts with external APIs or timers, use **Mocks** (`jest.fn()`) to simulate the behavior.

```js
// Mocking fetch
global.fetch = jest.fn(() =>
  Promise.resolve({ json: () => Promise.resolve({ name: 'Loc' }) })
);
```

<Callout type="warning" title="Test Edge Cases!">
  Bugs love hiding in edge cases! Always test empty arrays, null values, zero, and extremely large numbers. Don't just test the "happy path".
</Callout>

<Callout type="idea" title="Aim for Meaningful Coverage">
  Chasing 100% code coverage leads to poorly written, meaningless tests. Focus on covering core business logic, error handling, and complex functions. A strong 70% coverage is far better than a fragile 100%.
</Callout>

Good tests give you the confidence to refactor and ship code without fear. Start applying these techniques today!
