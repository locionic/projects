# Unit Tests in JavaScript: What Actually Changed My Testing Habits

I wrote bad unit tests for years. They passed when the code was wrong, broke when I refactored working code, and took longer to maintain than the code they were testing. The problem wasn't Jest or my testing framework — it was how I was thinking about tests.

Here are the specific changes that made my tests actually useful.

## The AAA Pattern Isn't Optional

I used to write tests like this:

```js
test('works', () => {
  const result = add(1, 2);
  expect(result).toBe(3);
  const result2 = add(-1, 1);
  expect(result2).toBe(0);
});
```

One test, two scenarios, and a name that tells you nothing. If it fails, you have to read the whole thing to figure out which case broke.

Following Arrange-Act-Assert forces clarity:

```js
test('adds two positive numbers', () => {
  // Arrange
  const a = 1, b = 2;
  // Act
  const result = add(a, b);
  // Assert
  expect(result).toBe(3);
});

test('handles negative numbers', () => {
  const result = add(-1, 1);
  expect(result).toBe(0);
});
```

Two separate tests, each verifying exactly one behavior. When one fails, you know instantly which case broke. The AAA structure makes you write smaller, more focused tests.

## Mock Only What You Own

Early on, I mocked everything — network calls, database queries, even `Date.now()`. The result was tests that passed in isolation but failed in integration because my mock didn't match reality.

The rule I follow now: **mock your boundary, not your dependencies.** If you're testing a function that calls `fetch()`, mock `fetch`. Don't mock your database layer, your Redis client, or your message queue — test those with integration tests instead.

```js
// Good: mock the boundary
global.fetch = jest.fn()
  .mockResolvedValue({ json: () => ({ name: 'Loc' }) });

// Bad: mocking internals that change during refactoring
jest.mock('../../db', () => ({
  query: jest.fn().mockResolvedValue([{ id: 1 }])
}));
```

When you mock internals, your tests break every time you refactor. When you mock the boundary, they only break when the external API changes.

## Edge Cases Matter More Than Coverage Percentage

The bugs that reach production almost never live in the happy path. They live in null values, empty arrays, and edge conditions the developer didn't think about.

```js
test('returns empty array for null input', () => {
  expect(processItems(null)).toEqual([]);
});

test('handles empty array', () => {
  expect(processItems([])).toEqual([]);
});

test('handles single item', () => {
  expect(processItems(['a'])).toEqual(['A']);
});
```

I aim for 70-80% coverage on core business logic and don't chase the last 20%. The tests that add value are the ones for error handling, edge cases, and complex logic — not the ones that verify a getter returns the value you just set.

<Callout type="info" title="The Test That Caught a Real Bug">
I once had a test for a "discount percentage" function that tested 50%, 25%, and 10%. The function worked fine for all of them. But a test for `0` — no discount — revealed that the code divided by the discount rate without checking for zero. One extra test, one division-by-zero that would have hit production.
</Callout>

## Descriptive Test Names Are a Form of Documentation

Compare these:

```js
test('test1', ...)
test('should work', ...)
```

vs.

```js
test('returns user profile when valid ID provided', ...)
test('throws NotFoundError when user does not exist', ...)
test('returns cached result without calling API for duplicate requests', ...)
```

The second set tells you what the system does. When onboarding a new developer to a codebase, the test file is often the clearest documentation of expected behavior. Name tests like you're describing a specification, not checking a box.

## What I Don't Bother Testing

- **Simple getters and setters** — If there's no logic, there's nothing to test
- **Framework behavior** — Jest already tested Jest
- **Cosmetic UI states** — Testing that a button turns red when loading tells you nothing about whether the loading state works
