## Introduction

Writing code is only half the job. Making sure that code works correctly — and keeps working as your project grows — is the other half. Unit tests are one of the most powerful tools a developer has, yet many developers either skip them or write tests that don't actually help.

In this post, I'll share practical techniques to write better unit tests in JavaScript using **Jest**, the most popular testing framework in the JavaScript ecosystem.

---

## What Is a Unit Test?

A unit test verifies that a single, isolated piece of code (usually a function) works as expected. The key word is **isolated** — a unit test should not depend on databases, APIs, or other external systems.
```js
// A simple function
function add(a, b) {
  return a + b;
}

// A unit test for it
test('adds two numbers correctly', () => {
  expect(add(2, 3)).toBe(5);
});
```

Simple, fast, and reliable. That's what a good unit test looks like.

---

## 1. Follow the AAA Pattern

The **Arrange, Act, Assert** pattern keeps your tests readable and consistent.

- **Arrange** — set up the data and conditions
- **Act** — call the function being tested
- **Assert** — verify the result
```js
test('should return full name', () => {
  // Arrange
  const firstName = 'Loc';
  const lastName = 'Truong';

  // Act
  const result = getFullName(firstName, lastName);

  // Assert
  expect(result).toBe('Loc Truong');
});
```

This structure makes it immediately clear what each test is doing.

---

## 2. Write Descriptive Test Names

A test name should explain **what** is being tested and **what the expected outcome is**. Avoid vague names like `test1` or `works correctly`.
```js
// ❌ Bad
test('user function', () => { ... });

// ✅ Good
test('should return null when user is not found', () => { ... });
```

When a test fails, a descriptive name tells you exactly what broke without having to read the code.

---

## 3. Test One Thing at a Time

Each test should verify exactly one behavior. If a test is checking multiple things, split it into separate tests.
```js
// ❌ Bad — testing too many things at once
test('user validation', () => {
  expect(validateUser({ name: '' })).toBe(false);
  expect(validateUser({ name: 'Loc', age: -1 })).toBe(false);
  expect(validateUser({ name: 'Loc', age: 25 })).toBe(true);
});

// ✅ Good — one behavior per test
test('should return false when name is empty', () => {
  expect(validateUser({ name: '' })).toBe(false);
});

test('should return false when age is negative', () => {
  expect(validateUser({ name: 'Loc', age: -1 })).toBe(false);
});

test('should return true for valid user', () => {
  expect(validateUser({ name: 'Loc', age: 25 })).toBe(true);
});
```

---

## 4. Use Mocks for External Dependencies

When your function depends on APIs, databases, or timers, use **mocks** to isolate it. This keeps tests fast and predictable.
```js
// Mock a fetch call
global.fetch = jest.fn(() =>
  Promise.resolve({
    json: () => Promise.resolve({ name: 'Loc Truong' }),
  })
);

test('should fetch user data', async () => {
  const user = await fetchUser(1);
  expect(user.name).toBe('Loc Truong');
  expect(fetch).toHaveBeenCalledWith('/api/users/1');
});
```

---

## 5. Test Edge Cases

Most bugs hide in edge cases — empty arrays, null values, zero, very large numbers. Make sure you test these scenarios.
```js
test('should return 0 for empty array', () => {
  expect(sum([])).toBe(0);
});

test('should handle null input gracefully', () => {
  expect(formatName(null)).toBe('');
});

test('should handle negative numbers', () => {
  expect(absolute(-5)).toBe(5);
});
```

---

## 6. Keep Tests Independent

Tests should never depend on each other. Each test should set up its own data and clean up after itself. Use `beforeEach` and `afterEach` for setup and teardown.
```js
let db;

beforeEach(() => {
  db = createTestDatabase();
});

afterEach(() => {
  db.clear();
});

test('should save user to database', () => {
  db.save({ name: 'Loc' });
  expect(db.find('Loc')).toBeTruthy();
});
```

---

## 7. Aim for Meaningful Coverage, Not 100%

Code coverage is a useful metric, but chasing 100% coverage can lead to meaningless tests. Focus on covering:

- Core business logic
- Error handling paths
- Edge cases
- Functions that change frequently

A well-tested 70% is more valuable than poorly written 100% coverage.

---

## Conclusion

Writing better unit tests is a skill that improves with practice. Start with these principles:

1. Follow the **AAA pattern**
2. Write **descriptive test names**
3. Test **one thing at a time**
4. **Mock** external dependencies
5. Always test **edge cases**
6. Keep tests **independent**
7. Focus on **meaningful coverage**

Good tests give you confidence to refactor, add features, and ship code without fear. Start applying these techniques today and you'll notice the difference immediately.
