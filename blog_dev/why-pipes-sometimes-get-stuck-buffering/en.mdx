# Why Pipes Sometimes Get "Stuck": Buffering

You chain a few Unix commands together. Things look good for a second. Then nothing. The cursor just sits there blinking at you.

Don't panic. You've just hit **buffering**.

I remember the first time this happened to me. I was piping a massive log file through `grep` and `awk`, waiting for results. After five minutes I killed the terminal, blamed the tool, and rewrote everything in Python. Turns out the original pipeline was fine. It just needed a flag.

Here's the thing: buffering is a performance trick. Programs save up data, usually around 8KB, before writing it anywhere. This cuts down on system calls and makes batch processing fast. Makes sense for files. But in a pipe? Data should flow continuously. When a program buffers its output, it holds everything hostage. Nothing moves downstream until that buffer fills up.

<Callout type="danger" title="The Problem">
In a pipe, data should flow continuously. If a program buffers its output, it won't pass data down the chain immediately. The result? Your pipeline appears frozen.
</Callout>

So which commands are the usual suspects? Let me save you the Googling I had to do.

### The Usual Suspects

Lots of everyday Unix tools buffer by default when writing to pipes. Here's how to shut that off:

* **`grep`**: `--line-buffered`
* **`sed`**: `-u`
* **`awk`**: `fflush()`
* **`tcpdump`**: `-l`
* **`jq`**: `-u`
* **`tr`**: `-u`

<Callout type="warning" title="A Word of Caution">
`cut` has no built-in way to disable buffering. Annoying, right? You'll need one of the workarounds below instead.
</Callout>

Most of the time, that's all you need. One little flag and your pipeline stops freezing. But what if the problem isn't in the tool itself? What if it's in your script?

### Your Code Might Be the Problem

It happens more than you'd think. You write a script that works fine on its own, pipe it into something else, and everything grinds to a halt. Here's how to fix it in a few common languages.

<Stepper>
  <Step title="Python">
    You've got options. Run with `python -u`, set `PYTHONUNBUFFERED=1` in your environment, or call `sys.stdout.reconfigure(line_buffering=False)`. My go-to is `print(x, flush=True)` — it's right there in the call and I don't have to remember environment variables.
  </Step>
  <Step title="C">
    Call `setvbuf()` on your standard streams to disable buffering explicitly. It's one line and it saves you a headache later.
  </Step>
  <Step title="Ruby and Perl">
    Ruby: `STDOUT.sync = true`. Perl: `$| = 1`. Both are single-line fixes I keep in my dotfiles.
  </Step>
</Stepper>

### When Flags Aren't Enough

Sometimes you don't control the source code. Or the command doesn't have a flag for this. Or you're like me and you just want one solution that works everywhere.

That's when you bring out the bigger tools.

<Callout type="idea" title="Tools That Save the Day">

**`stdbuf`** — This thing turns off buffering for almost any command. It works at the libc level, so it doesn't care what language the tool was written in. Just slap `stdbuf -oL` before your command and you're done.

**`unbuffer`** — This one's trickier. It makes the program think it's writing to a terminal (TTY). Programs usually skip buffering when writing to a terminal, so this forces them into line-by-line mode. Comes with the `expect` package on most systems.

**Rewrite the pipeline** — Sometimes the cleanest fix is restructuring. Multiple `grep` calls can often become a single `awk` script. Fewer stages means fewer buffering points.
</Callout>

### A Real Example

Here's something I ran into last week. I was tailing a log file and filtering for errors:

```bash
tail -f app.log | grep ERROR | awk '{print $1, $5}' > errors.txt
```

Nothing appeared in `errors.txt` for what felt like forever. I knew it was buffering, but which stage? Turns out `grep` was the culprit. Adding `--line-buffered` fixed it instantly:

```bash
tail -f app.log | grep --line-buffered ERROR | awk '{print $1, $5}' > errors.txt
```

Minutes of confusion, seconds of fix. That's buffering for you.

### One More Thing

You might wonder why this isn't the default behavior. Fair question. If everything ran unbuffered, every write would trigger a system call. That's fine for interactive use but terrible for batch jobs. Imagine processing a 10GB file one line at a time with a syscall per line. Your CPU would spend more time on overhead than actual work.

So buffering exists for good reason. You just need to know when to turn it off.

<Callout type="info" title="The Takeaway">
If your pipeline freezes, buffering is almost always the cause. Check your flags first, check your code second, and keep `stdbuf` in your back pocket for everything else.
</Callout>
