## The Core 7 (with real context)

<Stepper>
  <Step title="1. Tuple Unpacking for Swaps">
    
    No `temp` variable, no ceremony:
    
    ```python
    a, b = b, a
    ```
    
    This creates a tuple on the right, then destructures it on the left. Works in-place, zero overhead. The same pattern handles multi-values:
    
    ```python filename="unpacking_demo.py"
# Swap three values in one line
i, j, k = j, k, i

# Unpack a function's multiple return values
x_coord, y_coord, label = get_point()

# Swap dict entries
d = {'a': 1, 'b': 2}
d['a'], d['b'] = d['b'], d['a']  # Now {'a': 2, 'b': 1}
    ```
    
    <Quiz
      question="What does `a, b = b, a` actually do under the hood?"
      options={[
        "Uses a hidden temporary variable automatically",
        "Creates a tuple (b, a) then destructures it to a, b",
        "Swaps memory addresses directly",
        "Calls a special __swap__ method"
      ]}
      correctIndex={1}
      explanation="Python evaluates the right side first, creating a tuple `(b, a)`. Then it unpacks that tuple into `a` and `b` on the left side. No temp variable needed — it's just tuple packing/unpacking."
    />
  </Step>

  <Step title="2. enumerate() — Index + Value">
    
    Stop doing `for i in range(len(items))`. It's error-prone and reads backwards:
    
    ```python
    # ❌ Old way — off-by-one bugs hiding here
    for i in range(len(colors)):
        print(i, colors[i])
    
    # ✅ Python way — what you meant
    for i, color in enumerate(colors):
        print(i, color)
    ```
    
    <Alert type="tip" title="Pro move: start at 1">
      <code>enumerate(colors, start=1)</code> gives 1-based numbering—useful for display, error messages, test cases.
    </Alert>
    
    <Quiz
      question="What's the benefit of `enumerate(items, start=1)`?"
      options={[
        "Makes the loop run faster",
        "Gives 1-based index for display/error messages",
        "Skips the first item automatically",
        "Reverses the iteration order"
      ]}
      correctIndex={1}
      explanation="`start=1` makes the index 1-based instead of 0-based. This is useful when showing item numbers to users (they expect 1, 2, 3... not 0, 1, 2...)."
    />
  </Step>

  <Step title="3. zip() — Loop Two Sequences in Parallel">
    
    Cleaner than manual index management when two sequences must stay in sync:
    
    ```python filename="zip_examples.py"
names    = ["Alice", "Bob", "Charlie"]
scores   = [85, 92, 78]
subjects = ["Math", "Science", "English"]

# Pairing two lists
for name, score in zip(names, scores):
    print(f"{name}: {score}")

# Three-way zipping (tuples auto-unpacked)
for name, score, subject in zip(names, scores, subjects):
    print(f"{name} ({subject}): {score}")

# Dictionaries from two lists
scores_dict = dict(zip(names, scores))
# {'Alice': 85, 'Bob': 92, 'Charlie': 78}
    ```
    
    <Alert type="warning" title="zip truncates to the shortest">
      If <code>names</code> has 3 items and <code>scores</code> has 5, only 3 pairs are produced. Use <code>itertools.zip_longest</code> if you want padding behavior.
    </Alert>
    
    <Quiz
      question="What happens when you zip two lists of different lengths?"

      explanation="`zip()` stops when the shortest iterable is exhausted. If `names` has 3 items and `scores` has 5, you get 3 pairs. Use `itertools.zip_longest()` if you need padding."
    />
  </Step>

  <Step title="4. dict.get() — Avoid KeyError">
    
    The difference between "missing key expected" and "missing key is a bug":
    
    ```python filename="dict_get_demo.py"
config = {"port": 3000, "host": "localhost"}

# Returns default if key missing — no exception
port     = config.get("port", 8080)     # 3000
debug    = config.get("debug", False)   # False (key missing)
host     = config.get("host", "0.0.0.0") # "localhost"

# None is the implicit default
value = config.get("nonexistent")       # None
    ```

  <Step title="5. Set Lookups for O(1) Membership">
    
    Converting a list to a set before checking membership is one of the most common "hidden slowness" fixes:
    
    ```python filename="set_lookup.py"
allowed_actions = {"read", "write", "delete", "update"}

# O(1) — constant time regardless of list size
if action in allowed_actions:
    execute(action)

# ❌ With a list, this is O(n) — scales poorly
allowed_actions_list = ["read", "write", ...]
if action in allowed_actions_list:  # Slower as list grows
    execute(action)
    ```
    
    <Mermaid chart={`\ngraph LR\n    A["action in list<br/>O(n) — slower with size"] -->|Convert once| B[set(allowed_actions)]\n    B --> C["action in set<br/>O(1) — instant always"]\n`} theme="default" />
    
    <Alert type="info" title="Rule of thumb">
      If `x in collection` runs more than a handful of times per request and the collection has more than ~10 items, it should almost certainly be a <code>set</code>.
    </Alert>
    
    <Quiz
      question="Why is `item in some_set` faster than `item in some_list` for large collections?"
      options={[
        "Sets use hash tables for O(1) average lookup",
        "Sets are stored in a faster memory region",
        "Python optimizes set operations at compile time",
        "Lists have to check element types first"
      ]}
      correctIndex={0}
      explanation="Sets use hash tables — membership test is O(1) average case. Lists must scan sequentially O(n). For 10,000 items, set lookup is ~10,000x faster."
    />
  </Step>

  <Step title="6. Underscore `_` for Intentional Ignoring">
    
    Communicates "I'm not using this value on purpose":
    
    ```python filename="underscore_demo.py"
# Unpacking — ignore the key, keep the value
_, result = parse_line(line)

# Loop — ignore the index
for _ in range(3):
    print("repeat")

# Match with explicit wildcard (Python 3.10+)
match status:
    case 200:
        handle_ok()
    case 404:
        handle_missing()
    case _:
        handle_other()  # "anything else"

# Unpacking partial — keep start/end, ignore middle
first, *_, last = [1, 2, 3, 4, 5]
# first=1, last=5
    ```

    ```python filename="fstrings.py"
# Multiline (no more backslash continuation)

Name:   {name}
Status: {"Adult" if age >= 18 else "Minor"}
Score:  {95.5:.1f}%
"""
    ```

  <AccordionItem title="9. pathlib — The Modern File Path Library">
    
    Stop concatenating with <code>os.path.join()</code>:
    
    ```python filename="pathlib_demo.py"
from pathlib import Path

base = Path("/app/data")

# Fluent chaining — works cross-platform
config = base / "config" / f"env_{ENV}.yaml"
logs   = base / "logs" / "app.log"

# Read/write without manual open()
text = config.read_text(encoding="utf-8")
config.write_text(new_text, encoding="utf-8")

# Existence checks
if config.exists() and config.is_file():
    size = config.stat().st_size

# Iterate directory
for log_file in base.glob("**/*.log"):
    print(log_file.name, log_file.stat().st_size)
    ```
    
    <SuccessAlert title="Why pathlib over os.path?">
      <code>Path</code> uses operator overloading (<code>/</code> joins paths) which reads naturally. Methods are chainable. The same code works on Windows and Linux without changes.
    </SuccessAlert>

  <AccordionItem title="10. dataclasses — Less Boilerplate, Same Result">
    
    Before dataclasses, classes with many attributes needed verbose <code>__init__</code>, <code>__repr__</code>, and <code>__eq__</code>. Dataclasses generate those automatically:
    
    ```python filename="dataclass_example.py"
from dataclasses import dataclass, field
from typing import List

@dataclass
class Project:
    name: str
    slug: str
    tags: List[str] = field(default_factory=list)
    starred: bool = False

# ✅ Auto-generated __init__, __repr__, __eq__
p = Project(
    name="Portfolio",
    slug="portfolio",
    tags=["nextjs", "tailwind"],
    starred=True,
)
print(p)  # Project(name='Portfolio', slug='portfolio', ...)

# ✅ Frozen dataclass = immutable (no keyword mutability issues)
from dataclasses import dataclass

@dataclass(frozen=True)
class Point:
    x: float
    y: float

pt = Point(1.0, 2.0)
# pt.x = 3.0  # FrozenInstanceError
    ```

  <AccordionItem title="Bonus: Context Managers — With Defaultdict">
    
    Combine <code>defaultdict</code> with context managers for clean resource handling:
    
    ```python filename="context_manager_demo.py"
from collections import defaultdict
from contextlib import contextmanager
from threading import Lock

# Grouping — defaultdict eliminates "if key in dict" checks
grouped = defaultdict(list)
for item in items:
    grouped[item.category].append(item)

# Custom context manager for timing
@contextmanager
def timer(label):
    import time
    start = time.perf_counter()
    yield
    elapsed = time.perf_counter() - start
    print(f"{label}: {elapsed*1000:.2f}ms")

with timer("database query"):
    results = db.query("SELECT * FROM users")
    ```
    
    <Quiz
      question="What does `defaultdict(list)` eliminate?"
      options={[
        "The need to import collections",
        "The 'if key not in dict' check when appending to grouped data",
        "Memory overhead of regular dicts",
        "The need for type hints"
      ]}
      correctIndex={1}
      explanation="With `defaultdict(list)`, accessing a missing key automatically creates an empty list. So `grouped[item.category].append(item)` works even if the category hasn't been seen before — no `if category not in grouped: grouped[category] = []` needed."
    />
    
  </AccordionItem>
