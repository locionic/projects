# Enforcing the First AS in BGP AS_PATHs

Recent route hijacks reported by Spamhaus have highlighted the importance of enforcing the First AS in BGP AS_PATHs. In these hijacks, attackers created fake AS_PATHs to misdirect traffic and conceal their identity. This post examines the issue and proposes a simple solution: verifying that a BGP peer autonomous system always includes their network as the "First AS" in an advertised route.

## Examining Route Hijacks Involving Forged Paths

To understand the issue, let's examine a hijack reported by Spamhaus involving a prefix belonging to Orange S.A. Using the monocle tool, we can find a BGP UPDATE message related to the hijack:
```json
{
  "aggr_asn": null,
  "aggr_ip": null,
  "as_path": "48237 1299 199524 270118 17072 41128",
  "atomic": false,
  "collector": "rrc26",
  "communities": null,
  "local_pref": 0,
  "med": 0,
  "next_hop": "185.1.8.3",
  "origin": "IGP",
  "peer_asn": 48237,
  "peer_ip": "185.1.8.3",
  "prefix": "90.98.0.0/15",
  "timestamp": 1776039612.0,
  "type": "ANNOUNCE"
}
```
The AS_PATH attribute in this message suggests an implausible relationship between the networks involved. Specifically, the order of the ASes implies that an unused Orange France AS is buying transit from Mexican ISPs, which is then upstreamed to Gcore and Tier 1 providers.

## The Importance of First AS Checking

Routing traffic across the Internet is like shipping a package, where a log is kept of every courier that handles it. In BGP, this log is called the AS_PATH (Autonomous System Path). The AS_PATH attribute is used for path selection, loop prevention, and routing policies.

However, the AS_PATH can be easily manipulated for malicious purposes, such as forging the origin of a route or shortening the path to attract traffic. To prevent these attacks, it's essential to enforce the First AS in the AS_PATH.

## Example 1: Forged Origin Attacks

Consider an example where AS64506 cryptographically signs their routes with an RPKI ROA record, but AS64505 manipulates their AS_PATH to strip AS64505 and originate with AS64506. If AS64502 does not enforce the First AS, the route will appear RPKI-ROV valid and be accepted, effectively hijacking traffic.

## Example 2: Shortening the AS_PATH to Attract Traffic

In another example, AS64506 has two transit providers: AS64503 and AS64505. AS64505 strips itself from the path, and their peer AS64504 does not enforce the First AS. The BGP path selection algorithm will choose the route via AS64504 as the best path, allowing AS64505 to attract traffic and be paid for it.

## Enforcing First AS in AS_PATHs

To prevent these attacks, it's essential to enforce the First AS in the AS_PATH. This can be done by checking whether the leftmost AS in the AS_PATH attribute is equal to the autonomous system number of the peer that sent the message. If the check determines this is not the case, the Error Subcode MUST be set to Malformed AS_PATH.

## Measurement by Breaking the First AS Rule on Purpose

To measure the impact of not enforcing the First AS rule, we set up BGP announcements to neighbors where we purposely violated the rule ourselves. We allocated two IP prefixes and advertised them to Tier 1 External BGP (EBGP) neighbors with a Cloudflare-owned, non-13335 ASN (AS402542) in front of 13335.

Our expectation was that networks that do enforce-first-as would quietly drop the route via RFC 7606 withdrawal method, while networks that do not enforce-first-as would accept the route and install it for forwarding toward our test prefixes.

However, we found that some networks have not implemented the guidance published in RFC 7606 when receiving malformed BGP AS_PATHs and would reset BGP sessions instead of a treat-as-withdraw behavior.

## Conclusion

Enforcing the First AS in BGP AS_PATHs is crucial for internet routing security. By verifying that a BGP peer autonomous system always includes their network as the "First AS" in an advertised route, we can prevent route hijacks and ensure the integrity of the internet routing system.
