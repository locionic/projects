# Katalon, Playwright, Java — Interview Practice

I've sat on both sides of the QA automation interview table. More times than I can count. These six questions come up again and again. Not because they're trick questions. Because they reveal whether you've actually built things or just read about them.

<Callout type="idea" title="How to Use This">
  Don't memorize answers. Understand the concepts well enough to explain them without notes. Interviewers can tell the difference immediately.
</Callout>

---

## Katalon Studio

**Q1. What is Katalon Studio, and how is it different from Selenium?**

I used to give the textbook answer. "Katalon is an all-in-one platform built on Selenium and Appium." True, but not useful. Here's what I'd say now:

Selenium is a library. You build everything around it — test runner, reporting, CI integration, object repository, data handling. Katalon gives you all of that out of the box. An IDE. Built-in keywords. Test case management. Reports that stakeholders can actually read. CI/CD plugins that work without wrestling YAML for three days.

<Callout type="info" title="The Real Difference">
Selenium gives you a browser driver. Katalon gives you a testing platform. You're not choosing a library. You're choosing how much infrastructure you want to build yourself.
</Callout>

I've built frameworks from scratch with Selenium. Took months. Katalon? You're writing tests day one. That's the difference.

**Q2. How do you handle test data in Katalon Studio?**

Data-driven testing. That's the keyword they want to hear. But here's what actually matters:

Bind Excel, CSV, or database files to your test cases. Use Data Files and Global Variables to parameterize everything. But here's the part nobody mentions in tutorials: version control your test data. Put CSV files in git. Use environment-specific data files. Never hardcode credentials. Ever.

<Callout type="warning" title="Learned This the Hard Way">
Hardcoded credentials in test data committed to git. Enough said.
</Callout>

## Playwright

**Q3. What advantages does Playwright have over Selenium or Puppeteer?**

I switched from Selenium to Playwright two years ago. Haven't looked back.

<Callout type="warning" title="No Driver Hassle">
Playwright bundles browser binaries. No separate driver downloads. No version mismatches. No chromedriver.exe version hell.
</Callout>

Multi-browser support is real. Chromium, Firefox, WebKit — same API. Puppeteer only does Chromium. That matters when your users report bugs only in Safari.

Auto-waiting just works. No more `WebDriverWait` with custom ExpectedConditions. Playwright waits for elements to be actionable. It handles SPAs and shadow DOM without extra setup.

<Stepper>
  <Step title="Parallel Contexts">Run multiple independent browser contexts in one instance. Each has its own cookies, storage, cache.</Step>
  <Step title="Parallel Workers">Playwright Test runner distributes tests across CPU cores automatically.</Step>
</Stepper>

Parallel execution used to mean Selenium Grid. Now it's `npx playwright test --workers=4`. Done.

**Q4. How do you handle multiple browser contexts or parallel execution?**

<Stepper>
  <Step title="Browser Contexts">Create independent contexts in one browser instance. Each gets isolated cookies, localStorage, sessionStorage. Perfect for testing multi-user scenarios without spinning up multiple browsers.</Step>
  <Step title="Parallel Workers">Configure workers in playwright.config.ts. Each worker gets its own browser instance. Tests run in parallel automatically.</Step>
</Stepper>

I've run 200 tests in under three minutes this way. Try that with Selenium Grid.

## Java (for Test Automation)

**Q5. Explain OOP concepts in Java and how you apply them in automation.**

Interviewers love this question. Here's the honest answer — I use three of the four daily. The fourth? Rarely.

<Stepper>
  <Step title="Encapsulation">Page Object Model. Everything. Page objects hide locators and actions. Tests call `loginPage.login(user, pass)`. They don't know about IDs, XPaths, or waits.</Step>
  <Step title="Inheritance">BaseTest class handles setup, teardown, screenshot on failure, logging. Every test class extends it. DRY principle in action.</Step>
  <Step title="Polymorphism">Interface for browser operations. Different implementations for Chrome, Firefox, headless. Swap them at runtime via config.</Step>
  <Step title="Abstraction">Abstract base page class. Common methods like `waitForLoad()`, `scrollTo()`. Concrete pages implement page-specific logic.</Step>
</Stepper>

Abstraction is the one I use least. Most page objects don't need an abstract parent. Don't force patterns where they don't belong.

**Q6. How do you handle exceptions in Java test scripts?**

The textbook answer: try-catch-finally. The real answer: it depends on what you're trying to achieve.

<Callout type="info">
  Log the error. Capture a screenshot. Close the browser. Then decide: retry, skip, or fail. Never swallow an exception you can't handle. Let the test fail loud and clear.
</Callout>

I use a base test class with `@AfterMethod` that catches everything. Screenshot on failure. Browser cleanup. TestNG listener logs to Allure. The test itself? Clean. No try-catch clutter. The framework handles the mess.
