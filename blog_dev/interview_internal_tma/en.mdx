# Katalon, Playwright, Java — Interview Practice

I've been on both sides of the QA automation interview table. These six questions come up most often, with answers I'd actually give in a real conversation.

<Callout type="idea" title="How to Use This">
  Don't memorize. Understand the concept well enough to explain it without notes. Interviewers can tell the difference.
</Callout>

## Katalon Studio

**Q1. What is Katalon Studio, and how is it different from Selenium?**

<Callout type="info" title="The All-in-One Platform">
Katalon Studio is a complete test automation platform built on top of Selenium and Appium. 
Unlike Selenium (which is just a library), Katalon provides an IDE, built-in keywords, test case management, reporting, and CI/CD integrations out-of-the-box.
</Callout>

**Q2. How do you handle test data in Katalon Studio?**

You handle test data with Data-Driven Testing (DDT):
- Bind Excel, CSV, or database files to test cases.
- Use built-in **Data Files** and **Global Variables** to parameterize and reuse data.

## Playwright

**Q3. What advantages does Playwright have over Selenium or Puppeteer?**

<Callout type="warning" title="No Driver Hassle">
Playwright bundles browser binaries, eliminating the need to install separate drivers like in Selenium.
</Callout>

- **Multi-browser:** Supports Chromium, Firefox, and WebKit with one API. Puppeteer is Chromium-only.
- **Modern Features:** Auto-waiting works out of the box. SPAs and shadow DOM need no extra setup.
- **Parallelism:** Run multiple independent contexts for faster testing.

**Q4. How do you handle multiple browser contexts or parallel execution?**

<Stepper>
  <Step title="Browser Contexts">Create independent contexts in one browser instance, each with its own cookies and storage.</Step>
  <Step title="Parallel Execution">Use Playwright Test runner to distribute tests across multiple workers (CPU cores) for faster execution.</Step>
</Stepper>

## Java (for Test Automation)

**Q5. Explain OOP concepts in Java and their application in automation.**

<Stepper>
  <Step title="Encapsulation">Grouping data and methods in classes (e.g., Page Object Model).</Step>
  <Step title="Inheritance">Reusing code from base classes (e.g., BaseTest for setup/teardown).</Step>
  <Step title="Polymorphism">Overriding methods for different behaviors based on context.</Step>
  <Step title="Abstraction">Hiding implementation details, exposing only what's needed (e.g., abstract browser classes).</Step>
</Stepper>

**Q6. How do you handle exceptions in Java test scripts?**

You handle exceptions with `try-catch-finally`.
<Callout type="info">
  Log the error, capture a screenshot, close the browser, and optionally retry. Don't swallow exceptions you can't handle — let the test fail clearly.
</Callout>
