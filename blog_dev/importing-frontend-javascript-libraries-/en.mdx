# Importing Frontend JavaScript Libraries without a Build System

As a frontend developer, you may encounter situations where you need to import JavaScript libraries without using a build system. In this guide, we'll explore the different types of JavaScript files that libraries can provide and how to use them in your code.

## Types of JavaScript Files

There are three main types of JavaScript files that a library can provide:

1. **Classic**: This type of file defines a global variable and can be used with a simple `<script src>` tag.
2. **ES Module**: This type of file uses the `import` and `export` syntax and can be used with an import map or a build system.
3. **CommonJS**: This type of file uses the `require` and `module.exports` syntax and is typically used in Node.js environments.

## Finding the Files

To find the files provided by a library, you can check the library's npm package. You can do this by running `npm install` and then checking the `node_modules` directory. The files provided by the library will typically be located in the `dist` directory.

## Example Libraries

Let's take a look at three example libraries: Chart.js, @atcute/oauth-browser-client, and @atproto/oauth-client-browser.

### Chart.js

Chart.js provides three types of files:

* `chart.cjs`: a CommonJS file for Node.js environments
* `chart.js`: an ES Module file
* `chart.umd.js`: a UMD file that can be used with a simple `<script src>` tag

To use Chart.js, you can simply include the `chart.umd.js` file in your HTML:
```html
<script src="./chart.umd.js"></script>
```
### @atcute/oauth-browser-client

@atcute/oauth-browser-client provides an ES Module file:
```javascript
export { configureOAuth } from './environment.js';
export * from './errors.js';
export * from './resolvers.js';
```
To use this library, you'll need to create an import map in your HTML:
```html
<script type="importmap">
{
  "imports": {
    "nanoid": "./node_modules/nanoid/bin/dist/index.js",
    "nanoid/non-secure": "./node_modules/nanoid/non-secure/index.js",
    "nanoid/url-alphabet": "./node_modules/nanoid/url-alphabet/dist/index.js",
    "@atcute/oauth-browser-client": "./node_modules/@atcute/oauth-browser-client/dist/index.js",
    "@atcute/client": "./node_modules/@atcute/client/dist/index.js",
    "@atcute/client/utils/did": "./node_modules/@atcute/client/dist/utils/did.js"
  }
}
</script>
```
You can then import the library in your JavaScript code:
```javascript
import { configureOAuth } from '@atcute/oauth-browser-client';
```
### @atproto/oauth-client-browser

@atproto/oauth-client-browser provides a CommonJS file:
```javascript
__exportStar(require("@atproto/oauth-client"), exports);
__exportStar(require("./browser-oauth-client.js"), exports);
__exportStar(require("./errors.js"), exports);
var util_js_1 = require("./util.js");
```
To use this library, you can use a CDN like esm.sh to translate the CommonJS file to an ES Module:
```html
<script type="module" src="script.js"></script>
```
```javascript
import { BrowserOAuthClient } from "https://esm.sh/@atproto/oauth-client-browser@0.3.0";
```
## Conclusion

Importing frontend JavaScript libraries without a build system can be challenging, but it's possible with the right approach. By understanding the different types of JavaScript files that libraries can provide, you can choose the best approach for your project. Whether you're using a classic global variable, an ES Module, or a CommonJS file, there's a way to use the library without a build system.
