# Why I'm Learning Rust as a Web Developer

I spent six months avoiding Rust. "It's for kernel developers," I told myself. "Too hard for web stuff." Then I needed a CLI tool that was faster than Python and more portable than Node, and I got curious enough to try.

Two months later, I'm writing Rust regularly. Not because I switched careers, and not because every project needs it. Because knowing Rust has quietly made me a better JavaScript and Python developer.

## What Rust Actually Teaches You

The borrow checker — the thing everyone warns you about — isn't punishment. It's a teacher.

When you write JavaScript, you don't think about who owns a variable, when it gets freed, or if two parts of your code might mutate it at the same time. The runtime handles all of that, invisibly. Rust forces you to think about it explicitly, at compile time.

<Callout type="info">
You don't need to write production Rust to get value from it. Learning the ownership model will change how you reason about JavaScript, Python, Go — anything with mutable state.
</Callout>

The first few weeks I spent arguing with the compiler. After that I started understanding *why* it was complaining — and I noticed I was writing JavaScript differently. Fewer in-place mutations. More explicit ownership of data. Functions that don't silently share mutable references.

## The Mental Model Shift

In JavaScript, this feels perfectly natural:

```javascript
let data = fetchData();
processA(data);
processB(data);
```

But what does `processA` do to `data`? Maybe nothing. Maybe it mutates it. You have to read the function. Rust makes ownership explicit:

```rust
let data = fetch_data();
process_a(&data);       // borrows — data is read-only here
process_b(&mut data);   // mutates — borrow checker tracks this
```

Once that model clicks, you start noticing all the places in your JavaScript where you're passing an object to a function and hoping it doesn't modify it. Rust just makes you say it out loud.

## Where Rust Actually Shows Up in Web Dev

This isn't theoretical. Rust has real footholds in the web ecosystem:

**WebAssembly** — Rust compiles to WASM better than almost anything else. If you need a performance-critical piece of logic running in the browser — image processing, cryptography, parsing — Rust + WASM is a serious option. The [wasm-pack](https://rustwasm.github.io/wasm-pack/) tooling has gotten genuinely good.

**CLI tools** — Building a single-binary tool with [Clap](https://github.com/clap-rs/clap) is fast, and the result ships with no runtime dependencies. I built a small file-processing tool for a project, and distributing it meant handing someone a binary. That's it.

**Edge Functions** — Cloudflare Workers support Rust. Cold start times are much lower than Node, and for latency-sensitive middleware (auth checks, A/B routing), that matters.

**Developer tooling** — A lot of the tooling you're already using is written in Rust: Biome, SWC, Turbopack, Rolldown. These exist because Rust makes it practical to build JavaScript tooling that's 10–50x faster than the equivalent in Node.

## Getting Started Without Being Overwhelmed

The Rust book is free at [doc.rust-lang.org/book](https://doc.rust-lang.org/book/) and it's genuinely one of the better technical books I've read. Work through the first ten chapters. You'll hit the borrow checker hard around chapter 4. Stay with it.

<Callout type="idea" title="Where to start practically">
**Rustlings** — small exercises in the terminal. Good for muscle memory.  
**Advent of Code in Rust** — puzzle-sized problems where you're forced to solve real tasks.  
**Build a CLI tool** — pick something you'd normally use Python for and build it in Rust instead.
</Callout>

Don't try to write web servers in Rust on week one. That's the fastest way to get discouraged. Start with self-contained programs where you control all the inputs.

## My Take After Six Months

I'm not going to claim Rust replaced my day-to-day stack. Most of my work is still JavaScript and Python. But the mental models I've picked up — thinking about ownership, being explicit about mutation, caring about whether a function borrows or consumes its inputs — have genuinely seeped into how I write in those languages.

The investment is real. The compiler is unforgiving at first. But there's something deeply satisfying about writing code that the compiler has verified for correctness *before you run it*. Coming back to JavaScript after that, you start noticing every `undefined` and every implicit mutation as a thing you chose to accept.

That's not a bad thing to notice.
