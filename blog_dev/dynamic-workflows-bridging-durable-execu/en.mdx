# Dynamic Workflows: Bridging Durable Execution and Dynamic Deployment

When Cloudflare launched Workers eight years ago, it was a direct-to-developers platform. Over the years, the ecosystem has expanded to enable platforms to build on Workers directly and allow their customers to ship code through multi-tenant applications. Today, we see various applications on Workers, including those where AI writes the implementation, multi-tenant SaaS, agents that write and run their own tools, and CI/CD products.

## The Gap Between Durable and Dynamic Execution

Cloudflare Workflows is a durable execution engine that turns a `run(event, step)` function into a program that survives failures, can sleep for hours or days, and resumes exactly where it left off when the isolate is recycled. However, Workflows has always had one assumption baked in: the workflow code is part of the deployment. This works fine if you own all the code, but it stops working when you want to let your customer ship their workflow.

## Introducing Dynamic Workflows

Dynamic Workflows is a small library that lets a single Worker, the Worker Loader, route every `create()` call to a different tenant's code and have the Workflows engine dispatch `run(event, step)` back to that same code when the workflow executes. This library is roughly 300 lines of TypeScript and is available as `@cloudflare/dynamic-workflows`.

## How Dynamic Workflows Works

Here's an example of how Dynamic Workflows works:
```typescript
import { createDynamicWorkflowEntrypoint, DynamicWorkflowBinding, wrapWorkflowBinding } from '@cloudflare/dynamic-workflows';

// The library looks this class up on cloudflare:workers exports.
export { DynamicWorkflowBinding };

function loadTenant(env, tenantId) {
  return env.LOADER.get(tenantId, async () => ({
    compatibilityDate: '2026-01-01',
    mainModule: 'index.js',
    modules: { 'index.js': await fetchTenantCode(tenantId) },
    env: { WORKFLOWS: wrapWorkflowBinding({ tenantId }) },
  }));
}

// Register this as class_name in wrangler.jsonc.
export const DynamicWorkflow = createDynamicWorkflowEntrypoint<Env>(
  async ({ env, metadata }) => {
    const stub = loadTenant(env, metadata.tenantId);
    return stub.getEntrypoint('TenantWorkflow');
  }
);

export default {
  fetch(request, env) {
    const tenantId = request.headers.get('x-tenant-id');
    return loadTenant(env, tenantId).getEntrypoint().fetch(request);
  },
};
```
The tenant writes plain, idiomatic Workflows code, which is dispatched by the Dynamic Workflows library:
```typescript
import { WorkflowEntrypoint } from 'cloudflare:workers';

export class TenantWorkflow extends WorkflowEntrypoint {
  async run(event, step) {
    return step.do('greet', async () => `Hello, ${event.payload.name}!`);
  }
}

export default {
  async fetch(request, env) {
    const instance = await env.WORKFLOWS.create({ params: await request.json() });
    return Response.json({ id: await instance.id });
  },
};
```
## Three Layers of Execution

Dynamic Workflows consists of three layers:

1. The Workflows engine (platform) on top
2. Your Worker Loader in the middle
3. Your tenant's code (a Dynamic Worker) on the bottom

When a request reaches the Worker Loader, it routes the execution to the correct dynamic code on the fly. The rest of the execution is a handoff between these three layers, left-to-right in time: the request enters, bounces up to the engine, is persisted, and later bounces back down again.

## The Escape Hatch

If you want to subclass `WorkflowEntrypoint` yourself, the library exposes the lower-level `dispatchWorkflow` primitive that `createDynamicWorkflowEntrypoint` is built on:
```typescript
import { dispatchWorkflow } from '@cloudflare/dynamic-workflows';

export class MyDynamicWorkflow extends WorkflowEntrypoint {
  async run(event, step) {
    return dispatchWorkflow(
      { env: this.env, ctx: this.ctx },
      event,
      step,
      ({ metadata, env }) => loadRunnerForTenant(env, metadata),
    );
  }
}
```
## Conclusion

Dynamic Workflows is a powerful tool that enables multi-tenant applications to ship custom code without additional overhead. By bridging durable execution and dynamic deployment, Dynamic Workflows provides a scalable and efficient solution for platforms to build on Workers directly. The unit economics of running a platform like this are, frankly, absurd, and we're excited to see what developers will build with this technology.
