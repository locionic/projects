# Defending Against Frontier Cyber Models: Cloudflare's Architecture

A few weeks ago, we discussed Project Glasswing and our observations on the impact of cyber frontier models on our own code. The key takeaway was that the architecture surrounding a vulnerability matters more than the speed of patching. In this post, we'll delve into Cloudflare's architecture and how it helps defend against frontier cyber models.

## What Changes with Cyber Frontier Models

Cyber frontier models like Mythos can find vulnerabilities, reason through exploit chains, and generate working proofs faster than earlier models. While these models don't change the shape of an intrusion, they significantly impact the speed and scale of attacks. Against a hardened target, they still need to probe and adapt, often producing more noise than a careful human operator.

## The Speed of Discovery

Frontier models make it easier to search large bodies of public code, including open-source libraries. This doesn't mean every bug in a library is exploitable, but it does give attackers a shared surface to study at scale. When a real, reachable vulnerability exists, a model can help find it, reason about possible exploit paths, and generate proof-of-concept variants faster than maintainers and defenders can review every downstream use.

## Exploit Volume and Adaptation

A model can produce thousands of variations of a single exploit and run reconnaissance at the same scale. While this volume gives an attacker an advantage, it won't necessarily get them past signature-based detections. Many of those iterations will have the same underlying signature, so a rule that catches the first one will catch the rest. Adaptation is how they will get past signature-based detections.

## The Impact of a Vulnerability

No architecture catches everything. After a vulnerability is exploited, the question is: where can the attacker get to with one identity, one path, or one credential, before something else stops them? If the answer is "anywhere they want," the vulnerability was never the problem. The architecture around the vulnerability was.

## Cloudflare's Superpower: Visibility

We see roughly a fifth of the world's web traffic, which tells us, in real time, which payloads are mutating, which patterns are picking up, and where attacker tooling is moving next. Two teams turn this visibility into defense:

1. **Cloudforce One**: Our threat intelligence, research, and operations team, which sits within the Cloudflare security organization. They turn what we see across the network into insights the rest of the stack can act on.
2. **WAF Engine Team**: The team that owns the WAF engine that does the actual detecting. They release rules within hours of a proof-of-concept of an attack becoming known.

## Scores over Signatures

Signature-based defenses were built for a world where novel exploits were scarce and variations took weeks. With the advent of frontier models, this is no longer good enough. We layer ML-based detection in front of traditional signature-based WAF. The model is trained on a large body of past attack traffic and catches new variants of vulnerabilities before they're publicly known.

## The Architecture Around the Vulnerability

Our capabilities only matter once they're stacked in front of an application. The first layer in our defense-in-depth approach is the **WAF**. Anything that matches a known-bad pattern gets dropped before it reaches the application.

* **API Shield**: We run a positive security model on the API surface, describing what a valid request to each API looks like. Anything that doesn't fit doesn't get through.
* **Bot Management**: We catch probing traffic on our network before frontier models can build a map. We score every request on how likely it is to be automated.
* **Zero Trust Network Access**: We use this for every internal application. The implicit trust of being inside the network is replaced with explicit per-request identity and policy for every employee accessing every tool.
* **Require Access Protection**: We built this to ensure newly deployed or misconfigured applications can't be reachable before an access policy is in place.
* **IdP Federation**: We make it easier to keep a secure by default posture consistent across every Cloudflare account.
* **MCP Server Portal**: We give teams a controlled way to connect AI agents to enterprise systems. Agents access MCP servers that are centrally managed through a single portal, with every action logged.
* **AI Gateway**: We run this in front of our internal AI tools, with the same scoring and visibility as AI Security for Apps.

## Where Your Teams Can Start

Frontier models can help attackers find vulnerabilities, adapt payloads, and move faster, but they still have to pass through the layered defense you deploy in front of your application. That's where teams should start:

* Put inspection in front of public applications.
* Define what valid API traffic looks like.
* Use bot detection to limit automated probing.
* Require identity and access policy before any internal tool is reachable.
* For AI and agentic systems: Route model traffic through a gateway. Keep agents connected through approved MCP servers. Log what they do.

The goal is to make sure that when one layer misses, the next layer limits what the attacker can see, reach, or change.
