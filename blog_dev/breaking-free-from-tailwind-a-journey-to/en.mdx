# Breaking Free from Tailwind: A Journey to Structured CSS

As a developer, I've had a long-standing relationship with Tailwind, a popular utility-first CSS framework. Eight years ago, I was thrilled to discover Tailwind, which helped me create numerous small sites with ease. However, after spending the last week migrating a couple of sites away from Tailwind and towards more semantic HTML and vanilla CSS, I've gained a new appreciation for the importance of structuring my CSS code.

## The Realization

When I started thinking about structuring my CSS, I was intimidated. But as I delved into blog posts and articles discussing CSS organization techniques, I realized that every CSS codebase has multiple aspects to manage, such as layouts, fonts, colors, and common components. Tailwind has systems in place for some of these aspects, and I decided to imitate the systems I liked while creating my own.

## Structuring CSS: A Step-by-Step Approach

Here are the key aspects of my CSS codebase, along with my thoughts on how to structure each one:

### 1. Reset

I copied Tailwind's "preflight styles" to create a reset stylesheet. This ensures consistency across my site and helps me avoid unexpected styling issues.

```css
* {
  box-sizing: border-box;
}
```

### 2. Components

I organize my CSS by "components," similar to Vue or React components. Each component has a unique class, and its CSS is stored in a separate file. This approach makes it easier to manage and edit individual components without affecting others.

```html
<figure class="zine horizontal"><img src="whatever.jpg"></figure>
```

```css
.zine {
  /* styles */
  &.horizontal {
    /* styles */
  }
  &.vertical {
    /* styles */
  }
  &:hover {
    /* styles */
  }
}
```

### 3. Colors

I created a `colors.css` file with variables for all the colors used in my site. This ensures consistency and makes it easier to manage colors across the site.

```css
:root {
  --pink: #fea0c2;
  --pink-light: #F9B9B9;
  --red: #f91a55;
  --orange: rgb(222, 117, 31);
  /* more colors */
}
```

### 4. Font Sizes

I defined a set of font size variables, similar to Tailwind's font size classes. This allows me to easily set font sizes without having to remember specific values.

```css
:root {
  --size-xs: 0.75rem;
  --line-height-xs: 1rem;
  --size-sm: 0.875rem;
  --line-height-sm: 1.25rem;
  /* more font sizes */
}
```

```css
h3 {
  font-size: var(--size-lg);
  line-height: var(--line-height-lg);
}
```

### 5. Utilities

I created a set of utility classes for common styling needs, such as buttons or screenreader-only text. These classes are used across multiple components and help maintain consistency.

```css
.sr-only {
  /* styles */
}
```

### 6. Base Styles

I defined a set of base styles that apply across the entire site. These styles are carefully chosen to ensure consistency and are used sparingly.

```css
section {
  --inner-width: 950px;
  padding: 3rem max(1rem, (100% - var(--inner-width))/2);
}
```

### 7. Spacing

I'm still working on developing a principled approach to managing padding and margins. Currently, I'm using a combination of CSS grid and utility classes to manage spacing.

```css
section > *+* {
  margin-top: 1rem;
}
```

### 8. Responsive Design

I'm using CSS grid to create flexible, responsive layouts that don't rely on media queries. This approach has been challenging but rewarding.

```css
display: grid;
grid-template-columns: repeat(auto-fit, minmax(min(100%, 400px), max-content));
justify-content: center;
```

### 9. Build System

I'm using esbuild to bundle my CSS files for production. This allows me to take advantage of modern CSS features while still supporting older browsers.

```bash
esbuild style.css --bundle --loader:.svg=dataurl --loader:.woff2=file --outfile=/tmp/out.css
```

## Why Migrate Away from Tailwind?

I migrated away from Tailwind for several reasons:

* Tailwind has become more reliant on a build system, which I've avoided using.
* I've outgrown Tailwind's limitations and want to explore more advanced CSS features.
* I've mixed vanilla CSS and Tailwind in the same project, which has been difficult to maintain.
* I'm curious about writing more semantic HTML and exploring CSS features like container queries and subgrid.

## Conclusion

Migrating away from Tailwind has been a challenging but rewarding experience. I've gained a deeper understanding of CSS and developed a more principled approach to structuring my code. While Tailwind is a powerful tool, I believe that understanding the underlying CSS concepts is essential for creating maintainable, scalable, and performant web applications.
