# Rebuilding Search Architecture for High Availability in GitHub Enterprise Server

GitHub's search functionality is a critical component of its platform, powering features like search bars, filtering experiences, and release pages. To make search more durable and reduce maintenance time for GitHub Enterprise Server administrators, the company has spent the last year rebuilding its search architecture.

## Challenges with Previous Architecture

In previous versions of GitHub Enterprise Server, search indexes were prone to becoming damaged or locked during upgrades or maintenance, causing problems for administrators. This was due to the integration with Elasticsearch, which didn't support a leader/follower pattern. To work around this, GitHub engineers created an Elasticsearch cluster across primary and replica nodes, but this eventually led to issues like data inconsistencies and locked states.

## Introducing Cross Cluster Replication (CCR)

After years of trying to make the previous architecture more stable, GitHub has adopted Elasticsearch's Cross Cluster Replication (CCR) feature to support high availability in GitHub Enterprise Server. CCR allows for the replication of data between independent single-node Elasticsearch clusters, ensuring that data is durably persisted within the Elasticsearch cluster.

## How CCR Works

In the new architecture, each Enterprise server instance operates as an independent single-node Elasticsearch cluster. CCR copies data once it's been persisted to the Lucene segments, ensuring that data is replicated correctly. To enable CCR, GitHub Enterprise Server administrators need to bootstrap the process by attaching followers to existing indexes and enabling auto-follow for new indexes.

### Bootstrap Workflow

Here's an example of the bootstrap workflow:
```python
function bootstrap_ccr(primary, replica):
    # Fetch current indexes on each node
    primary_indexes = list_indexes(primary)
    replica_indexes = list_indexes(replica)

    # Filter out system indexes
    managed = filter(primary_indexes, is_managed_ghe_index)

    # Initialize follower patterns for indexes without followers
    for index in managed:
        if index not in replica_indexes:
            ensure_follower_index(replica, leader=primary, index=index)
        else:
            ensure_following(replica, leader=primary, index=index)

    # Setup auto-follower patterns for new indexes
    ensure_auto_follow_policy(
        replica,
        leader=primary,
        patterns=[managed_index_patterns],
        exclude=[system_index_patterns]
    )
```
## Getting Started with CCR Mode

To start using the new CCR mode, GitHub Enterprise Server administrators need to:

1. Reach out to support@github.com to request the new HA mode.
2. Download the required license.
3. Set `ghe-config app.elasticsearch.ccr true`.
4. Run a `config-apply` or upgrade to version 3.19.1.

When the GitHub Enterprise Server restarts, Elasticsearch will migrate the installation to use the new replication method. This update may take some time depending on the size of the instance.

## Conclusion

GitHub's rebuilt search architecture using CCR improves durability and reduces maintenance time for GitHub Enterprise Server administrators. While the new HA method is optional for now, it will become the default over the next two years. Administrators can try out the new HA mode and provide feedback to GitHub.
