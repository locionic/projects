# Cloudflare's Response to the "Copy Fail" Linux Vulnerability
===========================================================

On April 29, 2026, a Linux kernel local privilege escalation vulnerability, known as "Copy Fail" (CVE-2026-31431), was publicly disclosed. Cloudflare's Security and Engineering teams immediately assessed the vulnerability and took swift action to mitigate its impact.

## Background

Cloudflare operates a massive Linux server infrastructure across 330 cities worldwide. To manage updates effectively, we maintain a custom Linux kernel build based on the community's Long-Term Support (LTS) versions. Our kernel release process involves generating a new internal kernel build approximately every week, which undergoes testing in our staging data centers before a global rollout.

## The Copy Fail Vulnerability

The vulnerability affects the Linux kernel's internal crypto API, specifically the algif_aead module, which facilitates Authenticated Encryption with Associated Data (AEAD) ciphers. An unprivileged program can exploit this vulnerability by:

1. Opening an AF_ALG socket and binding to an AEAD template.
2. Setting a key and accepting a request socket.
3. Submitting input via sendmsg() or splice().
4. Executing the operation using recvmsg().

The vulnerability allows an attacker to manipulate the page cache, effectively editing a setuid binary and executing arbitrary code with root privileges.

## How We Responded

When the vulnerability was disclosed, our Security and Engineering teams worked in parallel to:

1. Map the blast radius: Determine which kernel versions were vulnerable and assess the potential exposure.
2. Validate coverage: Confirm that our existing behavioral detections could identify the exploit pattern.
3. Proactive threat hunting: Search for signs of exploitation before the vulnerability was publicly known.
4. Engineer a mitigation: Build a runtime mitigation to protect the fleet without breaking production services.
5. Continue software updates: Deliver an updated Linux kernel and roll it out across our servers.

## Validating Detection Coverage

Our security team confirmed that our existing endpoint detection would catch this exploit. Our servers run behavioral detection that continuously monitors process execution patterns. When our engineers validated the vulnerability internally, the detection platform flagged it within minutes without a signature update, rule change, or human intervention.

## Hunting for Exploitation

Our security team operated on the principle of assuming compromise until proven otherwise. We searched for distinctive traces in kernel logs, pulled access logs for affected systems, and reconstructed interactive activity on potentially affected infrastructure. We found no evidence of exploitation.

## Incident Timeline and Impact

The following timeline highlights the key events and milestones in our response:

* 2026-04-29 16:00: Copy Fail publicly disclosed.
* 2026-04-29 ~21:00: Security and Engineering teams began assessing fleet exposure and mitigation options.
* 2026-04-29 22:52: Security confirmed existing behavioral detection covered the Copy Fail exploit pattern.
* 2026-04-29 23:01: Existing behavioral detection generated a high-severity alert for exploit-like activity.
* 2026-04-30 03:14: Security incident declared to drive cross-functional collaboration and urgency.
* 2026-04-30 (morning): Engineering tested the bpf-lsm mitigation program and made it production-ready.
* 2026-04-30 14:25: Engineering incident declared to coordinate mitigation program and Linux patch rollout.
* 2026-04-30 ~17:00: Decision made to ship a patched build of the previous LTS line through reboot automation.
* 2026-04-30 (afternoon): Visibility pipeline (eBPF tracing of AF_ALG socket usage) deployed fleet-wide.
* 2026-04-30 (evening): bpf-lsm mitigation program rolled out behind a separate gate to fully mitigate the fleet.

## Mitigation

We pursued two mitigation strategies:

1. Removing the algif_aead module: This would have impacted software that leverages the kernel crypto API.
2. Bpf-lsm: We developed and deployed a BPF Linux Security Module program to deny the socket_bind LSM hook for everyone else, completely blocking the front door for any exploits.

The bpf-lsm mitigation program was rolled out behind a separate gate to fully mitigate the fleet. End-to-end verification on a previously-vulnerable test node confirmed the exploit no longer worked.

## Conclusion

Cloudflare's preparedness and swift response to the "Copy Fail" vulnerability ensured that our customers' data and services were not impacted. Our existing behavioral detections and proactive threat hunting allowed us to quickly identify and mitigate the vulnerability. We continue to invest in our security capabilities to protect our customers and the Internet at large.
