## Introduction

Every time you push code, you want to be sure it doesn't break anything. **CI/CD** (Continuous Integration / Continuous Deployment) automates this process — running tests, building your app, and deploying it automatically.

**GitHub Actions** is GitHub's built-in CI/CD platform. It's free for public repositories and easy to set up. In this guide, you'll learn how to create a complete CI/CD pipeline for a Next.js project.

---

## What Is CI/CD?

- **Continuous Integration (CI)** — automatically run tests every time code is pushed
- **Continuous Deployment (CD)** — automatically deploy your app when tests pass

Together they eliminate manual steps and catch bugs before they reach production.

---

## 1. How GitHub Actions Works

GitHub Actions uses **workflow files** stored in `.github/workflows/`. Each workflow is a YAML file that defines:

- **Triggers** — when to run (on push, pull request, schedule)
- **Jobs** — what to do (install, test, build, deploy)
- **Steps** — individual commands inside each job

---

## 2. Your First Workflow

Create this file in your project:
```
.github/workflows/ci.yml
```
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

      - name: Build project
        run: npm run build
```

Every time you push to `main` or open a pull request, GitHub will automatically run this workflow.

---

## 3. Adding Playwright E2E Tests

If you have Playwright tests, add this step after your build:
```yaml
      - name: Install Playwright browsers
        run: npx playwright install --with-deps

      - name: Run E2E tests
        run: npx playwright test
```

---

## 4. Deploy to Vercel Automatically

Vercel already auto-deploys when you push to GitHub, but you can add a manual deploy step for more control:
```yaml
      - name: Deploy to Vercel
        run: npx vercel --prod --token=${{ secrets.VERCEL_TOKEN }}
```

To add your Vercel token:
1. Go to **vercel.com** → Settings → Tokens → Create token
2. Go to your GitHub repo → Settings → Secrets → New secret
3. Name it `VERCEL_TOKEN` and paste your token

---

## 5. Viewing Results

After pushing, go to your GitHub repo and click the **Actions** tab. You'll see:

- ✅ Green checkmark — all steps passed
- ❌ Red cross — something failed, click to see the logs

---

## 6. Best Practices

- **Run tests on every PR** — catch bugs before merging
- **Cache dependencies** — use `cache: 'npm'` to speed up workflows
- **Use secrets** — never hardcode API keys, use `${{ secrets.YOUR_SECRET }}`
- **Keep workflows small** — one workflow per concern (CI, deploy, lint)

---

## Conclusion

GitHub Actions makes CI/CD accessible to every developer. With a simple YAML file you can:

1. Run tests automatically on every push
2. Build your project to catch errors early
3. Deploy automatically when everything passes

Start with a basic CI workflow today and expand it as your project grows.
