# A Data Model for Git: Improving Documentation through Community Efforts

This fall, I embarked on a journey to improve Git's documentation. As someone who has written about topics they think need better explanations, I decided to take a different approach this time – contribute directly to the official documentation. Alongside Marie, we made several changes to the Git documentation, including creating a new data model document.

## A Data Model for Git

While working on the documentation, we noticed that Git frequently uses terms like "object," "reference," and "index" without providing a clear explanation of their meanings or relationships to core concepts like "commit" and "branch." To address this, we wrote a new "data model" document that provides an accurate and concise explanation of Git's internal workings. The document is available [here](https://git-scm.com/docs/git-data-model), and it will be included on the Git website after the next release.

Creating an accurate data model proved to be a challenging task. I had to learn new details and make several changes during the review process. For example, I discovered how merge conflicts are stored in the staging area. The experience reinforced the importance of having a clear and accurate data model to understand how Git works.

## Updates to Git Push, Git Pull, and More

In addition to creating the data model, I worked on updating the introduction to several core man pages, including `git push`, `git pull`, `git add`, and `git checkout`. To ensure that the changes were effective, I sought feedback from test readers on Mastodon.

## Getting Test Readers to Identify Problems

I asked for test readers to read the current documentation and provide feedback on what they found confusing or unclear. About 80 test readers responded, providing valuable insights into the documentation's shortcomings. The feedback included:

* Terminology that needed explanation (e.g., "pathspec," "reference," "upstream")
* Confusing sentences or sections
* Suggestions for additions (e.g., common use cases)
* Inconsistencies in the documentation

Most test readers had been using Git for at least 5-10 years, which helped identify areas where the documentation needed improvement.

## The Man Page Changes

Based on the feedback, we updated the following man pages:

* `git add` (before, after)
* `git checkout` (before, after)
* `git push` (before, after)
* `git pull` (before, after)

The changes to `git push` and `git pull` were particularly significant, as we added sections explaining the terms "upstream branch" and "push refspec." These changes required careful consideration to ensure that the documentation was both clear and accurate.

## On the Process for Contributing to Git

Contributing to Git's documentation was a complex process that required understanding the project's development workflow. I used GitGitGadget to create pull requests, which were then converted into emails with patches attached. I also used my email client to respond to emails and wrapped my text to 80 character lines to conform to the mailing list norm.

I would like to thank the many people who helped me navigate the contribution process and review the changes, including Emily Shaffer, Johannes Schindelin, Patrick Steinhardt, Ben Knoble, Junio Hamano, and others.
