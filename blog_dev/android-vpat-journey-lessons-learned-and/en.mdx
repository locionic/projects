# Android VPAT Journey: Lessons Learned and Solutions Implemented

## Background

A Voluntary Product Accessibility Template (VPAT) is a document that evaluates a product's alignment with accessibility standards. At Slack, we conducted a VPAT for our Android app in 2024, following a significant UI change. This process helped us identify accessibility issues across our app, which we addressed through a comprehensive triage process. In this article, we share the major recurring themes of Android accessibility issues we encountered, along with the resolution strategies and lessons learned.

## Identified Themes

During our VPAT process, we identified several accessibility issues that can be grouped into the following themes:

* **Error messages are inaccessible**: Error messages were not communicated to screen reader users, making it difficult for them to understand the outcome of their actions.
* **Headings are not identified**: Headings did not convey clear semantic information, making it difficult for screen reader users to navigate efficiently.
* **Missing a11y label on EditField**: Some edit fields did not have permanent labels, which can cause issues for users with cognitive difficulties.
* **Incorrect number of items in list**: TalkBack announced an incorrect number of items in lists, which can be confusing for users.
* **Drag and drop in the workspace switcher is inaccessible**: The drag-and-drop feature in the workspace switcher required users to select a workspace and drag it to a new location, which can be difficult for users with dexterity issues.
* **Strikethrough information not conveyed to screen reader users**: The strikethrough format was not conveyed to screen reader users, which can make it difficult for them to understand the content.
* **Errors indicated by color alone**: Error messages were displayed in red text, which can be inaccessible to users who cannot perceive color.

## Issue Resolution

We implemented several solutions to address the accessibility issues we identified:

### P1: Error messages are inaccessible

We modified the `OutlinedTextField` to announce errors immediately below the editing field. We also updated the `SKBanner` to announce errors to screen reader users.

```java
// Modified OutlinedTextField
public class OutlinedTextField extends TextInputLayout {
    // ...
    public void setError(CharSequence error) {
        // Announce error to screen reader users
        announceForAccessibility(error);
    }
}

// Updated SKBanner
public class SKBanner extends RelativeLayout {
    // ...
    public void setError(CharSequence error) {
        // Announce error to screen reader users
        announceForAccessibility(error);
    }
}
```

### P2: Headings are not identified

We identified and fixed issues with missing headings inside lists. We also tested other apps to ensure our implementation was consistent with Android standards.

### P2: Missing a11y label on EditField

We added explicit labels to edit fields, such as the search field, to help users easily identify the purpose of the field.

```java
// Added explicit label to search field
public class SearchField extends EditText {
    public SearchField(Context context) {
        super(context);
        // Add explicit label
        setContentDescription("Search");
    }
}
```

### P2: Incorrect number of items in list

We introduced a new `SKListAccessibilityDelegate` to overwrite the accessibility collection info with the correct number of items.

```java
// Introduced SKListAccessibilityDelegate
public class SKListAccessibilityDelegate extends AccessibilityDelegate {
    // ...
    public void onInitializeAccessibilityNodeInfo(View host, AccessibilityNodeInfo info) {
        // Overwrite accessibility collection info with correct number of items
        info.setCollectionInfo(CollectionInfo.obtain(getItemCount(), true, false));
    }
}
```

### P2: Drag and drop in the workspace switcher is inaccessible

We implemented an Edit mode in the workspace switcher, which allows users to move workspaces using custom actions.

```java
// Implemented Edit mode in workspace switcher
public class WorkspaceSwitcher extends RecyclerView {
    // ...
    public void enterEditMode() {
        // Display six-dot drag handle for each workspace row
        for (int i = 0; i < getItemCount(); i++) {
            WorkspaceRow row = getWorkspaceRowAt(i);
            row.setDragHandleVisible(true);
        }
    }
}
```

### P2: Strikethrough information not conveyed to screen reader users

We decided not to convey strikethrough information to screen reader users, as it can be verbose and confusing.

### P3: Errors indicated by color alone

We updated our design to display error messages in a more intuitive way, such as using a red border or an error icon.

```java
// Updated error message design
public class ErrorMessage extends RelativeLayout {
    // ...
    public void setError(CharSequence error) {
        // Display error message with red border or error icon
        setErrorDrawable(getResources().getDrawable(R.drawable.error_border));
    }
}
```

## Conclusion

Through our VPAT journey, we gained valuable insights into the importance of accessibility in mobile app development. We learned that seemingly minor UI changes can have a significant impact on our users' experience. We also recognized the need to be more diligent in adding TalkBack custom actions for gestures that may not be easily discoverable. Our experience highlights the importance of continuous evaluation and improvement of accessibility features in mobile apps.
