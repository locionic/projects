I used to think “best Python libraries” was just a listicle people pasted into docs, then I watched my own weekend projects get derailed because I picked something that sounded popular but didn’t actually fit the job.

So here’s my version: not a vague roll call, but the libraries I’d actually bet on going into 2026. I’m grouping them by what they’re good at, and I’m a little picky about the criteria—“works” isn’t the same as “works well when you ship.”

## Web development + APIs

**FastAPI**  
Still my go-to when I want speed *and* sane ergonomics. You get request/response modeling, validation that doesn’t feel like punishment, and a nice path from a small service to something serious.

**Django**  
If you want batteries included and you’re building more than “just an API”—admin, auth, migrations, the whole web app package—Django earns its keep. It’s opinionated in a way that can be frustrating at first, then incredibly comforting once the project grows.

**Flask**  
Lightweight and flexible, and it’s still great for smaller apps and internal tools. I wouldn’t choose it for a brand-new major API when FastAPI is available, but I’ve never felt like it was “wrong.”

## Data work + files

**pandas**  
The cliché answer, unfortunately, is popular for a reason. For cleaning CSV/Excel, reshaping data, joining tables, and doing the daily grind of analysis, pandas is still the tool that most people can jump into immediately.

**polars**  
If performance matters and you don’t mind learning slightly different patterns, Polars can be a big win. I’ve seen it turn “this is taking forever” into “okay, I can iterate,” mostly because it’s built to move data efficiently.

**pyarrow**  
When Parquet and Arrow/columnar workflows are the center of your world, pyarrow helps you keep things fast and predictable. It’s also a nice complement to pandas-style work when you want more control over the data path.

## Machine learning

**scikit-learn**  
For classic ML—classification, regression, clustering, preprocessing, evaluation—scikit-learn is still the backbone. The API is consistent enough that experiments don’t collapse into chaos.

**PyTorch**  
If you’re training deep learning models and you want control, PyTorch is my default. It’s also simply the common language people expect in teams and research, which matters more than people admit.

**XGBoost** (and similar)  
For tabular problems where you want strong results without going full “deep learning engineering,” XGBoost is hard to beat. CatBoost is also worth a look if your data has patterns that it handles really well. This is often where you get “good enough and fast” without regrets.

## LLM tooling + NLP-ish stuff

**Hugging Face Transformers**  
If you’re running or fine-tuning modern models, this is the default starting point. Huge model catalog, lots of examples, and generally solid ergonomics for experimentation.

**Hugging Face Datasets**  
Useful when your bottleneck is loading data reliably instead of scraping and stitching files by hand. It saves time and keeps dataset handling less brittle.

**spaCy**  
For production-style NLP pipelines (tokenization, tagging, extraction), spaCy is still a practical pick. It’s fast, and it doesn’t try to be a generative everything machine.

## Async + HTTP + networking

**requests**  
I’m not going to pretend async is always the right call. For normal HTTP client work, requests is still the most comfortable choice: simple, readable, and widely understood.

**httpx**  
If you want async support or you like a cleaner API surface for HTTP clients, httpx is a strong alternative. It’s become a go-to for teams that want fewer surprises.

**aiohttp**  
Powerful, but it’s more “you’re building an async networking system” than “you just need an HTTP call.” I’d reach for it when you genuinely need the lower-level async control.

## Testing + code quality

**pytest**  
Still the king for testing in Python. Fixtures, plugins, parametrization—you can build a test setup that grows with the project instead of fighting it.

**ruff**  
My favorite kind of tooling: fast feedback. If you adopt Ruff, you stop arguing about formatting and many style issues quietly disappear, which keeps PR reviews focused on actual behavior.

**mypy**  
Optional, but useful if you like catching whole classes of errors before runtime. It won’t save you from bad logic, but it can stop a bunch of “why is this None?” pain early.

## Dev tooling (the stuff you touch every day)

**Poetry**  
Dependency management that feels sane. It’s not perfect for every workflow, but it’s a big upgrade from the older “works on my machine” setups that are still haunting many repos.

**Click**  
If you build CLIs, Click is reliable and clean. Argument parsing stays readable, and your command code doesn’t turn into a boilerplate monster.

**Typer**  
For command-line apps where you want the code to feel more like Python functions than CLI plumbing, Typer is excellent. It’s friendly, fast to build with, and the help text is usually great out of the gate.
