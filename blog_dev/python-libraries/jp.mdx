「ベストなPythonライブラリ」は、とりあえずリストをドキュメントに貼り付けたようなものだと思っていました。

でも、実際に自分の週末プロジェクトがコケた経験があって考えが変わりました。  
“人気だからそれにしよう” って選んだのに、作業の性質にまったく合ってなくて、予定がぐちゃぐちゃになったんです。

なので、ここでは僕なりの答えを出します。  
あいまいな羅列じゃなくて、2026年に向けて本気で賭けるならこのあたり、というライブラリたち。用途ごとにまとめつつ、基準もちょっとだけ厳しめに見ています――「動く」と「出荷してもちゃんと“うまく動く”」は別物だから。

## Web開発 + API

**FastAPI**  
スピードと、ちゃんとした使いやすさを両立したいときは、結局これがいちばん。リクエスト/レスポンスのモデリングができて、バリデーションも“罰ゲーム感”がなく、さらに小さなサービスから本格的な規模まで育てる道筋も用意されています。

**Django**  
“ただのAPI”以上を作るなら。つまり、管理画面・認証・マイグレーションまで含めた、いわゆるWebアプリ一式を組むなら、Djangoがちゃんと仕事をします。最初はちょっと窮屈に感じるくらいには思想が強いんですが、プロジェクトが育ってくると、逆にそれがすごく安心感になるタイプです。

**Flask**  
軽量で柔軟、そして小さめのアプリや社内ツールにもまだまだ強い。FastAPIがある今、まったく新しいメジャーAPIでFlaskを選ぶか？と聞かれたら僕は迷うと思います。  
でも、だからといって「間違いだ」と感じたことは一度もありません。

## データ作業 + ファイル

**pandas**  
残念だけど、結論としては“定番だから人気”なんですよね。CSV/Excelの掃除、データの形を組み替える作業、テーブルの結合、そして分析の毎日の地味な作業まで――pandasは、今でも「とりあえず始められる」人が一番多い道具です。

**polars**  
性能が大事で、ちょっとだけ違うパターンを学ぶ気があるなら、Polarsは大きな勝ちになり得ます。  
僕の経験だと「これ、永遠に終わらないのでは？」が「よし、反復できる」になることが多い。理由は単純で、データを効率よく動かすように作られているからです。

**pyarrow**  
ParquetやArrow/カラムナワークフローが中心の世界なら、pyarrowは速さと予測可能性を保つのに役立ちます。さらに、データの流れをもう少しコントロールしたいときは、pandas的な作業の良い相棒にもなってくれます。

## 機械学習

**scikit-learn**  
古典的なML――分類・回帰・クラスタリング、前処理、評価。scikit-learnはいまだに土台です。APIがかなり一貫してるので、実験がカオスに崩れていきにくいのも大きい。

**PyTorch**  
深層学習モデルを学習させて、きちんと制御したいなら僕のデフォルトはPyTorch。  
それに、チームや研究の文脈で“期待される共通言語”でもあります。人が思っている以上に、ここって重要です。

**XGBoost**（や類似のもの）  
表形式の問題で、深層学習エンジニアリング全振りせずに強い結果を狙いたいなら、XGBoostはなかなか手放せません。データにうまくハマるパターンがあるならCatBoostも見る価値あり。たいていは「十分良くて速い」まで持っていけて、後悔が少ないゾーンです。

## LLMツール + NLPっぽいもの

**Hugging Face Transformers**  
最新のモデルを実行したり微調整したりするなら、最初のスタート地点としてはここがデフォルト。巨大なモデルカタログ、たくさんの例、そして実験しやすい“手触り”がだいたいしっかりしています。

**Hugging Face Datasets**  
ボトルネックが「スクレイピングしてファイルを手でつなぎ合わせる」ことじゃなく、データを確実に読み込む方にあるなら便利です。時間を節約できるし、データセットの扱いが壊れにくくなります。

**spaCy**  
本番寄りのNLPパイプライン（トークナイズ、タグ付け、抽出）なら、spaCyはまだ実用的な選択肢です。速いし、何でもかんでも“生成モデル万能マシン”にしようとしてこないのが良い。

## Async + HTTP + ネットワーク

**requests**  
asyncが常に正しいとは、もちろん言いません。普通のHTTPクライアント業務なら、requestsは今でもいちばん安心できます。シンプルで読みやすくて、広く理解されている。

**httpx**  
asyncサポートが欲しい、あるいはHTTPクライアントのAPI面をスッキリさせたいなら、httpxは有力な代替です。少ない驚きで済ませたいチームの定番になりつつあります。

**aiohttp**  
強力だけど、これは「HTTP呼び出しをするだけ」ではなくて、「asyncなネットワークシステムを作っていく」側の道具です。低レベル寄りのasync制御が本当に必要なときに選ぶのがいいと思います。

## テスト + コード品質

**pytest**  
Pythonのテスト界の王様はまだpytest。フィクスチャ、プラグイン、パラメータ化――プロジェクトの成長に合わせてテスト環境を育てていけます。戦いになりにくい。

**ruff**  
僕が好きなタイプのツールは「速いフィードバック」です。Ruffを導入すると、フォーマットや多くのスタイル問題についての揉め事が静かに消えていきます。その結果、PRレビューは“実際の挙動”に集中できます。

**mypy**  
任意だけど、実行前にエラーの一群を潰したいなら便利。ロジックのまずさから守ってくれるわけじゃないですが、「なんでNoneなんだ？」系の痛みを早めに止めてくれることは多いです。

## 開発ツール（毎日触るやつ）

**Poetry**  
依存関係の管理が“だいたい健全”。すべてのワークフローに完璧、というわけではないけど、“いまだに多くのリポジトリで怨霊みたいに残っている”古い「環境によって動く（My machineで動く）」系セットアップからのアップグレードとしてはかなり大きい。

**Click**  
CLIを作るならClickは堅実でキレイ。引数パースは読みやすく保たれるし、コマンドコードがボイラープレートの怪物に変質しにくいです。

**Typer**  
CLI配下の“配管”よりも、コードの感じをもっとPythonの関数っぽくしたいコマンドラインアプリ向け。Typerはかなり優秀です。フレンドリーで作るのも速いし、ヘルプ文もだいたい最初から良い出来。
