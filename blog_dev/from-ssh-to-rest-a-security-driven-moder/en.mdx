# From SSH to REST: A Security-Driven Modernization of Slack's EMR Data Pipelines

## Introduction

By 2024, Slack's data platform had accumulated over 700 SSH-based operators orchestrating critical data pipelines. These jobs required direct SSH access to production AWS Elastic MapReduce (EMR) clusters, resulting in a massive security surface. To address this issue, the team decided to migrate all 700+ jobs to a REST-based architecture, eliminating SSH entirely. This post shares the story of how they achieved this feat with zero downtime.

## The Old Way: Simple but Problematic

Slack's data platform was built around 2017 using a straightforward pattern. Airflow, their data pipeline orchestrator, needed to run jobs on EMR clusters, and SSH was the most direct path. The team used SSHOperator to connect to the EMR master node, execute a command, and complete the task. This pattern proliferated across the platform, with teams building custom SSH-based operators for different use cases.

### Example Code: SSHOperator

```python
task = SSHOperator(
    task_id='run_spark_job',
    ssh_conn_id='emr_master',
    command='spark-submit /path/to/job.py',
)
```

## The Real Cost of SSH

The team faced several potential security risks and operational challenges with SSH:

* Direct SSH access to compute clusters increases the potential attack surface.
* Key distribution and rotation across orchestration workers adds operational overhead.
* Achieving fine-grained audit granularity requires correlating logs across multiple systems.
* Permission management can grow complex, often requiring dedicated security groups and custom configurations.
* Jobs ran directly on EMR master nodes instead of being distributed, causing resource contention.
* When Kubernetes pods restarted, SSH connections broke and jobs failed.
* Long-running jobs became "zombies" that kept executing after their connections terminated.
* No reliable way to determine if a job succeeded or failed when connections dropped.

## Understanding the Foundation: REST-Based Job Submission

Before diving into the solution, it's essential to understand what REST-based job submission means and why it matters. When you SSH into a machine and run a command, you create a direct, stateful connection. If that connection drops, the command might keep running, fail, or leave orphaned processes hanging around. You have no reliable way to reconnect and check status.

### The REST Alternative

Modern compute engines (YARN, Trino, Snowflake) expose HTTP APIs for job submission. Instead of maintaining a connection, you:

1. POST a job request → receive a job ID
2. GET job status using the ID → check if it's running, completed, or failed
3. DELETE the job → cleanly cancel, if needed

The job lifecycle is managed server-side. Your client can crash and restart, and the job keeps running while you can still query its status.

## The YARN Piece

For Hadoop workloads (MapReduce, Spark, Hive), YARN is the resource manager with a REST API for job submission. However, YARN's API is designed for Hadoop jobs. What about the 300+ CLI-based jobs running arbitrary shell commands like `aws s3 sync` or `hadoop distcp`? That's where YARN Distributed Shell comes in.

### The Breakthrough: YARN Distributed Shell

YARN Distributed Shell is a little-known feature that allows any shell script to run in a proper YARN container with resource allocation and lifecycle management. It was the key breakthrough that made the migration possible.

## The Solution: Quarry

Quarry is Slack's REST-based job submission gateway. It sits between various services and compute engines (Airflow being the biggest user), handling:

* Authentication: Service-to-service tokens instead of SSH keys
* Job submission: REST APIs to YARN, Trino, and Snowflake
* State tracking: Server-side monitoring of job status
* Lifecycle management: Clean cancellation and cleanup through REST APIs
* Observability: Structured logs, metrics, and tracing for all job submissions

### The Architecture Shift

Before:

Airflow → SSH Connection → EMR Master Node → Execute Command

After:

Airflow → Quarry REST API → YARN ResourceManager → EMR Container

## The Quarry Advantage

With YARN Distributed Shell support, Quarry became the universal job submission gateway. Whether you're running a Spark job, a Hive query, or a simple shell script, it all goes through the same REST API. No SSH credentials. No direct cluster access. Just REST API calls with proper authentication and server-side job tracking.

## The Migration Journey

The team knew this wasn't going to be a quick fix. They had 700+ production jobs across 8 independent data regions, each with unique network configurations and data sovereignty requirements. Critical workloads, like search indexing, couldn't tolerate any downtime.

### The Approach: Incremental and Phased

1. Phase 1 – Proof of Concept: Started with pilot jobs to validate the Quarry-based approach.
2. Phase 2 – Security Review: Engaged security teams to plan credential elimination and ensure the REST-based approach met security requirements.
3. Phase 3 – OKR-Driven Execution: Made it a Key Result with executive visibility.
4. Phase 4 – Bulk Migration: Heavy cross-team coordination to migrate remaining workloads across all regions.
5. Phase 5 – Final Cleanup: Completed overlooked DAGs and deprecated all legacy SSH-based operators.

### Migration by the Numbers

* 700+ jobs migrated across 7 operator types
* 8 independent data regions with coordinated rollouts
* 5 teams transitioned to new operators
* Zero downtime for business-critical services
* Completed in 3 quarters, from initial pilot to 100% SSH elimination

## The Challenges

No migration this size goes smoothly. The team encountered two significant challenges:

1. Virtual Memory Check Failures: Jobs that'd been running fine via SSH were now failing with vmem check errors. The root cause was that SSH commands ran directly on the master node, bypassing YARN's resource enforcement entirely. Quarry submits jobs properly to YARN, which enforces resource limits.
2. Network Segregation and EKM Connectivity: During migration of dev search infrastructure, the team encountered issues with network segregation and EKM connectivity. They worked with the security team to resolve these issues and ensure a smooth migration.

By migrating from SSH to a REST-based architecture, Slack's data platform team improved security, reliability, and scalability. They eliminated SSH entirely, reducing the attack surface and operational overhead. The team's experience demonstrates the importance of a well-planned and incremental approach to large-scale migrations.
