<Callout type="danger" title="Common Misconception">
  Many tutorials and even experienced developers oversimplify this. The distinction between *reassignment* and *mutation* is the key to understanding JavaScript behavior at a deeper level—getting it wrong leads to subtle bugs in state management.
</Callout>

`const` in JavaScript does **not** freeze your values. It locks the *variable binding*—meaning you can't point the variable at an entirely different value. This is one of the most misunderstood concepts in JavaScript.

---

## The Binding vs. The Value

<Stepper>
  <Step title="Reassignment (Blocked)">
    The binding itself is constant. You cannot point `x` somewhere else:
    
    ```js
    const x = 5;
    x = 10;  // TypeError: Assignment to constant variable.
    
    const user = { name: "Ava" };
    user = {};  // TypeError: Assignment to constant variable.
    ```
  </Step>

  <Step title="Mutation (Allowed)">
    You didn't change the binding—you changed the *object/value it points to*. JavaScript allows this:
    
    ```js
    const user = { name: "Ava" };
    user.name = "Morgan";  // ✅ Allowed — same object, different property
    
    const items = [1, 2, 3];
    items.push(4);         // ✅ Allowed — same array reference
    items[0] = 99;         // ✅ Allowed — same array reference
    ```
  </Step>

  <Step title="The Memory Model">
    Think of `const` as a "permanent pointer" rather than an "immutable object":
    
    ```js
    const obj = { a: 1, nested: { b: 2 } };
    
    // The pointer (variable name) is locked to this object
    // But the object itself is still open for business
    
    obj.a = 100;           // ✅ Fine
    obj.nested.b = 200;    // ✅ Fine
    obj.newProp = "new";   // ✅ Fine
    
    // Only the final line fails — reassigning the variable itself
    obj = {};              // ❌ TypeError
    ```
  </Step>
</Stepper>

## Why JavaScript Designed It This Way

JavaScript uses a `const` keyword that signals intent rather than enforcing immutability. This choice relates to JavaScript's history as a lightweight scripting language, but also has advantages in everyday development.

<Tabs defaultTab="Reasons">
  <Tab label="Reasons" icon={<span>🤔</span>}>
    <ol className="space-y-3 text-sm">
      <li>
        <strong>Performance:</strong> The JavaScript engine knows that the binding won't change, enabling certain optimizations.
      </li>
      <li>
        <strong>Intent signaling:</strong> `const` tells other developers: "This variable should not be reassigned."
      </li>
      <li>
        <strong>Flexibility:</strong> Some objects need mutation (adding properties, pushing array elements) but shouldn't be replaced.
      </li>
      <li>
        <strong>Backward compatibility:</strong> JavaScript was designed with a minimalist approach, and adding true deep immutability from the start wasn't feasible.
      </li>
    </ol>
  </Tab>

  <Tab label="Use Cases" icon={<span>🎯</span>}>
    <ul className="space-y-2 text-sm">
      <li><strong>Loop variables:</strong> `const i` in a for-loop—the loop variable itself shouldn't be reassigned.</li>
      <li><strong>Arrow functions:</strong> `const fetch = async () => { ... }`—the function reference is fixed.</li>
      <li><strong>Imports:</strong> `import React from 'react'`—the imported binding is fixed per module scope.</li>
      <li><strong>React components:</strong> `const App = () => { ... }`—classic pattern for function components.</li>
    </ul>
  </Tab>

## Real Immutability Options

<Callout type="info" title="Quick Reference">
  If you truly need immutability, JavaScript provides several tools—each with different levels of depth and performance implications.
</Callout>

<Accordion type="single">
  <AccordionItem title="Option 1: Object.freeze() — Shallow Freeze">
    
    ```javascript filename="freeze_demo.js" showLineNumbers
const shallow = Object.freeze({
    name: "Ava",
    settings: {
        notifications: true
    }
});

shallow.name = "Morgan";               // ❌ TypeError
shallow.newProp = "test";              // ❌ TypeError

// ❗ Watch out: nested objects are NOT frozen
shallow.settings.notifications = false; // ✅ Works! Shallow only
shallow.settings = { notifications: false }; // ❌ TypeError (top-level only)

// Verify:
Object.isFrozen(shallow);             // true
Object.isFrozen(shallow.settings);    // false — still mutable!
    ```
    
    <Alert type="warning" title="Shallow behavior">
      `Object.freeze` does *not* recursively freeze nested objects. You must manually freeze every nested object for a truly immutable structure—which is rarely worth it.
    </Alert>
    
  </AccordionItem>

  <AccordionItem title="Option 2: StructuredClone — Deep Copy">
    
    ```javascript filename="structured_clone_demo.js" showLineNumbers
const original = {
    name: "Ava",
    settings: {
        notifications: true,
        theme: "dark"
    },
    tags: ["dev", "tech"]
};

// Creates a true deep copy (no shared references)
const copy = structuredClone(original);

// ✅ Mutating the copy has zero effect on original
copy.name = "Morgan";
copy.settings.notifications = false;
copy.tags.push("blog");

console.log(original.name);             // "Ava"
console.log(original.settings.notifications); // true
console.log(original.tags);             // ["dev", "tech"]

console.log(copy.name);                 // "Morgan"
console.log(copy.tags);                 // ["dev", "tech", "blog"]
    ```
    
    <SuccessAlert title="Why structuredClone?">
      It handles ```Date```, ```RegExp```, ```Map```, ```Set```, ArrayBuffers, and even circular references. It's the modern replacement for <code>JSON.parse(JSON.stringify())</code>, preserving types that JSON loses.
    </SuccessAlert>
    
  </AccordionItem>

  <AccordionItem title="Option 3: Object.preventExtensions / seal / freeze">
    
    <Table>
      <TableHeader>

        <TableCell>Can add props?</TableCell>
        <TableCell>Can delete props?</TableCell>

      <TableBody>
        <TableRow>

          <TableCell>❌</TableCell>
        </TableRow>
        <TableRow>
          <TableCell><code>Object.seal()</code></TableCell>
          <TableCell>❌</TableCell>
          <TableCell>❌</TableCell>
          <TableCell>✅</TableCell>
          <TableCell>❌</TableCell>
        </TableRow>
        <TableRow>
          <TableCell><code>Object.freeze()</code></TableCell>
          <TableCell>❌</TableCell>
          <TableCell>❌</TableCell>
          <TableCell>❌</TableCell>
          <TableCell>❌</TableCell>
        </TableRow>
      </TableBody>
    </Table>
    
    <NoteAlert title="Practical advice">
      These methods are mostly useful for defensive coding or when you want to explicitly lock down an API contract. In application logic, immutable patterns (spreading, structuredClone) are usually clearer and more flexible.
    </NoteAlert>
    
  </AccordionItem>

  <AccordionItem title="Option 4: Immutable libraries (Immer) — The Modern Approach">
    
    <InfoAlert title="You want immutable-looking code that's actually mutable under the hood">
      
      <code>Immer</code> takes a unique approach: you write code that <em>looks</em> mutable, and it produces a new immutable state automatically.
      
      ```javascript filename="with_immer.js" showLineNumbers
import { produce } from "immer";

const nextState = produce(currentState, (draft) => {
    // "mutate" the draft — feels natural
    draft.items.push({ id: 3, name: "Pencil" });
    draft.items[0].quantity = 5;
});

// nextState is a brand new object
// currentState is unchanged
// But the code reads like you're modifying it
      ```
      
      This pattern powers Redux Toolkit's `createSlice` and many modern state management systems.
      
    </InfoAlert>
    
  </AccordionItem>
</Accordion>

## Common Questions

<Accordion type="single">
  <AccordionItem title="But if const doesn't make it immutable, what even is the point?">
    
    `const` prevents *accidental reassignment*—the most common bug. It tells humans and really smart linters that this binding should stay the same. For true read-only guarantees, combine it with other patterns.
    
  </AccordionItem>

  <AccordionItem title="Can I use const with forEach or map?">
    
    Yes. And you should! In loops, `const item` creates a *new binding per iteration*, so it's safe and actually preferred:
    
    ```js
    const items = [1, 2, 3];
    
    items.forEach((item) => {        // ✅ item is per-iteration
        console.log(item);
    });
    
    items.map((item) => item * 2);   // ✅ item is per-iteration
    ```
    
  </AccordionItem>

  <AccordionItem title="What about const in loops (let vs const)?">
    
    ```js
    const funcs = [];
    
    for (const i = 0; i < 5; i++) {
        funcs.push(() => console.log(i));
    }
    funcs[0]();  // 0 — all log their specific i value ✓
    
    // But this works DIFFERENTLY:
    for (let j = 0; j < 5; j++) {
        funcs.push(() => console.log(j));
    }
    funcs[0]();  // 0 — same result here
    ```
    
    The distinction only matters in `var`-style function-scoped hoisting scenarios.
    
  </AccordionItem>
</Accordion>

## TL;DR

<Callout type="idea" title="One-sentence takeaway">
  `const` means "this variable won't be reassigned"—not "this value is immutable." Use it every day. For immutability, pair it with spread syntax, structuredClone, or a library like Immer.
</Callout>
