<Callout type="danger" title="The Half-Truth">
  "const doesn't mean immutable because you can change an object." This statement is half-right, and it's exactly why developers get misled!
</Callout>

`const` in JavaScript does **not** freeze your values. It freezes your **variable binding**—meaning you can't reassign the variable to a different value entirely.

<Stepper>
  <Step title="Reassignment (Fails)">
    ```js
    const x = 5;
    x = 10; // ❌ TypeError
    ```
    This fails because `x` is a constant binding. You can't point it somewhere else.
  </Step>
  <Step title="Mutation (Succeeds)">
    ```js
    const user = { name: "Ava" };
    user.name = "Morgan"; // ✅ Works!
    ```
    You didn't reassign `user`; you just mutated the object it points to.
  </Step>
</Stepper>

### Arrays Behave The Same Way

```js
const items = [1, 2, 3];
items.push(4); // ✅ Works!
items = [];    // ❌ TypeError
```

<Callout type="info" title="The Right Mental Model">
  The **binding** stays fixed, but the **value** may still be mutable. JavaScript cares about language semantics, not raw memory addresses!
</Callout>

### Real Immutability

If you actually want to prevent anyone from changing an object, `const` isn't enough.

<Callout type="idea" title="Use Object.freeze()">
  ```js
  const user = Object.freeze({ name: "Ava" });
  user.name = "Morgan"; // ❌ TypeError in strict mode
  ```
  Note: `Object.freeze` is shallow! Nested objects will still be mutable unless you freeze them too.
</Callout>

The mistake isn't claiming `const` isn't immutable. The mistake is assuming `const` is shorthand for deep immutability. Keep your bindings locked, but remember your objects are still open for business!
