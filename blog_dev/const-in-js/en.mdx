I keep seeing the same confusion in JavaScript: “`const` doesn’t mean immutable because you can change an object.” That line is usually half-right, and it’s exactly why people get misled.

`const` in JavaScript doesn’t freeze your values. It freezes your **variable binding**—meaning you can’t reassign the variable to a different value.

```js
const x = 5;
x = 10; // ❌ TypeError: Assignment to constant variable.
```

That fails because `x` is a constant binding. You’re trying to point `x` somewhere else.

Now compare that to objects:

```js
const user = { name: "Ava" };
user.name = "Morgan"; // ✅ works
```

This works because you didn’t reassign `user`. You mutated the object that `user` already references.

So the distinction is simple but easy to blur:

- `const` stops reassignment (the reference/binding can’t change)
- it does **not** automatically stop mutation inside objects and arrays

### Arrays behave the same way
```js
const items = [1, 2, 3];
items.push(4); // ✅ works
items = [];    // ❌ TypeError
```

Again, `items` isn’t being reassigned. The array object itself is.

### “But the memory address doesn’t change…”
That’s the right intuition, but phrasing it that way can send people down the wrong rabbit hole. JavaScript doesn’t require you to think in raw memory addresses, and the spec talks in terms of language semantics, not how bytes sit in RAM.

A better mental model is: **the binding stays fixed; the value may still be mutable.** With objects, the binding can stay the same while the contents change.

### If you really want immutability
If your goal is “nobody should be able to change the object,” then `const` isn’t enough. You’ll need to enforce immutability yourself—commonly with `Object.freeze`.

```js
const user = Object.freeze({ name: "Ava" });
user.name = "Morgan"; // ❌ TypeError (in strict mode) or fails silently otherwise
```

Just be aware: `Object.freeze` is shallow. Nested objects won’t be frozen unless you freeze them too.

### The actual mistake
The mistake isn’t claiming `const` isn’t immutable. The mistake is assuming `const` is shorthand for **deep immutability**.

`const` makes reassignment illegal. It doesn’t automatically make the underlying object unchangeable. You can mutate an object referenced by a `const` variable all day long, and JavaScript will happily let you—as long as you don’t reassign the variable itself.
