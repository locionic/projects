最近ずっと、JavaScriptで同じ混乱を見かけます。「“constは不変じゃない。オブジェクトは変えられるから”」──この言い方は、たいてい半分正しくて、だからこそ人を誤解させやすいんです。

JavaScriptの `const` は、値をフリーズしません。フリーズするのは **変数の束縛（variable binding）** です。つまり、変数を別の値に“置き換える”再代入ができない、ということ。

```js
const x = 5;
x = 10; // ❌ TypeError: Assignment to constant variable.
```

これが失敗するのは、`x` が定数の束縛だからです。`x` を別の場所（別の値）へ向けようとしているんですね。

じゃあオブジェクトはどうかというと：

```js
const user = { name: "Ava" };
user.name = "Morgan"; // ✅ 動きます
```

これは `user` を再代入していないから動きます。`user` が参照している、すでにあるオブジェクトそのものを変えているだけです。

つまり、違いはシンプル。でも、そこがつい曖昧になる。

- `const` は再代入を止める（参照／束縛は変えられない）
- しかし、オブジェクトや配列の中身のミューテーションまで自動で止めるわけではない

### 配列も同じ
```js
const items = [1, 2, 3];
items.push(4); // ✅ 動きます
items = [];    // ❌ TypeError
```

これも同様で、`items` は再代入されていません。変わっているのは、配列オブジェクトそのものです。

### 「でもメモリアドレスは変わらない…」
その直感は合ってます。でも、その言い方のままだと、よくわからない方向の“ウサギ穴”に人を連れていきがちです。JavaScriptは、生のメモリアドレスを意識する必要はありませんし、仕様も「RAM上のバイトがどう配置されるか」という話ではなく、言語としての意味論で語っています。

より良い頭の使い方はこれです：**束縛は固定のまま、値は（中身としては）まだミューテブルになり得る。**  
オブジェクトなら、「束縛は同じまま、中身だけが変わる」が成立します。

### 本当に不変にしたいなら
目的が「誰にもオブジェクトを変えさせたくない」なら、`const` だけでは足りません。自分で不変性を強制する必要があります。よく使われるのは `Object.freeze` です。

```js
const user = Object.freeze({ name: "Ava" });
user.name = "Morgan"; // ❌ strictモードならTypeError（それ以外は黙って失敗することもあります）
```

ただし注意点があります。`Object.freeze` は浅い（shallow）です。ネストしたオブジェクトは、同じようにfreezeしない限り凍結されません。

### 実際の間違い
間違いは、「constは不変じゃない」と言うことではありません。問題なのは、`const` を **深い不変性（deep immutability）** の省略形だと決めつけてしまうことです。

`const` は再代入を禁止します。でも、参照されている“元のオブジェクト”を勝手に変更不能にしてくれるわけではありません。`const` で参照されているオブジェクトなら、日が暮れるまで自由にミューテーションできます。もちろん、変数本体を再代入しなければ、という条件つきで。
