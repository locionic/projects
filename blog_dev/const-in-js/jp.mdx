<Callout type="warning" title="よくある半分の正解">
  「constは不変じゃない。オブジェクトは変えられるから」──この言い方は半分正しくて、だからこそ人を誤解させやすいんです。
</Callout>

JavaScriptの `const` は、値をフリーズしません。フリーズするのは **変数の束縛（variable binding）** です。つまり、変数を別の値に“置き換える”再代入ができないということです。

<Stepper>
  <Step title="再代入（失敗する）">
    ```js
    const x = 5;
    x = 10; // ❌ TypeError
    ```
    これは `x` が定数の束縛だからです。
  </Step>
  <Step title="ミューテーション（成功する）">
    ```js
    const user = { name: "Ava" };
    user.name = "Morgan"; // ✅ 動きます
    ```
    これは `user` を再代入していないからです。すでに参照しているオブジェクトそのものを変えているだけです。
  </Step>
</Stepper>

### 配列も同じ

```js
const items = [1, 2, 3];
items.push(4); // ✅ 動きます
items = [];    // ❌ TypeError
```

<Callout type="info" title="より良い考え方">
  「メモリアドレスが変わらないから…」と考えるより、**「束縛は固定のまま、値はミューテブルになり得る」**と考えましょう。オブジェクトなら、「束縛は同じまま、中身だけが変わる」が成立します。
</Callout>

### 本当に不変にしたいなら

誰もオブジェクトを変えられないようにしたいなら、`const` だけでは足りません。

<Callout type="idea" title="Object.freeze() の出番">
  ```js
  const user = Object.freeze({ name: "Ava" });
  user.name = "Morgan"; // ❌ TypeError
  ```
  ※ただし `Object.freeze` は浅い（shallow）ので、ネストされたオブジェクトには効きません！
</Callout>

間違いは、「constは不変じゃない」と言うことではありません。`const` が**深い不変性（deep immutability）**をもたらすと勘違いすることです。束縛はロックしつつ、オブジェクトは変更可能なままであることを覚えておきましょう！
