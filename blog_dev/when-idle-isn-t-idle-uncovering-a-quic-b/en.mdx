# When "idle" isn't idle: Uncovering a QUIC Bug in CUBIC Congestion Control

CUBIC, the default congestion controller in Linux, plays a crucial role in governing how most TCP and QUIC connections on the public Internet probe for available bandwidth, back off when they detect loss, and recover afterward. At Cloudflare, our open-source implementation of QUIC, quiche, uses CUBIC as its default congestion controller. In this post, we'll delve into a bug that caused CUBIC's congestion window (cwnd) to get permanently pinned at its minimum and never recover from a congestion collapse event.

## CUBIC's Logic in a Nutshell

Before we dive into the core problem, let's quickly review Congestion Control Algorithms (CCAs). The central knob a CCA turns is the congestion window (cwnd): the sender-side cap on how many bytes can be in flight (sent but not yet acknowledged) at any moment. A larger cwnd lets the sender push more data per round trip; a smaller cwnd throttles it. Every loss-based CCA, CUBIC included, aims to maximize data transfer by inferring the "available bandwidth" of the network.

## The Symptom: A Test that Fails 61% of the Time

Our investigation started with the report of unexpected failures in our ingress proxy integration test pipeline. The test setup involved a Quiche HTTP/3 client and server running at locally (localhost) RTT = 10ms, with a 10 MB file download over HTTP/3 using CUBIC congestion control and 30% random packet loss injected during the first two seconds. After two seconds, loss stops entirely. The test has a generous 10-second timeout to complete the download, which is expected to be completed in four or five seconds.

## The Anomaly: 999 State Transitions with Zero Loss

We instrumented quiche's qlog output with packet loss events and built visualizations to understand what was happening inside the congestion controller. After the two-second mark, packet loss stops entirely, but the number of bytes in flight remains flat, contradicting the core logic of the CUBIC algorithm. Our analysis shows that CUBIC enters a rapid oscillation between congestion avoidance state and recovery state — 999 transitions in approximately 6.7 seconds.

## Tracing the Root Cause

The bug's root cause lies in an optimization introduced in the Linux kernel's CUBIC implementation in 2017. The optimization aimed to adjust the congestion recovery start time to account for the gap in sending when the application resumes sending after an idle period. However, this optimization was not properly ported to quiche, leading to a bug that causes the congestion window to get permanently pinned at its minimum.

## The Fix

The fix involves modifying the `on_packet_sent` function in quiche's CUBIC implementation to correctly handle the idle period adjustment. Specifically, the fix involves checking if the sending burst is restarting and adjusting the congestion recovery start time accordingly.

## Conclusion

In this post, we've walked through a bug in the Linux kernel's CUBIC congestion control algorithm that caused QUIC connections to fail to recover after a congestion collapse event. We've also discussed the root cause of the bug and the fix. This experience highlights the importance of careful testing and validation of congestion control algorithms, especially when porting them to new environments.

```rust
// cubic.rs — on_packet_sent() (simplified)
/// Updates the state when a packet is sent.
fn on_packet_sent(&mut self, bytes_in_flight: usize, now: Instant, ...) {
    // If the sending burst is restarting (i.e., bytes_in_flight was zero before this send),
    // adjust the congestion recovery start time to account for the gap in sending.
    if bytes_in_flight == 0 {
        let delta = now - self.last_sent_time;
        self.congestion_recovery_start_time += delta;
    }
    // Record the time of this send event.
    self.last_sent_time = now;
}
```
