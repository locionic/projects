# Next.js Hydration Errors: Why They Happen and How to Fix Them

I still remember the first time I saw a giant red hydration error in my browser console. The message was cryptic, the stack trace pointed to a minified React file, and my UI looked completely broken for a split second before snapping into place. 

I thought my database was down. It wasn't. I had just tried to render a dynamic timezone string directly in my component.

If you're building with Next.js (or any SSR React framework), you will run into hydration errors. Let's break down exactly what they mean, why they happen, and the three most common ways to fix them.

---

## What Actually Is "Hydration"?

Before we fix the error, we need to understand the process. Next.js does two things when a user requests a page:

1. **Pre-rendering (Server Side):** Next.js runs your React components on the server, generates raw HTML, and sends it to the browser. The user sees the page instantly, but it isn't interactive yet (buttons don't work, inputs are inert).
2. **Hydration (Client Side):** React runs on the client, downloads the JavaScript bundle, walks the server-rendered HTML structure, and attaches event listeners (like `onClick`) to make it alive.

Here is the golden rule of SSR: **The HTML generated on the server must exactly match the HTML generated on the client's first render.**

If there is even a single character of difference, React panics. It throws a hydration mismatch error, discards the server HTML, and rebuilds the page from scratch. This slows down your site and ruins the performance benefits of SSR.

## Villain #1: Client-Only Globals (`window` and `localStorage`)

The most common mistake is referencing browser APIs directly in your render path. The server has no concept of `window` or `document`.

```tsx
// ❌ THIS WILL CRASH ON THE SERVER
function MyComponent() {
  const width = window.innerWidth; // window is undefined on the server!
  return <div>Width: {width}</div>;
}
```

Even if you wrap it in a typeof check to prevent the crash, you still get a hydration error because the server output will be different from the client output.

<DiffView
  oldCode={`// ❌ Hydration Error: server gets "Server", client gets width
function ResponsiveLayout() {
  const isMobile = typeof window !== 'undefined' && window.innerWidth < 768;
  return <div>{isMobile ? 'Mobile' : 'Desktop'}</div>;
}`}
  newCode={`// ✅ Safe: state is initialized on client mount
import { useState, useEffect } from 'react';

function ResponsiveLayout() {
  const [isMobile, setIsMobile] = useState(false);

  useEffect(() => {
    setIsMobile(window.innerWidth < 768);
  }, []);

  return <div>{isMobile ? 'Mobile' : 'Desktop'}</div>;
}`}
  language="tsx"
  mode="unified"
/>

<Callout type="info" title="The Hook Strategy">
  For browser APIs, always defer the state update to `useEffect`. Since `useEffect` only runs in the browser after mount, it never executes on the server. The initial HTML matches, and the client updates the UI safely afterward.
</Callout>

## Villain #2: Dynamic Data (Dates, Times, and Random Numbers)

If you display the current time or a random number, the server will generate one value, and the client (running milliseconds later) will generate a different one.

```tsx
// ❌ Server: "Rendered at 10:00:00", Client: "Rendered at 10:00:02"
function CurrentTime() {
  return <div>Time: {new Date().toLocaleTimeString()}</div>;
}
```

There are two ways to solve this depending on what you need.

### Solution A: Defer to Client Mount (Recommended)

If the data doesn't need to be there for SEO, wait until the component mounts to show it.

<Stepper>
  <Step title="Initialize Mount State">
    Create a boolean state to track if the component has mounted in the browser.
  </Step>
  <Step title="Trigger in useEffect">
    Flip the boolean to `true` inside `useEffect`.
  </Step>
  <Step title="Conditional Render">
    Show a loading placeholder on the server and the actual date on the client.
  </Step>
</Stepper>

```tsx
import { useState, useEffect } from 'react';

export default function CurrentTime() {
  const [mounted, setMounted] = useState(false);

  useEffect(() => {
    setMounted(true);
  }, []);

  if (!mounted) return <div>Loading time...</div>;

  return <div>Time: {new Date().toLocaleTimeString()}</div>;
}
```

### Solution B: Suppress Warning (For Static Content)

If the discrepancy is harmless (like a static timezone rendering or an intentional mismatch), you can tell React to ignore it using the `suppressHydrationWarning` attribute.

```tsx
// This disables the console error for this specific element
<div suppressHydrationWarning>Time: {new Date().toLocaleTimeString()}</div>
```

<Callout type="warning" title="Warning">
  `suppressHydrationWarning` only works one level deep and does not fix the underlying performance cost of React mismatch rebuilding. Use it sparingly for simple text nodes (like dates or timestamps), not large UI trees.
</Callout>

## Villain #3: Invalid HTML Tag Nesting

This is the sneakiest cause of hydration errors. It has nothing to do with JavaScript state, but rather the way browsers parse invalid HTML tags.

If you nest a `<div>` inside a `<p>` tag, the browser's HTML parser will automatically close the `<p>` tag before the `<div>` is opened. This transforms your intended DOM structure into something completely different.

React expects the structure on the server to match the DOM structure parsed by the browser. When the browser rewrites the tags, React spots the mismatch and errors out.

</p>`}
  newCode={`// ✅ Valid HTML: use divs or spans instead
<div>

</div>`}

Common HTML nesting mistakes that trigger hydration errors:
* Placing `<div>` or `<p>` inside another `<p>`
* Placing `<ul>` or `<ol>` directly inside a `<p>`
* Putting a `<tr>` directly under a `<table>` instead of wrapping it in `<tbody>`
* Putting block elements (`h1`, `form`, `section`) inside inline elements (`a`, `span`)

## How to Debug Mismatches Fast

Next.js has improved its error messages, but they can still be hard to trace. Here is my checklist when debugging a hydration issue:

1. **Check the Browser Console:** Look for the specific diff. Next.js will often output: `Expected server HTML to contain a <div> in <a>`.
2. **Disable Browser Extensions:** Last year, a translator extension was injecting scripts into my page DOM, which triggered a hydration error because the server didn't have those script tags. Test in Incognito Mode.
3. **Use the "Client-Only" Component Trick:** If you have a legacy third-party library that relies heavily on window APIs and you can't rewrite it, import it dynamically with SSR disabled:

```tsx
import dynamic from 'next/dynamic';

const DynamicNoSSR = dynamic(
  () => import('../components/HeavyChart'),
  { ssr: false }
);
```
