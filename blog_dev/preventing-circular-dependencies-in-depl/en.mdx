# Preventing Circular Dependencies in Deployment Scripts with eBPF

At GitHub, we host our own source code on github.com, which means we're our own biggest customer. However, this creates a circular dependency: if github.com goes down, we can't access our own source code to deploy a fix. To mitigate this, we maintain a mirror of our code and built assets for rolling back. But this isn't enough, as deployment scripts can still introduce circular dependencies on internal services or download binaries from GitHub.

## Types of Circular Dependencies

Let's consider a hypothetical scenario where a MySQL outage occurs, causing GitHub to be unable to serve release data from repositories. To resolve the incident, we need to roll out a configuration change to the stateful MySQL nodes that are impacted. This configuration change is applied by executing a deploy script on each node. There are three types of circular dependencies that could impact GitHub during this scenario:

* **Direct dependency**: The MySQL deploy script attempts to pull the latest release of an open-source tool from GitHub, which is unavailable due to the outage.
* **Hidden dependencies**: The MySQL deploy script uses a servicing tool that checks GitHub for updates, which fails or hangs due to the outage.
* **Transient dependencies**: The MySQL deploy script calls an internal service, which in turn attempts to fetch the latest release of an open-source tool from GitHub, causing the failure to propagate back to the deploy script.

## Solving Circular Dependencies with eBPF

To prevent these circular dependencies, we started exploring eBPF, which allows us to load custom programs into the Linux kernel and hook into core system primitives like networking. We were particularly interested in the BPF_PROG_TYPE_CGROUP_SKB program type, which lets us hook network egress from a particular cGroup. A cGroup is a Linux primitive that enforces resource limits and isolation for sets of processes.

We created a proof of concept using the cilium/ebpf library, which simplifies the process of authoring, building, and running eBPF programs. We loaded a pre-compiled eBPF program into the kernel and attached it to a cGroup, which allowed us to selectively monitor and block network calls from the deployment script.

## Conditional Network Filtering with eBPF

We built a per-process conditional network filtering system using eBPF. We created a cGroup and placed the deployment script inside it, then limited the outbound network access of only that script. We used the BPF_PROG_TYPE_CGROUP_SKB program type to hook network egress from the cGroup and intercept DNS queries.

We also created a DNS-based blocked list using eBPF. We used the BPF_PROG_TYPE_CGROUP_SOCK_ADDR program type to hook syscalls to create sockets and change the destination IP. We rewrote any connect4 syscall targeting DNS (Port 53) to localhost:53, which allowed us to intercept DNS queries from the cGroup and forward them to a userspace DNS proxy.

## Correlating Blocked DNS Requests with Processes

We correlated blocked DNS requests back to the specific command or process that triggered them, allowing teams to more easily debug and fix issues. We used the BPF_PROG_TYPE_CGROUP_SKB program type to capture the DNS transaction ID and Process ID (PID) that initiated the request. We placed this information into another eBPF Map tracking DNS Transaction ID -> Process ID.

## Conclusion

We can now conditionally block domains that would cause circular dependencies from deployment scripts, inform the owning team which command triggered the blocked request, provide an audit list of all domains contacted during a deployment, and enforce CPU and memory limits on deploy scripts using cGroups. Our new circular dependency detection process is live after a six-month rollout, and we're continuously improving it to ensure a more stable GitHub platform.

## Getting Started with eBPF

If you're interested in exploring eBPF, start by reviewing the examples in cilium/ebpf and the great documentation on the docs.ebpf.io site. You can also try open-source tools powered by eBPF, like bpftrace for deep tracing or ptcpdump to get TCP dumps with container-level metadata.
