テストが「通ってる」のに、コードは明らかに間違っている――なんでだろうって、長いこと考えちゃいました。結論から言うと、私が思っていたものを実際には実行できていなかっただけです。pytestが、静かにそれを許してくれてたんですね。

というわけで、ちゃんとしたpytestチュートリアルやります。最初期にありがちな落とし穴を潰して、すぐに「出荷の自信」が持てるやつ。

## pytestのインストールとセットアップ（揉めないやつ）

pytestを、プロジェクトが動いてる環境に入れてください。

```bash
pip install pytest
```

以上です。魔法のフラグも、特別な小細工もいりません。

で、プロジェクトのルートから実行：

```bash
pytest
```

pytestは、命名規則に合うファイルを探します（デフォルトだと `test_*.py` と `*_test.py` を探します）。何も実行されないなら、たいてい原因はコードじゃなくてファイル名です。

### 配線が合ってることを確認するミニテスト

`test_math.py` を作ります。

```python
def test_addition():
    assert 1 + 1 == 2
```

実行：

```bash
pytest -q
```

「緑の “1 passed”」が見えたら、準備OKです。

## 基本の構成要素：`test_`関数 + `assert`

pytestの流れはシンプルです。テスト関数を書いて、普通の `assert` を使うだけ。

```python
def test_truth():
    assert True
```

もしアサーションが失敗したら、pytestは役に立つトレースバックを出してくれますし、「ただの assert」だと思ってたよりずっと見やすい比較結果を表示してくれることが多いです。

あと大事な点：新しいテスト用DSLを覚える必要はありません。Pythonのままです。

## 1個だけテストを回す（コードを触ってる最中ならなおさら）

テストファイルが増えてくると、毎回ぜんぶ動かすのはすぐ飽きます。

ファイル1つだけ実行：

```bash
pytest test_api.py
```

特定のテストだけ実行：

```bash
pytest test_api.py::test_user_login
```

こうすると、コードを変えてる最中のフィードバックループが一気に速くなります。

## 役に立つ失敗：ちゃんと比較してくれるやつ

pytestが得意な「助かる失敗」の例です。リストの等価性をアサートするとき。

```python
def test_ordering():
    actual = [3, 1, 2]
    expected = [1, 2, 3]
    assert actual == expected
```

失敗しても、pytestは「壊れました」って叫ぶだけじゃありません。どこが一致しなかったのかを指してくれるので、デバッグの時間が短縮できます。

## fixtures：同じ準備を繰り返さず共有する

fixturesは、セットアップ処理を一箇所にまとめて、複数のテストで使い回せる仕組みです。fixture関数を定義しておくと、pytestがその名前でテスト関数に注入してくれます。

例：

```python
import sqlite3
import pytest

@pytest.fixture
def db():
    conn = sqlite3.connect(":memory:")
    yield conn
    conn.close()
```

使う：

```python
def test_something(db):
    cur = db.cursor()
    cur.execute("select 1")
    assert cur.fetchone() == (1,)
```

この `yield` 形式が重要です。テストが終わった後の後片付けを、ネスト地獄になりがちな `try/finally` の塊にせずに実現できます。

## parametrize：同じテストをコピペしない

テストのロジックは同じで、入力や期待する出力だけが違うなら、`pytest.mark.parametrize` を使いましょう。

```python
import pytest

@pytest.mark.parametrize(
    "a,b,expected",
    [
        (1, 2, 3),
        (2, 2, 4),
        (10, -1, 9),
    ],
)
def test_add(a, b, expected):
    assert a + b == expected
```

こうすると、テストファイルが「重複コードの壁」みたいになるのを避けつつ、エッジケースをしっかり拾いやすくなります。

## 例外を正しくテストする

たいていの場合、例外を手でキャッチする必要はありません。`pytest.raises` を使えばOKです。

```python
import pytest

def divide(a, b):
    return a / b

def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        divide(1, 0)
```

エラーメッセージも一致させたいなら：

```python
with pytest.raises(ZeroDivisionError, match="division by zero"):
    divide(1, 0)
```

## マーカー：テストにタグを付けて、正しいものだけ回す

マーカーは、テストをカテゴリ分けする簡単な方法です。例：ユニット vs 結合 vs 遅いテスト。

```python
import pytest

@pytest.mark.integration
def test_external_api():
    ...
```

結合テストだけ実行：

```bash
pytest -m integration
```

結合テストをスキップ：

```bash
pytest -m "not integration"
```

私はマーカーが好きです。ローカル実行は速く保ちつつ、必要なら重めのチェックもちゃんと用意できるから。

## テストのスキップ：使っていいけど、崇拝はしない

まだ実行できないテストがあるなら、スキップできます。

```python
import pytest

@pytest.mark.skip(reason="Not implemented yet")
def test_future_feature():
    ...
```

動的にスキップすることも可能です。

```python
import sys
import pytest

def test_platform_specific():
    if sys.platform != "win32":
        pytest.skip("Windows-only test")
```

スキップは、たまになら全然アリです。でも戦略にしないこと。

## そのままコピペできるミニ例

**`src/calculator.py`**
```python
def add(a, b):
    return a + b
```

**`tests/test_calculator.py`**
```python
from src.calculator import add
import pytest

@pytest.mark.parametrize(
    "a,b,expected",
    [(1, 2, 3), (10, 5, 15), (-1, 1, 0)],
)
def test_add(a, b, expected):
    assert add(a, b) == expected

def test_add_type_error():
    with pytest.raises(TypeError):
        add("1", 2)
```

そして実行：

```bash
pytest
```
