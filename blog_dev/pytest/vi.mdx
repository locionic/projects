Tôi đã mất quá nhiều thời gian để tự hỏi vì sao test của mình lại “pass” dù code thì chắc chắn sai. Cuối cùng mới vỡ ra: mình không hề chạy đúng những gì mình nghĩ mình đang chạy. Pytest cứ âm thầm để tôi trót lọt.

Vậy nên, thôi thì làm một tutorial pytest “đúng bài” nhé—loại giúp bạn xử lý được các lỗi vặt lúc đầu và có thể ship với sự tự tin nhanh chóng.

## Cài đặt và thiết lập pytest (khỏi cần drama)

Bạn chỉ cần cài pytest trong môi trường nơi dự án của bạn đang nằm:

```bash
pip install pytest
```

Hết. Không có cờ bùa phép nào cả. Không có “special sauce”.

Rồi chạy từ thư mục gốc của dự án:

```bash
pytest
```

Pytest sẽ dò các file theo quy ước đặt tên của nó (mặc định tìm `test_*.py` và `*_test.py`). Nếu không có gì chạy, thường là do tên file—chứ không phải do code của bạn.

### Một test nhỏ để chắc chắn mọi thứ đã vào đúng đường

Tạo `test_math.py`:

```python
def test_addition():
    assert 1 + 1 == 2
```

Chạy:

```bash
pytest -q
```

Nếu bạn thấy dòng xanh kiểu “1 passed” là mọi thứ đã ổn.

## Khối xây dựng cơ bản: hàm `test_` + `assert`

Toàn bộ workflow của pytest cực kỳ đơn giản: bạn viết một hàm test và dùng `assert` thuần túy.

```python
def test_truth():
    assert True
```

Khi một assertion thất bại, pytest sẽ in ra traceback hữu ích—và thường là cách so sánh rõ ràng hơn những gì bạn tưởng tượng từ việc “chỉ assert thôi”.

Và thêm nữa: bạn không cần học thêm một kiểu ngôn ngữ test DSL mới. Python vẫn là Python.

## Chạy riêng một test (vì bạn đang iterate)

Nếu bạn có nhiều file test, chạy tất cả mỗi lần sửa code sẽ rất nhanh… chán đời.

Chạy chỉ một file:

```bash
pytest test_api.py
```

Chạy một test cụ thể:

```bash
pytest test_api.py::test_user_login
```

Cách này là nhanh nhất để rút ngắn vòng phản hồi khi bạn liên tục thay code.

## Những lỗi thực sự giúp được bạn

Đây là kiểu output lỗi mà pytest làm tốt—đặc biệt là khi so sánh equality của list:

```python
def test_ordering():
    actual = [3, 1, 2]
    expected = [1, 2, 3]
    assert actual == expected
```

Khi test fail, pytest không chỉ gào “nó bị hỏng rồi”. Nó chỉ ra phần nào không khớp—mà điều đó tiết kiệm kha khá thời gian khi debug.

## Fixtures: setup dùng chung mà không phải lặp lại

Fixtures cho phép bạn gom logic setup lại và tái sử dụng giữa các test. Bạn định nghĩa một hàm fixture, rồi pytest sẽ “tiêm” nó vào hàm test theo đúng tên tham số.

Ví dụ:

```python
import sqlite3
import pytest

@pytest.fixture
def db():
    conn = sqlite3.connect(":memory:")
    yield conn
    conn.close()
```

Dùng như này:

```python
def test_something(db):
    cur = db.cursor()
    cur.execute("select 1")
    assert cur.fetchone() == (1,)
```

Cú pháp `yield` này quan trọng. Nó giúp bạn làm phần cleanup sau khi test chạy xong mà không biến mọi thứ thành một mớ try/finally lồng ghép rối mắt.

## Parametrized tests: ngừng copy-paste một bài test

Nếu logic test giống nhau, chỉ khác input/kết quả kỳ vọng, hãy dùng `pytest.mark.parametrize`.

```python
import pytest

@pytest.mark.parametrize(
    "a,b,expected",
    [
        (1, 2, 3),
        (2, 2, 4),
        (10, -1, 9),
    ],
)
def test_add(a, b, expected):
    assert a + b == expected
```

Cách này giúp bạn bao phủ case biên mà không cần biến file test thành “bức tường” code bị lặp.

## Kiểm thử exception đúng cách

Đa số thời gian bạn không cần bắt exception thủ công. Dùng `pytest.raises`.

```python
import pytest

def divide(a, b):
    return a / b

def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        divide(1, 0)
```

Nếu bạn muốn match luôn cả message lỗi:

```python
with pytest.raises(ZeroDivisionError, match="division by zero"):
    divide(1, 0)
```

## Markers: gắn nhãn test để chạy đúng nhóm

Markers là cách đơn giản để phân loại test. Ví dụ: unit vs integration vs slow.

```python
import pytest

@pytest.mark.integration
def test_external_api():
    ...
```

Chỉ chạy integration tests:

```bash
pytest -m integration
```

Bỏ qua integration tests:

```bash
pytest -m "not integration"
```

Mình thích markers vì nó giúp chạy local nhanh, trong khi vẫn giữ sẵn các check nặng hơn để bật lên khi cần.

## Skip test (dùng thì dùng, đừng tôn thờ)

Nếu một test chưa chạy được, bạn có thể skip nó.

```python
import pytest

@pytest.mark.skip(reason="Not implemented yet")
def test_future_feature():
    ...
```

Hoặc skip động:

```python
import sys
import pytest

def test_platform_specific():
    if sys.platform != "win32":
        pytest.skip("Windows-only test")
```

Skip là ổn thỉnh thoảng. Nó không phải là chiến lược.

## Mini-example nhanh để bạn copy

**`src/calculator.py`**
```python
def add(a, b):
    return a + b
```

**`tests/test_calculator.py`**
```python
from src.calculator import add
import pytest

@pytest.mark.parametrize(
    "a,b,expected",
    [(1, 2, 3), (10, 5, 15), (-1, 1, 0)],
)
def test_add(a, b, expected):
    assert add(a, b) == expected

def test_add_type_error():
    with pytest.raises(TypeError):
        add("1", 2)
```

Rồi chạy:

```bash
pytest
```
