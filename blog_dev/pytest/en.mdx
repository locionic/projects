I spent way too long wondering why my tests were “passing” even when the code was definitely wrong. Turned out I wasn’t actually running what I thought I was running. Pytest quietly let me get away with it.

So yeah: let’s do a real pytest tutorial, the kind that fixes those early-life gotchas and gets you shipping confidence quickly.

## Installing and setting up pytest (without the drama)

You’ll want pytest installed in the environment where your project lives:

```bash
pip install pytest
```

That’s it. No magic flags. No special sauce.

Then run it from your project root:

```bash
pytest
```

Pytest hunts for files matching its naming pattern (by default it looks for `test_*.py` and `*_test.py`). If nothing runs, it’s usually filenames—not your code.

### A tiny test to confirm everything’s wired

Create `test_math.py`:

```python
def test_addition():
    assert 1 + 1 == 2
```

Run:

```bash
pytest -q
```

If you see a green “1 passed”, you’re good to go.

## The basic building block: `test_` functions + `assert`

Pytest’s whole workflow is simple: write a test function and use plain `assert`.

```python
def test_truth():
    assert True
```

When an assertion fails, pytest shows you a helpful traceback and often a more readable comparison output than you’d expect from “just assert”.

Also: you don’t have to learn a new testing DSL. Python stays Python.

## Running just one test (because you’re iterating)

If you’ve got a bunch of test files, running everything every time gets old fast.

Run only one file:

```bash
pytest test_api.py
```

Run one specific test:

```bash
pytest test_api.py::test_user_login
```

This is the quickest way to speed up your feedback loop while you’re actively changing code.

## Failures that actually help you

Here’s the kind of failure output pytest is good at—asserting list equality:

```python
def test_ordering():
    actual = [3, 1, 2]
    expected = [1, 2, 3]
    assert actual == expected
```

When this fails, pytest doesn’t just scream “it broke.” It points you to what didn’t match, which saves time when you’re debugging.

## Fixtures: shared setup without repeating yourself

Fixtures let you centralize setup logic and reuse it across tests. You define a fixture function, and pytest injects it into your test function by name.

Example:

```python
import sqlite3
import pytest

@pytest.fixture
def db():
    conn = sqlite3.connect(":memory:")
    yield conn
    conn.close()
```

Use it:

```python
def test_something(db):
    cur = db.cursor()
    cur.execute("select 1")
    assert cur.fetchone() == (1,)
```

That `yield` form matters. It lets you do cleanup after the test runs without turning everything into a nested mess of try/finally blocks.

## Parametrized tests: stop copy-pasting the same test

If the test logic is the same but inputs/expected outputs differ, use `pytest.mark.parametrize`.

```python
import pytest

@pytest.mark.parametrize(
    "a,b,expected",
    [
        (1, 2, 3),
        (2, 2, 4),
        (10, -1, 9),
    ],
)
def test_add(a, b, expected):
    assert a + b == expected
```

This makes it easier to cover edge cases without turning your test file into a wall of duplicated code.

## Testing exceptions correctly

You don’t need to catch exceptions manually most of the time. Use `pytest.raises`.

```python
import pytest

def divide(a, b):
    return a / b

def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        divide(1, 0)
```

If you want to match the error message too:

```python
with pytest.raises(ZeroDivisionError, match="division by zero"):
    divide(1, 0)
```

## Markers: tag tests so you can run the right ones

Markers are a simple way to categorize tests. For example: unit vs integration vs slow.

```python
import pytest

@pytest.mark.integration
def test_external_api():
    ...
```

Run only integration tests:

```bash
pytest -m integration
```

Skip integration tests:

```bash
pytest -m "not integration"
```

I like markers because they help you keep local runs fast while still having heavier checks available.

## Skipping tests (use it, but don’t worship it)

If a test can’t run yet, you can skip it.

```python
import pytest

@pytest.mark.skip(reason="Not implemented yet")
def test_future_feature():
    ...
```

Or skip dynamically:

```python
import sys
import pytest

def test_platform_specific():
    if sys.platform != "win32":
        pytest.skip("Windows-only test")
```

Skipping is fine occasionally. It’s not a strategy.

## A quick mini-example you can copy

**`src/calculator.py`**
```python
def add(a, b):
    return a + b
```

**`tests/test_calculator.py`**
```python
from src.calculator import add
import pytest

@pytest.mark.parametrize(
    "a,b,expected",
    [(1, 2, 3), (10, 5, 15), (-1, 1, 0)],
)
def test_add(a, b, expected):
    assert add(a, b) == expected

def test_add_type_error():
    with pytest.raises(TypeError):
        add("1", 2)
```

Then run:

```bash
pytest
```
