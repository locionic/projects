# Practical Pytest: Stop Guessing, Start Testing

I spent way too long wondering why my tests were “passing” when my code was definitely broken. Pytest is incredibly powerful, but it quietly lets you get away with bad habits early on.

<Callout type="idea" title="The Golden Rule">
Tests should fail when the code is broken. If you write a test and you haven't seen it fail on purpose at least once, don't trust it.
</Callout>

---

## 1. The Core Workflow

<Stepper>
  <Step title="Install and Run">
    Install with `pip install pytest`. Run it by typing `pytest` in your project root. It will automatically find files named `test_*.py`.
  </Step>
  
  <Step title="Write a Basic Test">
    Pytest doesn't require a heavy DSL. Just write functions starting with `test_` and use plain Python `assert` statements.
    ```python
    def test_addition():
        assert 1 + 1 == 2
    ```
  </Step>
  
  <Step title="Target Specific Tests">
    Speed up your feedback loop by running only the specific test you are working on:
    `pytest test_math.py::test_addition`
  </Step>
</Stepper>

---

## 2. Powerful Pytest Features

<Callout type="info" title="Fixtures: Shared Setup">
Use fixtures to set up database connections or dummy data once, instead of copy-pasting it into every test.
```python
@pytest.fixture
def empty_cart():
    return ShoppingCart()

def test_add_item(empty_cart):
    empty_cart.add("apple")
    assert len(empty_cart) == 1
```
</Callout>

<Callout type="info" title="Parametrize: Test Multiple Cases">
Stop writing `test_add_positive`, `test_add_negative`. Use parametrization to run the same logic with different inputs.
```python
@pytest.mark.parametrize("a,b,expected", [(1,2,3), (-1,1,0)])
def test_add(a, b, expected):
    assert add(a, b) == expected
```
</Callout>

<Callout type="warning" title="Testing Exceptions">
Don't use `try/except` to test for errors. Use `pytest.raises` to guarantee an exception is thrown.
```python
def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        divide(1, 0)
```
</Callout>
