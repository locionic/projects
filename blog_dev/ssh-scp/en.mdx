# SSH and SCP: The Two Tools Every Developer Should Actually Understand

If you've spent more than a week working with servers, you've probably typed `ssh` into a terminal without thinking too hard about what's actually happening. And that's fine — until it's not. Until you're locked out of a remote machine at 2am, or you can't figure out why your file transfer keeps failing, or someone asks you why port 22 matters and you just kind of wave your hands.

This post is for that moment. Let's actually talk about SSH and SCP — what they are, how they work, and how to use them without constantly Googling the flags.

---

## What Even Is SSH?

SSH stands for **Secure Shell**. It's a protocol that lets you log into another computer over a network and run commands on it as if you were sitting right in front of it. Before SSH, people used tools like Telnet — which sent everything, including your password, in plain text over the wire. Anyone sniffing the network could read it. SSH fixed that by encrypting the entire session.

The protocol was invented by Tatu Ylönen in 1995 after a password-sniffing attack hit his university's network. He wrote the first version in a matter of weeks and released it for free. It caught on fast, and today it's essentially the standard for remote server access.

---

## Port 22: Why That Number?

When you connect to a server via SSH, you're almost always hitting **port 22**. This is the default port assigned to SSH by IANA (the Internet Assigned Numbers Authority), and it's been that way since the protocol was standardized.

Ports below 1024 are called *well-known ports* and are reserved for system-level services. Port 22 sits comfortably in that range alongside things like port 80 (HTTP) and port 443 (HTTPS).

A lot of sysadmins will move SSH to a non-standard port — like 2222 or 22222 — as a basic security measure. It doesn't actually stop a determined attacker (port scanning is trivial), but it dramatically cuts down on automated brute-force bots that just hammer port 22 all day looking for weak passwords. It's security through obscurity, and it works as a first layer.

To connect to a non-default port, you just pass the `-p` flag:

```bash
ssh -p 2222 user@yourserver.com
```

## The Basics: Connecting to a Server

The simplest SSH command looks like this:

```bash
ssh user@hostname
```

For example:

```bash
ssh alice@192.168.1.100
```

The first time you connect to a server, SSH will warn you that it doesn't recognize the host and show you a fingerprint. This is the server proving its identity. If you see this and say yes, SSH saves the fingerprint in `~/.ssh/known_hosts`. The next time you connect, it checks that the fingerprint matches. If it doesn't — maybe the server was reinstalled, or maybe someone is pulling a man-in-the-middle attack — SSH will refuse the connection and yell at you.

Pay attention to that warning. It exists for good reason.

## SSH Keys: Stop Typing Your Password

Passwords are fine for getting started, but SSH key authentication is faster, more secure, and honestly just more satisfying once you set it up.

Here's the idea: you generate a pair of keys — a **private key** that lives on your machine and never leaves it, and a **public key** that you put on the server. When you connect, SSH uses cryptography to prove you hold the private key without ever transmitting it. The server checks the public key and lets you in.

To generate a key pair:

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

This creates two files: `~/.ssh/id_ed25519` (private, guard it with your life) and `~/.ssh/id_ed25519.pub` (public, you can share this freely).

Then copy the public key to your server:

```bash
ssh-copy-id user@yourserver.com
```

That's it. Next time you SSH in, no password required. The server finds your public key in `~/.ssh/authorized_keys` and you're in.

> **A note on key types:** You'll see older setups using RSA keys (`-t rsa`). RSA is still fine, but Ed25519 is faster, shorter, and considered more secure. Use Ed25519 for new setups.

## The SSH Config File: Your Quality-of-Life Upgrade

If you're regularly connecting to multiple servers with different usernames, ports, and key files, typing the full command every time gets old fast. The SSH config file solves this.

Create or edit `~/.ssh/config`:

```
Host myserver
    HostName 192.168.1.100
    User alice
    Port 2222
    IdentityFile ~/.ssh/id_ed25519

Host staging
    HostName staging.mycompany.com
    User deploy
    IdentityFile ~/.ssh/deploy_key
```

Now instead of:

```bash
ssh -p 2222 -i ~/.ssh/id_ed25519 alice@192.168.1.100
```

You just type:

```bash
ssh myserver
```

It's a small thing, but it compounds. Once you have a good config file, you'll wonder how you lived without it.

## SCP: Copying Files Over SSH

**SCP** stands for **Secure Copy Protocol**. It uses SSH under the hood to transfer files between machines — same encryption, same authentication, same port. If you can SSH into a server, you can SCP files to and from it.

The basic syntax:

```bash
scp source destination
```

Copying a file **from your machine to a server**:

```bash
scp myfile.txt alice@192.168.1.100:/home/alice/
```

Copying a file **from a server to your machine**:

```bash
scp alice@192.168.1.100:/home/alice/myfile.txt ./
```

Copying an **entire directory** (use the `-r` flag for recursive):

```bash
scp -r myproject/ alice@192.168.1.100:/home/alice/
```

And if your server is on a non-default port, you use a capital `-P` (note: this trips people up because SSH uses lowercase `-p` but SCP uses uppercase `-P`):

```bash
scp -P 2222 myfile.txt alice@192.168.1.100:/home/alice/
```

If you've set up your SSH config file, SCP respects it too:

```bash
scp myfile.txt myserver:/home/alice/
```

Clean.

## SCP vs. Other Transfer Options

SCP is simple and gets the job done, but it's worth knowing your options:

- **`rsync`** — The go-to for syncing directories. It only transfers what's changed, which makes it much faster for large folders. Also works over SSH. If you're doing any kind of deployment or backup, learn `rsync`.
- **SFTP** — Another SSH-based protocol, more feature-rich than SCP (you can browse, rename, delete files interactively). Many GUI clients like Cyberduck or FileZilla use SFTP.
- **`scp` vs `rsync`** — For a quick one-off file copy, SCP is fine. For anything repeated or directory-based, use `rsync -avz`.

## A Few Security Fundamentals

Since SSH is the front door to your servers, it's worth spending five minutes tightening it up:

**Disable password authentication.** Once you've set up key-based auth, turn off passwords entirely. Edit `/etc/ssh/sshd_config` on the server and set:

```
PasswordAuthentication no
```

Then restart SSH: `sudo systemctl restart sshd`

**Disable root login.** You shouldn't be SSHing in as root. Use a regular user account and `sudo` for elevated commands:

```
PermitRootLogin no
```

**Use a firewall.** Only allow port 22 (or your custom SSH port) from IP addresses you actually use, if possible. Tools like `ufw` on Ubuntu make this straightforward.

**Consider `fail2ban`.** It's a daemon that monitors login attempts and automatically bans IPs that fail too many times. It's not a silver bullet but it's effective against brute-force bots.

## Wrapping Up

SSH and SCP aren't glamorous topics. They're infrastructure — the stuff that has to just work. But taking an hour to actually understand them pays dividends. You'll move faster, you'll debug problems more confidently, and you'll sleep better knowing your servers aren't wide open.

Start with key-based auth if you haven't already. Set up your SSH config file. And next time someone asks why SSH runs on port 22, you'll actually have an answer.
