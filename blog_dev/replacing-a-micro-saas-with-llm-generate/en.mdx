# Replacing a Micro-SaaS with LLM-Generated Code: A 20-Minute Experiment

The rise of Large Language Models (LLMs) has sparked debates about their potential impact on the software-as-a-service (SaaS) industry. While some claim that LLMs will disrupt the SaaS market, others are more skeptical. As someone who has been following this discussion, I decided to put the idea to the test by replacing a micro-SaaS I was using with LLM-generated code.

## The Original Micro-SaaS

The micro-SaaS in question was Shoutout.io, a service that displayed testimonials on my website. It cost $120/year and provided a simple backend for managing testimonials. However, when I logged in to retrieve an annual invoice, I found that the billing section was broken. Despite emailing support, I received a broken link instead of a working invoice. This experience made me wonder if I could rebuild the functionality myself using an LLM.

## The Experiment

I used Codex, an LLM-powered coding tool, to recreate the functionality of Shoutout.io. The process took exactly 20 minutes from start to finish. Here's a breakdown of the steps I took:

1. Asked Codex to create a plan for removing the third-party dependency and hosting testimonials in my codebase.
2. Tweaked the plan to use a modular approach with testimonials stored in a separate JSON file.
3. Added a build step to generate HTML from the JSON file.
4. Tested the solution.
5. Tweaked the UX and generated a schema.
6. Deployed the new solution.

## Results and Implications

The end result was visually identical to the original micro-SaaS, but without the third-party dependency. This experiment highlights several implications for SaaS products and software engineers:

* **Software engineers can replace SaaS functionality quickly**: With LLMs, developers can recreate SaaS functionality in a fraction of the time it would take to build from scratch. However, this requires comfort with the command line and verifying code output.
* **Non-developers may struggle to replicate SaaS functionality**: While it's possible for non-developers to use LLMs to recreate SaaS functionality, it may take longer and require more effort.
* **SaaS products that don't provide ongoing value are at risk**: Shoutout.io's static nature made it easy to replace. In contrast, SaaS products that provide ongoing value, such as compliance updates or analytics, are less likely to be replaced.
* **Buying and selling SaaS businesses may become less profitable**: The ease of replacing SaaS functionality with LLM-generated code could make it less attractive to buy and sell SaaS businesses.

## Conclusion

Replacing a micro-SaaS with LLM-generated code in 20 minutes was a surprising experience. While this experiment highlights the potential of LLMs to disrupt the SaaS market, it also underscores the importance of providing ongoing value to customers. As LLMs continue to evolve, it's essential for SaaS products to focus on delivering unique value propositions that can't be easily replicated with code.
