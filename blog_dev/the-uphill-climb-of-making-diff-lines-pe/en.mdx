# The Uphill Climb of Making Diff Lines Performant

Pull requests are the lifeblood of GitHub, and as engineers, we spend a significant amount of time reviewing and managing them. With GitHub's massive scale, where pull requests can range from small one-line fixes to changes spanning thousands of files and millions of lines, the pull request review experience must remain fast and responsive. Recently, we shipped the new React-based experience for the Files changed tab, with a primary goal of ensuring a more performant experience across the board, especially for large pull requests.

## Performance Improvements by Pull Request Size and Complexity

When investigating and planning our next steps for improving performance issues, it became clear that there wouldn't be a single silver bullet. Techniques that preserve every feature and browser-native behavior can still hit a ceiling at the extreme end. Instead, we developed a set of strategies, each designed to address a specific pull request size and complexity. These strategies focused on the following themes:

* Focused optimizations for diff-line components
* Making the primary diff experience efficient for most pull requests
* Gracefully degrading with virtualization for the largest pull requests
* Investing in foundational components and rendering improvements

## First Steps: Optimizing Diff Lines

Our team's goal was to improve pull request performance by reducing memory and JavaScript heap size, reducing the DOM node count, and reducing our average Interaction to Next Paint (INP) and significantly improving our p95 and p99 measurements. To achieve this, we focused on simplification: less state, fewer elements, less JavaScript, and fewer React components.

### What Worked and What Didn’t with v1

In v1, each diff line was expensive to render. In unified view, a single line required roughly 10 DOM elements; in split view, closer to 15. This approach made sense initially, but it proved unsustainable for our largest pull requests.

### Small Changes Make a Large Impact: v2

No change is too small when it comes to performance, especially at scale. For example, we removed unnecessary `<code>` tags from our line number cells. While dropping two DOM nodes per diff line might appear minor, across 10,000 lines, that's 20,000 fewer nodes in the DOM.

### Simplifying the Component Tree

For v2, we removed deeply nested component trees, opting for dedicated components for each split and unified diff line. While this led to some code duplication, it simplified data access and reduced complexity.

### Moving Complex State to Conditionally Rendered Child Components

The most impactful change from v1 to v2 was moving app state for commenting and context menus into their respective components. By moving the commenting state into the nested components for each diff line, we ensured that the diff-line component's main responsibility is just rendering code.

### O(1) Data Access and Less “useEffect” Hooks

We adopted a two-part strategy to address O(n) lookups and extra re-rendering through useEffect hooks. First, we restricted useEffect usage strictly to the top level of diff files. We also established linting rules to prevent the introduction of useEffect hooks in line-wrapping React components.

## Virtualization for Our Largest Pull Requests

When working with massive pull requests, the usual performance tricks just don't cut it. That's where window virtualization steps in. We integrated TanStack Virtual into our diff view, ensuring that only the visible portion of the diff list is present in the DOM at any time.

## Further Performance Optimizations

To push performance even further, we tackled several major areas across our stack, each delivering meaningful wins for speed and responsiveness. We focused on trimming unnecessary React re-renders, honing our state management, and optimizing rendering to hydrate only visible diff lines.

## Results

Our efforts had a massive impact, with significant reductions in memory usage, DOM nodes, and INP times. For a numeric look, check out the results below:

| Metric | v1 | v2 | Improvement |
| --- | --- | --- | --- |
| Total lines of code | 2,800 | 2,000 | 27% less |
| Total unique component types | 19 | 10 | 47% fewer |
| Total components rendered | ~183,504 | ~50,004 | 74% fewer |
| Total DOM nodes | ~200,000 | ~180,000 | 10% fewer |
| Total memory usage | ~150-250 MB | ~80-120 MB | ~50% less |
| INP on a large pull request | ~450 ms | ~100 ms | ~78% faster |
