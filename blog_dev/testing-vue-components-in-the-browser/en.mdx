When working on frontend projects, it's common to encounter issues with testing. Traditional testing frameworks like Playwright can be slow and unwieldy, requiring Node code to orchestrate tests. This can lead to a lack of testing, making it difficult to make changes with confidence.

## The Solution
--------------

One approach to testing Vue components is to run tests directly in the browser. This can be achieved using a testing framework like QUnit, which allows you to write unit tests and end-to-end integration tests for your Vue components.

### Step 1: Set up the Component for Testing
-----------------------------------------

To set up the component for testing, you'll need to make it available in the test environment. One way to do this is to add your components to the `window` object, making them accessible in the test code.

```javascript
const components = {
  'Feedback': FeedbackComponent,
  // ...
}

window._components = components;
```

Next, create a `mountComponent` function that renders the component in a temporary invisible div. This div will be removed from the DOM after the test is complete.

```javascript
function mountComponent(template, data) {
  const app = Vue.createApp({
    template: template,
    data: () => data,
  })

  for (const [c, v] of Object.entries(window._components)) {
    app.component(c, v);
  }

  const div = document.getElementById('qunit-fixture')
    .appendChild(document.createElement('div'));

  return div;
}
```

### Step 2: Add Fixture Data
-------------------------

If your tests require data from a database, you'll need to set up some test data. One way to do this is to write SQL to set up test data in your database and add an endpoint to your dev server to reset the test data to a known state.

```javascript
async function reset() {
  return fetch('/api/reset_test_data', { method: "POST" })
}
```

### Step 3: Write a Basic Test
---------------------------

Here's an example of a basic test using QUnit:

```javascript
QUnit.test('renders feedback content', async function (assert) {
  const { div } = mountComponent(
    '<Page :feedbacks="feedbacks" id=2 image=2 page_hash=2 />',
    { feedbacks: [testFeedback] },
  );

  assert.ok(div.textContent.includes('loved this section'));
})
```

## Challenges and Solutions
---------------------------

### Waiting for Parts of the Page to Render
------------------------------------------

When testing Vue components, you'll often need to wait for parts of the page to render before proceeding with the test. One way to do this is to use a `waitFor` function that polls the DOM every 20ms to check if a condition has been met.

```javascript
function waitFor(condition, timeout = 2000) {
  return new Promise((resolve, reject) => {
    const intervalId = setInterval(() => {
      if (condition()) {
        clearInterval(intervalId);
        resolve();
      }
    }, 20);

    setTimeout(() => {
      clearInterval(intervalId);
      reject(new Error('Timeout'));
    }, timeout);
  });
}
```

### Figuring Out the Right Thing to Wait For
--------------------------------------------

In some cases, it may not be straightforward to determine what to wait for in the DOM. One solution is to add a random value to the DOM when an important action is complete, allowing you to wait for that value to appear.

### Adding CSS Classes to Identify Things
-----------------------------------------

When writing tests, it's often necessary to add CSS classes to identify elements in the DOM. However, this approach may not be ideal, as it can make the app less accessible. A better approach may be to use `getByRole` or `data-testid` attributes to identify elements.

### Filling Out Forms is Tricky
------------------------------

When filling out forms in tests, it's not enough to simply set the value of the input field. You'll also need to dispatch an event to tell Vue that the element has changed. For example, checkbox and textarea elements require different types of events.

```javascript
textarea.value = 'banana banana banana';
textarea.dispatchEvent(new Event('input'));

checkbox.checked = true;
checkbox.dispatchEvent(new Event('change'));
```

### Test Coverage
-----------------

To get an idea of test coverage, you can use Chrome's built-in code coverage feature for JavaScript and CSS. This will show you which lines of code are covered by your tests.

## Conclusion
----------

Testing Vue components in the browser can be a challenging task, but with the right approach, it's possible to write effective tests that give you confidence in your code. By using a testing framework like QUnit and following best practices for waiting for parts of the page to render, identifying elements, and filling out forms, you can write comprehensive tests that cover your Vue components.
