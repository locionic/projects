# Testing Vue Components in the Browser with QUnit

Running tests directly in the browser instead of through Playwright or a Node runner is an approach I reached for when I needed quick feedback loops. The setup is lightweight — just QUnit and your components — and the tests run as fast as the browser can render.

This isn't the right choice for every project, but when it fits, it's noticeably faster than spinning up a full E2E framework.

## The Setup

Make your components globally accessible by registering them on the `window` object:

```javascript
window._components = {
  'Feedback': FeedbackComponent,
  'CommentList': CommentListComponent,
};
```

Then create a helper to mount a component into a clean `div`:

```javascript
function mountComponent(template, data) {
  const app = Vue.createApp({ template, data: () => data });
  for (const [name, component] of Object.entries(window._components)) {
    app.component(name, component);
  }
  const fixture = document.getElementById('qunit-fixture');
  const mountPoint = document.createElement('div');
  fixture.appendChild(mountPoint);
  mountPoint.appendChild(app.mount().$el);
  return mountPoint;
}
```

The `qunit-fixture` element is special in QUnit — after each test, its contents are reset automatically, so you don't need manual cleanup.

## Handling Async Rendering

Vue updates the DOM asynchronously. After changing component state, you need to wait for the next tick before asserting:

```javascript
function waitFor(condition, timeout = 2000) {
  return new Promise((resolve, reject) => {
    const start = Date.now();
    const check = () => {
      if (condition()) return resolve();
      if (Date.now() - start > timeout) return reject(new Error('Timeout'));
      requestAnimationFrame(check);
    };
    requestAnimationFrame(check);
  });
}

QUnit.test('renders feedback content', async function (assert) {
  const div = mountComponent(
    '<Page :feedbacks="items" />',
    { items: [{ text: 'loved this section' }] },
  );
  await waitFor(() => div.textContent.includes('loved this section'));
  assert.ok(true);
});
```

Using `requestAnimationFrame` instead of `setInterval` is better here — it stops polling as soon as the condition is met instead of running on a fixed timer.

## Handling Form Inputs

Setting a value on an input isn't enough — you need to dispatch an event so Vue registers the change:

```javascript
const textarea = div.querySelector('textarea');
textarea.value = 'test input';
textarea.dispatchEvent(new Event('input'));

const checkbox = div.querySelector('input[type="checkbox"]');
checkbox.checked = true;
checkbox.dispatchEvent(new Event('change'));
```

Without the `dispatchEvent` call, Vue's reactivity system doesn't pick up the change because it's listening for DOM events, not property changes.

## Setting Up Test Data

If your component fetches data, set up a simple endpoint on your dev server that resets data to a known state:

```javascript
async function resetTestData() {
  await fetch('/api/reset-test-data', { method: 'POST' });
}
```

Run this before each test to ensure isolation. The endpoint should be a dev-only route that's not available in production.

<Callout type="info" title="When Browser Testing Makes Sense">
This approach works well for component libraries and UI-heavy apps where you want fast, focused tests without E2E overhead. It doesn't replace integration tests. For data flow, API integration, and cross-page navigation, I still use Playwright.
</Callout>
