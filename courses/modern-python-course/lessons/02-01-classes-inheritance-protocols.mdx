# Classes, Inheritance & Protocols

Python's OOP is built on a different philosophy than Java or C++. Understanding protocols (duck typing) is the key to writing idiomatic Python.

## Classes Are Runtime Objects

```python
class User:
    # Class variable — shared across instances
    role = "user"

    def __init__(self, name: str):
        self.name = name  # instance variable

    def greet(self) -> str:
        return f"Hello, {self.name}"
```

## Protocols (Structural Subtyping)

Python 3.8+ supports static duck typing via protocols:

```python
from typing import Protocol

class Drawable(Protocol):
    def draw(self) -> None: ...

class Circle:
    def draw(self) -> None:
        print("Drawing circle")

class Square:
    def draw(self) -> None:
        print("Drawing square")

# No inheritance needed — both satisfy Drawable
def render(shape: Drawable) -> None:
    shape.draw()

render(Circle())  # OK
render(Square())  # OK
```

<Callout type="info" title="Protocols vs ABCs">
Protocols use structural typing (duck typing) — any object with the right methods satisfies it. ABCs use nominal typing (explicit inheritance). Prefer protocols for flexibility.
</Callout>

## Method Resolution Order (MRO)

```python
class A:
    def method(self): print("A")

class B(A):
    def method(self): print("B")

class C(A):
    def method(self): print("C")

class D(B, C):
    pass

D().method()  # B (MRO: D → B → C → A)
print(D.__mro__)  # (<class 'D'>, <class 'B'>, <class 'C'>, <class 'A'>, <class 'object'>)
```

<Flashcard term="MRO (Method Resolution Order)" definition="Python uses C3 linearization. Child classes come before parents, and the order of base classes matters. Use ClassName.__mro__ to inspect it." category="Python OOP" />

## Key Takeaways

- Use Protocols for structural typing instead of ABCs
- Understand MRO when using multiple inheritance (or avoid it)
- Composition over inheritance is especially true in Python
- Use `__slots__` to reduce memory overhead in data-heavy classes
